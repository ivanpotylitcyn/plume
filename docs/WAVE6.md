# Волна 6 — закрытие проекта (сведение остатков в 0 + мягкий замок)

Чеклист шестой волны. Цель — довести **правую половину** потока до конца:
`комплектация → передача → закрытие`. После передач у проекта остаются лоты с
остатком (`live>0`); чтобы **закрыть** проект, их надо свести в 0 закрывающими
документами и поставить мягкий замок-веху на статус проекта. Продолжает паттерн
волн 2–5 (записываемый документ + движок = единственный источник правил +
`rebuild_movements` = склад остаётся проекцией). Значимые решения — в
[JOURNAL.md](JOURNAL.md) (запись 2026-07-02, «Волна 6»).

**Цель волны 6:** в панели закрытия проекта (под дашбордом дефицита) видеть
остаточные лоты → сводить каждый в 0 одним кликом: **списать** (`Writeoff` → серый
путь, чистый `−ISSUE`) или **на баланс** (`Requisition` → белый «Собственный
склад», отпочкование с рождением лота-потомка) → когда остатков нет, **закрыть
проект** (`active → closed`, мягкий замок); переоткрытие свободно.

## Объём (вариант B — оба выхода)

**Входит:** списание (`Writeoff`, чистый `−ISSUE`) + требование/отпочкование
(`Requisition`, `−ISSUE` источника + рождение лота-потомка `+RECEIPT`) → записываемые
DRF-эндпоинты → панель закрытия проекта (`project_closure`) + мосты «списать»/«на
баланс» + `close_project`/`reopen_project` → седьмой и восьмой режимы фронта
(«Списания» 🗑, «Требования» ⇄) + панель закрытия в режиме проекта → юнит-тесты +
HTTP-smoke.

**НЕ входит (следующие волны):** инвентаризация (`Inventory` — «найденные»/излишки
партии, обратная операция), командный свод + консолидированный `Procurement`-дашборд,
экспорт `order.xlsx`, pegging, два числа бюджета + себестоимость/экономия,
логин-экран, UI вложений.

## Решения (см. JOURNAL 2026-07-02, «Волна 6»)

- **Схему БД не трогаем** — `Writeoff`/`WriteoffLine`/`Requisition`/`RequisitionLine`
  уже были; `rebuild_movements` с волны 1 генерит по ним `−ISSUE`/`+RECEIPT`. Волна 6
  добавляет только записываемую надстройку. Диаграммы README не меняются.
- **Списание = чистое выбытие** (`−ISSUE`, born-лота нет — `Writeoff` не origin).
  «В серый склад» — конвенция учёта; ре-материализация серых остатков — актом
  `Inventory` (следующая волна). Списываем только своё (`lot.project ==
  writeoff.project`). Кол-во не клампим (пересписать в минус можно — недостача
  информативнее нуля, мутабельная ДНК).
- **Требование = отпочкование**: `−ISSUE` источника + рождение лота-потомка в
  проекте-получателе (`+RECEIPT`), потомок наследует item/цену/название/зав.№ и
  `predecessor` → источник (генеалогия/провенанс для kitting из остатков). Источник —
  из **любого** проекта (постановка своего на баланс → белый; заём у соседнего
  активного B→A). Один источник = одна строка (пара строка↔потомок однозначна).
  Правка/удаление синхронят и источник, и потомок; guard «потомок не потреблён ниже».
- **Замка на списании/требовании нет** (у моделей нет поля-статуса, как у `Transfer`
  до обкатки) — правимо всегда; корректность держат guard'ы + `PROTECT`. Замок наступает
  **на уровне проекта** (`close_project`).
- **Панель закрытия** (`project_closure`) — остаточные лоты (`live≠0`): положительные
  сводим в 0 выходами, отрицательные — аномалия «подбей лоты» (недостача). `can_close`
  = внешний проект без остатков. Внутренние склады (белый/серый) постоянны — не
  закрываются.
- **`close_project` — мягкий замок-веха** (зеркалит approve/post прошлых волн): gate
  «остатков нет», `active → closed` + `closed_at`. Ничего не разрушает — реюз/
  переоткрытие (`reopen_project`) свободны.
- **Мосты панели** (payoff, как «дефицит → заказ»): `writeoff_lot` / `requisition_lot`
  — один клик уводит остаток лота в 0 (find-or-create документ + строка), возвращают
  свежую панель (пользователь остаётся и смотрит, как остатки тают).

## Этапы

### Этап 1 — Движок закрытия (чистый Python + тесты) ← ядро волны
- [x] `writeoff_cockpit` + `create_writeoff` + `add|update|remove_writeoff_line`
  (`−ISSUE`, guard чужого проекта, qty>0, коррекция).
- [x] `requisition_cockpit` + `create_requisition` + `add|update|remove_requisition_line`
  (`−ISSUE` источника + born-`+RECEIPT`, наследование цены/провенанса, guard
  источник≠получатель / дубль источника / потомок потреблён).
- [x] `project_closure` (остаточные лоты + готовность) + `close_project`/`reopen_project`
  (gate внешний+нет остатков) + мосты `writeoff_lot`/`requisition_lot` +
  `all_available_lots` (сквозной пикер источника) + `_internal_project` (белый синглтон).
- [x] Юнит-тесты (20): списание `−ISSUE`/минус/коррекция/чужой проект; требование
  born-потомок с ценой/предком, синк источник↔потомок, дубль/само-в-себя; панель
  блокирует при остатке, мосты сводят в 0, close/reopen, аномалия-минус, внутренний
  не закрывается.
- _Готово, когда:_ тесты зелёные, живой остаток источника/потомка совпадает с движком.

### Этап 3 — Фронт: режимы «Списания»/«Требования» + панель закрытия
- [x] Седьмой/восьмой режимы activity-bar (🗑 / ⇄): списки + призрачная строка

## Докрутка на обкатке (2026-07-02)

По итогам клика предметника:
- [x] **Видимость внутренних складов.** Требование корректно кладёт лоты на WHITE
  (проверено), но экрана остатков у внутренних проектов не было (дефицит пуст, панель
  закрытия скрыта) → «сток есть, показать негде». Добавлен `ProjectStockPanel`
  (read-only список живых лотов, переиспуёт `project_available_lots`): для проектов
  kind≠external режим проекта показывает **«Остатки склада»** вместо дефицита/закрытия.
  Списание в GREY ничего не кладёт — **by design** (выбытие из учёта; серую
  ре-материализацию даст `Inventory`, волна 7).
- [x] **Правка шапки прямо в кокпите** (сквозная, все 6 кокпитов): инлайн-поля
  номер/дата + мягкие (причина/примечание/кол-во образцов) автосейвом (`CommitInput`,
  строка `.hdr-edit` под подзаголовком), read-only под замком. **Структурные якоря
  (проект/поставщик — дом лотов) не правим инлайн** (смена переселяет лоты — отдельная
  операция). Движок: `update_receipt/purchase/transfer/writeoff/requisition/kitting` +
  PATCH на detail-вьюхах. Nullable-дата (заказ/комплектация) очищается пустой строкой.
- Терминологию замка (списание/требование «нельзя закрыть/пробить») синхронизируем
  после MVP — отложено осознанно.
