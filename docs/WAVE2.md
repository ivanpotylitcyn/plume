# Волна 2 — записываемое ядро + кокпит комплектации

Чеклист второй волны. Цель — увести команду из Django admin: сделать продукт
**записываемым** в ключевом потоке (комплектация) и заложить переиспользуемый
паттерн прямого редактирования. Значимые решения — в [JOURNAL.md](JOURNAL.md)
(запись 2026-07-01, «Волна 2»).

**Цель волны 2:** открыть комплектацию → вести сборку прямо в UI: пробивать
компоненты (пайка), править количества автосейвом, закрывать акт (рождается
лот-прибор). Кокпит красит призрачные строки движком (`▲/●/✓`).

## Объём

**Входит:** проекция кокпита (`kitting_cockpit`) → записываемые DRF-эндпоинты
(пайка / автосейв / закрытие / переоткрытие) → третий режим фронта «Комплектации»
с созданием и кокпитом → юнит-тесты на мутации.

**НЕ входит (следующие волны):** ввод остальных документов (`Receipt`/`Transfer`/
`Writeoff`/`Requisition`/`Inventory`), логин-экран, командный свод + `Procurement`,
два числа бюджета + себестоимость/экономия, pegging, заём между проектами
(`Requisition`), UI вложений.

## Решения (см. JOURNAL 2026-07-01)

- **Схему БД не трогали** — `Kitting`/`KittingLine`/`Lot` уже были.
- **Мутации — единым источником правил в движке** (`add/update/remove_kitting_line`,
  `close_kitting`, `reopen_kitting`); каждая дёргает `rebuild_movements` → склад
  остаётся чистой проекцией. Вьюхи только разбирают запрос и мапят
  `ValidationError` → 400.
- **Пайка = промоушн призрачной строки** в реальную `KittingLine` (+ `-ISSUE`).
  Закрытие рождает лот-прибор (`+RECEIPT`) со **снимком себестоимости**
  = `Σ(qty×цена лотов) / кол-во`. Переоткрытие — мягкий замок с guard по потомкам.
- **Автор без логин-экрана:** `_actor(request)` = залогиненный юзер или дефолтный
  суперюзер (авторство остаётся «на документах»). Форма входа — отдельная волна.
- **CSRF:** в dev фронт анонимен (другой origin через прокси) → DRF не форсит CSRF;
  фронт всё равно шлёт `X-CSRFToken` из cookie, если он есть (прод-совместимо).
- **Заём между проектами в пайке запрещён** — только свои лоты проекта (заём —
  через `Requisition`, позже).

## Этапы

### Этап 1 — Движок кокпита (чистый Python + тесты) ← ядро волны
- [x] `kitting_cockpit(kitting)` — BOM 1 уровень, реальные (пробитые) строки +
  призрачные строки, покрашенные `_coverage` (✓/●/▲), лоты-кандидаты
  (`available_lots`).
- [x] Мутации: `add/update/remove_kitting_line`, `close_kitting` (снимок
  себестоимости), `reopen_kitting` (guard по потомкам). Все — через
  `rebuild_movements`.
- [x] Юнит-тесты (12 новых): призрачные строки, пайка (+ отказ по чужому проекту/
  компоненту), автосейв/удаление, закрытие + себестоимость, переоткрытие + guard.
- _Готово, когда:_ тесты зелёные, склад пересобирается корректно на демо-данных.

### Этап 2 — Записываемый DRF
- [x] `GET/POST /api/kittings/` (список / создание), `GET /api/kittings/<pk>/`
  (кокпит), `POST .../lines/` (пайка), `PATCH/DELETE /api/kitting-lines/<pk>/`
  (автосейв qty / удаление), `POST .../close/` и `.../reopen/`.
- [x] `_actor(request)` (автор), `_dec` (Decimal без float-погрешности),
  `ValidationError` → 400 с `detail`.
- _Готово, когда:_ HTTP-путь через test Client: create 201, пайка 201, валидация
  400, close/reopen 200.

### Этап 3 — Фронт: режим «Комплектации» + кокпит
- [x] Третий режим activity-bar (⛭): список комплектаций (глиф статуса) +
  призрачная строка «＋ Новая» (проект + производимый прибор + кол-во).
- [x] `KittingView` — кокпит: реальные строки зелёные с **автосейвом qty**
  (blur/Enter, без кнопки «Сохранить») и удалением; призрачные строки с пикером
  лота-кандидата и «спаять»; закрыть/переоткрыть; показ рождённого лота-прибора.
- [x] API-клиент: мутирующие хелперы (`send` с CSRF), типы кокпита.
- _Готово, когда:_ открываешь комплектацию — ведёшь сборку в UI, числа сходятся
  с движком.

## Локальный запуск (памятка)

```
cd backend && docker compose up -d           # MySQL 8.0.25 на :3307
cd backend && .venv/bin/python manage.py runserver 8000
cd frontend && npm run dev                    # http://localhost:5173
```
