## Ф5 — потребность и план до листьев (разведка 2026-07-13)

**Симптом (Иван):** стоимость составной части «ЛК-1» не попадает в «план (прогноз)»
проекта. Добавишь «МУКВ» прямо (не узлом ЛК-1) — план обновляется; добавишь ЛК-1
(который содержит МУКВ, ДО, УДМ) — план = 0.

**Корень.** Движок моделирует **2 уровня** (прибор → прямой BOM = покупные материалы),
а данные — **рекурсивный N-уровневый BOM**. Везде, где считается потребность/план,
итерируется ровно `demand.target_item.bom_lines` — **один уровень**. Это документированное
MVP-упрощение (CLAUDE.md «1 уровень, терминируя на лотах»; подзаголовок формы «(1 уровень
BOM)»). Три места:
- `project_deficit` ([engine.py:508]) — секции «Потребность» + аккордеон прибора;
- `_project_estimate` ([engine.py:604]) — **план (прогноз)**;
- `command_deficit` ([engine.py:2139]) — командный свод (та же болезнь).
- Комплектация ([engine.py:763]) — 1 уровень **правильно** (сборка потребляет прямых
  детей-лотов), НЕ трогаем.

**Почему план=0 для ЛК-1.** В `_project_estimate` для производимого компонента стоит
`if component.produced: continue` — пропуск, в расчёте что стоимость узла учтётся в
`_project_cost` (себестоимость). Но `_project_cost` считает только **фактически собранные**
лоты-цели (проведённый Kitting). Пока ничего не собрано и МУКВ — не верхняя цель, он даёт
ноль. У ЛК-1 все дети производимые → все пропущены → estimate=0. МУКВ добавленный прямо
имеет дети-листья → они оцениваются → план=2933 ₽.

**Двойной счёт.** `estimated_cost` производимого узла — уже **роллап-сумма листьев**
([engine.py:3011] `_rollup_cost`). Суммировать план и по узлам, и по листьям = задвоение.
Правильно: **план = Σ по листьям** (у листа ручная цена), узлы разузловываются насквозь,
не суммируясь. Роллап `estimated_cost` узла остаётся только для витрины изделия, не для
плана проекта.

**Проектируемое решение.** Новый рекурсивный хелпер `explode_demand(item, qty, project)`
→ `{leaf_item: qty}`: производимый узел разузловывается в детей (×`bl.qty`), покупной —
лист-терминал. Страховка от циклов (`visiting`, как в `_rollup_cost`); производимый узел
без BOM → «неполнота» (как `incomplete` в роллапе), вклад 0. Свод потребности проекта =
Σ `explode_demand` по всем `demand`. Затем:
- **Потребность** (секция + свод) = по листьям, `_coverage` на лист (как сейчас, но
  листья вместо 1-го уровня). Следствие: «＋ в заказ» появляется только на **покупных
  листьях** (ДО/МУКВ заказать нельзя — это уходит).
- **План** = Σ по листьям `(on_order+to_order) × leaf.estimated_cost`. Пропуск `produced`
  исчезает (разузлование и так даёт только листья) → двойного счёта нет.
- `command_deficit` — тем же хелпером.
- `_project_cost`/себестоимость/экономия — отдельная ось (снимки собранных приборов для
  КП), НЕ трогаем; план и себестоимость не складываются, задвоения между ними нет.

**Развилка «терминируя на лотах» (нетинг) — РЕШЕНО: пока БЕЗ нетинга.** Строгий MRP
(вычитать готовые лоты подсборки на каждом узле перед разузлованием остатка) — следующим
шагом. Сейчас: структурное разузлование до листьев; покрытие складом/заказом считает
`_coverage` на самом листе. Риск перезаказа, если подсборка уже собрана, — известен и
осознан; чинит текущий симптом (план=0 у прибора-из-подсборок). ⚠️ Нетинг подсборок —
кандидат в следующую волну.

**Развилка аккордеон «Приборы» — РЕШЕНО (уточнено Ф5b).** Сначала аккордеон свели к
плоским листьям, но Иван попросил **видеть вложенность**. Итог (Ф5b): аккордеон = **дерево
BOM** (все уровни: прибор → подсборка → лист, плоский pre-order + `depth`), свод
«Потребность» остаётся плоско по листьям, план — по листьям. Лист несёт покрытие ✓/●/▲ +
склад; узел-подсборка — структурная строка (статус = worst-of поддерева, без «＋ в заказ»
— заказ живёт в своде). Бэкенд: `_demand_tree`; поле `demand.lines` → `demand.tree`
(`is_leaf`, `depth`). Фронт: `DeficitTreeNode`, `TreeRow` с отступом по глубине. +1 тест
(worst-of узла), сюита 320. Реальный verify на сиде: ИЗДЕЛИЕ-А → ПЛАТА-1(подсборка) →
РЕЗ-10К(лист), КОРПУС-1, ВИНТ-М3.

**Ф5b доработка (сворачивание) — сделано.** Подсборки в дереве **сворачиваются по клику
шеврона** и **свёрнуты по умолчанию** (при раскрытии прибора видно только прямые
компоненты; глубже — по шевронам). Чисто фронт: `DeviceTree` держит `expanded:Set` и
скрывает строки глубже свёрнутого узла (плоский pre-order + `depth`); у узлов — кликабельный
`.chev`, у листьев — spacer для выравнивания. Бэкенд без изменений. Количество разузлованных
строк править нельзя (только через BomLine) — это Ивану понравилось, сохранено.

**Радиус фронта.** Структуры `components`/`lines`/бюджета по форме не меняются (лист — тот
же `Item`), правки минимальны; подзаголовок «(1 уровень BOM)» обновить. Тесты (313, В15)
частично проверяют 1-уровневое поведение — обновить + добавить многоуровневые кейсы.
