# Волна 16 — план реализации

Живой план-чеклист шестнадцатой волны. Источник «что делаем сейчас и в каком
порядке». Значимые решения по ходу — в [JOURNAL.md](JOURNAL.md); здесь — объём,
развилки и этапы. Заполняется по мере уточнения (тема волны — эргономика справочника
изделий + импорт BOM из CAD).

**Статус: ПЛАНИРОВАНИЕ** (наброски от 2026-07-12, детали уточняются).

## Объём (три пункта, сформулированы Иваном 2026-07-12)

### Ф1 — Фильтр по категориям в режиме «Изделия»
Добавить в список режима «Изделия» фильтр-дропдаун по категории — **ровно как уже
сделано** в режиме «Ордера» фильтр по типу документа (`kind`). Начал вводить/выбрал
категорию → список сокращается.

- Паттерн-образец: `OrderList` в [App.tsx](../frontend/src/App.tsx) — `kindOptions` +
  `<select className="list-proj">` (фильтр по типу) рядом с текстовым фильтром.
- Целевой компонент: `ModeList` (у него уже есть `projectFilter` для «Заказов» —
  сделать аналогичный опциональный `categoryFilter`, опции из `Category.label`
  строк). Строка `ItemRow` несёт `category` (label/icon) — фильтровать по нему.
- Показывать дропдаун только когда категорий > 1 (как `projOptions.length > 1`).

### Ф2 — Кнопка «Загрузить BOM» в форме редактирования изделия
В форме изделия ([ItemView.tsx](../frontend/src/ItemView.tsx), в блоке «Состав (BOM)»,
под замком/для `composable`) — кнопка **«Загрузить BOM»**. По ней грузим BOM,
**напрямую экспортированный из Altium Designer**. Компоненты в этом BOM — изделия из
библиотеки, которые мы уже синхронизировали CSV-импортом в волне 15 → сопоставление
строк BOM с существующими `Item` по **`design_item_id`** (заказной PN, ключ
библиотеки; та же ось сопоставления, что и в В15-синке).

- **ФОРМАТ BOM — НЕ ОПРЕДЕЛЁН.** Будет либо **xlsx**, либо **csv** — Иван уточнит
  позже (образец файла даст). До этого — не проектировать парсер жёстко.
- Переиспользовать наработки В15: серверный парсер + диф/применение, форма загрузки
  (`uploadMulti`/`upload` в [api.ts](../frontend/src/api.ts)), паттерн диф-вью
  (`LibraryImportView`). BOM-импорт логически близок: строки → сопоставление с
  `Item` по `design_item_id` → создание/сверка `BomLine` у текущего изделия.
- Открытые вопросы (уточнить с Иваном при старте):
  - Формат файла (xlsx/csv), кодировка, разделитель, имена/порядок колонок
    (`Design Item Id` + количество? уровни вложенности? refdes?).
  - Плоский BOM (один уровень, компоненты текущего изделия) или иерархический?
  - Поведение при отсутствии компонента в справочнике (строка BOM ссылается на
    `design_item_id`, которого нет как `Item`): ошибка / пропуск / предложить
    синхронизировать библиотеку сперва?
  - Замена состава целиком vs слияние (диф-вью с галочками, как в В15)?
  - Количество/позиция (`BomLine.qty`/`position`) — откуда в экспорте Altium.

### Ф3 — Фильтр вместо большого select при добавлении компонента
В строке-призраке добавления компонента (`AddComponent` в
[ItemView.tsx](../frontend/src/ItemView.tsx)) большой `<select>` со всеми изделиями
неудобен. Заменить на **фильтр-поле с type-ahead** — как текстовый фильтр вверху
списков «Изделия»/«Ордера»: вводишь → список кандидатов сокращается → выбираешь
существующий компонент. Отсев самого изделия и уже добавленных остаётся (как сейчас).
