# Руководство по интерфейсу — Plume

Однозначная спецификация интерфейса для чистового прохода и всех будущих экранов.
Составлено на интервью 2026-07-04. Принципы и «почему» — в
[UI_PRINCIPLES.md](UI_PRINCIPLES.md); здесь — «что именно». При расхождении
в конкретике побеждает это руководство.

Статус: **черновик на вычитке**. Открытые места помечены `[ОТКРЫТО]`.

## 1. Словарь частей интерфейса

Русские функциональные имена. Ими пишем инструкции и разговариваем.

| Имя | Что это |
|-----|---------|
| **Панель режимов** | узкая колонка иконок слева; каждая иконка — режим (лупа на данные) |
| **Список режима** | средняя колонка: перечень объектов выбранного режима |
| **Рабочее поле** | центральная зона; в ней открывается форма выбранного объекта |
| **Форма** | экран объекта в рабочем поле: заголовок + шапка-свойства + строки + окружение («форма поставки», «форма изделия») |
| **Заголовок формы** | первая строка формы: название + мета-строка + индикатор сохранения + замок шапки |
| **Шапка** | панель свойств объекта (пары «подпись: значение») под заголовком |
| **Окружение** | read-only-проекции линзы вокруг объекта (where-used, остатки, дефицит) |
| **Призрачная строка / призрачный пункт** | пустая строка таблицы, появляющаяся сама; пункт «＋ Новая …» первым в списке режима |
| **Замок формы** | бесплатный переключатель редактирования всей формы — шапки и строк (см. §5) |
| **Фиксация** | документное состояние (posted/closed): документ **зафиксирован** — факты учтены движком, форма read-only; обратная операция — «снять фиксацию» (см. §5) |
| **Глифы статусов** | сквозной словарь ● ▲ ✓ 🔒 ○ (см. UI_PRINCIPLES §7) |

Упразднено: слово **«кокпит»** (было размыто) — говорим «форма ‹объекта›»;
**строка состояния** внизу — удаляется совсем (автосейв-индикатор переехал в
заголовок формы; счётчики сущностей никто не читал).

## 2. Режимы и иконки

Иконки панели режимов — **Codicons** (родной набор VS Code, MIT, самохостим SVG
как шрифты в `public/`). Монохром: активный режим — белый + полоса слева,
остальные — серые. Эмодзи с панели уходят; **глифы статусов в строках остаются
как есть** (там цвет несёт смысл).

Порядок панели — **поток жизненного цикла изделия**, не история волн: планирование
(закупки) → исполнение (заказы) → приёмка (поставки) → сборка → выбытие/сверка.

| Режим | Codicon | Смысл |
|-------|---------|-------|
| Проекты | `project` | дефицит, панель проекта |
| Изделия | `circuit-board` | справочник, остатки, состав (BOM) |
| **Закупки** | `table` | командный свод, order.xlsx (было «Закупки-план») |
| Заказы | `checklist` | обязательства поставщику |
| **Поставки** | `inbox` | УПД, рождение лотов |
| Комплектации | `tools` | сборка, списание под прибор |
| Передачи | `export` | отгрузка заказчику |
| Требования | `arrow-swap` | отпочкование, постановка на баланс |
| Списания | `trash` | выбытие, серый путь |
| Инвентаризации | `search` | найденные партии |
| Выход | `sign-out` | внизу панели |

**Переименование: Receipt = «Поставка»** (появление товаров с накладной), режим
«Поставки». Заменить везде: панель, формы, тексты. Глоссарий README («Приход /
УПД») обновить при ближайшей правке README — предложить отдельно.

Остальные имена режимов — без изменений.

## 3. Типографика

Два шрифта, оба самохостятся (OFL):

- **JetBrains Mono** (400/500) — всё **идентифицируемое и счётное**: коды,
  номера документов, количества, деньги, даты, бейджи, мета-строки, **строки
  списков режима** (mono выравнивает похожие названия в колонку).
  Числа — всегда `tabular-nums`, выравнивание вправо.
- **Inter** (400/500) — всё **литературное**: названия изделий и проектов,
  описания, комментарии, подписи полей, кнопки. Заменяет Golos Text
  (файлы Golos удалить вместе с заменой).

**Правило границы:** в одной строке не смешивать mono и текстовый шрифт одного
размера и веса — блоки разводятся размером, весом, цветом или строками. Главный
пример — заголовок формы (§4).

## 4. Заголовок формы (единый для всех форм)

Вариант «название первое»:

```
ООО «Электроконтакт»                    ✓ сохранено  [🔓]
УПД 141 · 02.06.2026 · P-001
```

- 1-я строка: **литературное название** объекта — Inter 500, крупно, белым.
  Справа вверху: **замок формы**, слева от него — индикатор сохранения
  (как в мокапах-примерках).
- 2-я строка: **мета-строка** — JetBrains Mono, мелко, приглушённо:
  глиф статуса + код/номер · дата · проект (набор зависит от типа объекта,
  разделитель `·`).
- **Индикатор сохранения** живёт здесь (не в строке состояния). Состояния на
  разлокованной форме: `● редактируется` (покой, оранжевый) →
  `● сохраняется…` (идёт PATCH) → `✓ сохранено` (зелёный) →
  `ошибка: …` (красный). На залокованной и зафиксированной форме индикатора нет.

## 5. Замок формы и фиксация документа

Два разных механизма в **одном слоте** справа вверху заголовка. Различаются
видом: **контур = вид, заливка = фиксация**. Мокап трёх состояний — артефакт
«Два замка» (интервью 2026-07-05).

**Замок формы** — интерфейсный, бесплатный, личный (нигде не хранится,
данных не трогает):

- **Закрыт** (по умолчанию у существующих объектов): контурный 🔒, **вся форма —
  чистый текст без единого поля ввода**: шапка, строки, всё. Эта чистота —
  ценность сама по себе; закрытый вид — основной вид формы.
- **Открыт**: контурный 🔓 — редактируется всё редактируемое: поля шапки и
  строк — инпуты, призрачная строка, автосейв. Переключение мгновенное, в обе
  стороны, без подтверждений; ничего не перестраивается и не уезжает.
- У **черновиков** (только что созданных) замок открыт сразу.

**Фиксация документа** (posted/closed; в README-модели — «мягкий замок») —
смысловая, общая для всех, живёт в данных. Слово-объединитель всех
терминальных статусов: проведённая поставка, проведённая комплектация,
закрытый проект — всё это «документ **зафиксирован**».

- Зафиксированный документ: в слоте замка — **чип с заливкой и подписью**
  (`🔒 зафиксирована`), вся форма read-only (серый стиль), в мета-строке —
  серый глиф 🔒, индикатора сохранения нет.
- Клик по чипу **предлагает снять фиксацию** — подтверждение **всегда**
  (короткое; при зависимых — развёрнутая модалка «поедут N зависимых»).
  После согласия документ — черновик, форма сразу разлокована.
- **Фиксация — отдельная осознанная операция** («Зафиксировать», с
  валидацией), НЕ закрытие замка формы: снять фиксацию через слот замка
  можно, поставить — нельзя. Где живёт кнопка «Зафиксировать» — решается
  в инвентаре действий (§9).
- У **справочников** (изделие, поставщик) фиксации нет — только контурная
  пара замка. Закрытый проект ведёт себя как документ.
- **Окружение** (проекции линзы) — read-only всегда, независимо от всего.

## 6. Редактируемость полей

Принцип: **редактируемо всё, что хранится в документе, — включая автора,
даты и структурные якоря** (проект документа, поставщик УПД, изделие-цель
комплектации). Жизнь заставляет: документ заводит один, а отвечает другой;
дату правят задним числом; поставку заводят не на тот проект. Исключений
ровно три: суррогатный **id**, **вычисленные проекции линзы** (правятся в
документе-факте) и поля **зафиксированных** документов — до снятия фиксации.

Смена якоря — обычное поле шапки под замком формы, но с последствиями по
данным (лоты поставки следуют за её проектом и т.п.): движок валидирует и
**отказывает с объяснением**, когда перепривязка уже небезопасна (зависимые
в других документах). Это ослабляет правило 1 UI_PRINCIPLES («якоря инлайн
не правятся») — детали по каждому якорю зафиксирует матрица полей.

Подтверждённые дыры, закрываемые чистовым проходом:

- **Состав изделия (BOM)** — на форме изделия: редактируемая таблица «Состав»
  (компонент + количество, призрачная строка). Where-used остаётся read-only
  окружением. Сейчас BOM вообще не редактируется из интерфейса.
- **Состав проекта (потребность, `ProjectDemand`)** — на форме проекта:
  изделия + количества, та же механика.
- **Свойства изделия** (код, название, вид, единица) — под замком формы.
- **Шапки документов** (дата, номер, комментарий) — под замком формы.
- **Свойства лота** (цена, название из УПД, заводской номер, примечание).
- **Справочники** (поставщик, проект: реквизиты, бюджет, статус, название).

**Раздел ответственности замка (принят 2026-07-05).** На справочных формах
(изделие, проект) замок §5 закрывает **реквизиты/идентити в шапке** (у изделия —
артикул, название, вид, ед.изм., оценка, «производимое»; у проекта — название,
бюджет, дата начала). **Операционные списки — состав (BOM) и потребность
(`ProjectDemand`) — живут без замка**, правятся прямым автосейвом (как строки
кокпитов). Логика: списки — ежедневный рабочий инструмент, идентити — редкая правка,
которую бережём замком. Фиксации (posted/closed) у справочных форм нет — только
интерфейсный замок. Реализовано: `estimated_cost` изделия и `budget` проекта — под
замком; BOM/потребность — живые.

`[ЗАКРЫТО Ф2i]` **Матрица полей** «форма × поле → всегда / под замком / read-only /
скрыто» — [FIELD_MATRIX.md](FIELD_MATRIX.md). Шесть документов-близнецов схлопнуты в
**один row-set единого «Ордера» (`StockDocument`)** с видимостью полей по `kind`
(раздел B матрицы: шапка + строка/born-лот) — это же спец единой формы `<OrderForm
kind=…>`. Канон ордера:

- **Шапка** (`project`/`user`/`date`/`number`/`note` — общие в MTI-родителе;
  `contractor`/`purchase`/`target_item`/`qty`/`reason` — специфика ребёнка): всё
  редактируемо под замком; `date`+`number` обязательны для всех видов кроме
  комплектации (`REQUIRED_HEADER_BY_KIND`, Ф2d); `project`/`contractor`/`target_item` —
  **якоря** ⚓ (движок валидирует, может отказать при зависимых).
- **Строка**: рождение лота (`item`/`qty`/`unit_cost`/`part_number`/`lot_name`,
  `predecessor` у инвентаризации) — редактируемо; ссылка-якорь (`lot`/`source_lot`/
  `component`) — класс `＋` (задаётся при добавлении, смена = удалить+добавить);
  перемещение несёт `from_location`/`to_location`.
- **Замок** — единый мягкий `status {draft⇄posted}` на всех видах; окружение (проекции
  линзы) read-only всегда.

Остаток очереди — «Свод расхождений» #A матрицы: вывести правку якорей/авторства шапки
на форму (поля в модели есть, форма пока не показывает).

## 7. Списки режима

Единый шаблон для всех десяти режимов:

- Первым — призрачный пункт **«＋ Новая ‹сущность›»** (правило 1 канона).
- Строка списка: `глиф · моно-код` — везде одинаково. **Подписи нет** (не
  читалась, грузила интерфейс); полное название живёт на форме, а
  фильтр-строка ищет в том числе по названию, хоть оно и не показано.
- **Глиф есть у каждого режима:**
  - *документы* — глиф статуса из словаря (● черновик, 🔒 зафиксирован…);
  - *изделия* — серый **codicon вида** (вид — не статус, цвет не нужен):
    `circuit-board` компонент, `tools` производимое, `package` прибор;
  - *проекты* — глиф **от движка-линзы**: ▲ есть дефицит к заказу,
    ● заказано — ждём поступления, ✓ потребность покрыта, 🔒 закрыт.
    Список проектов тем самым — мини-сводка линзы.
- Сортировка: **документы — новые сверху** (зафиксированные приглушены),
  **справочники — по коду**.
- Группировки по проекту нет («все УПД одинаковые»).
- Над списком: **фильтр-строка** (сужение по коду/названию по мере набора)
  и **дропдаун-фильтр по проекту** — во всех режимах, где у объекта есть
  проект (сценарий: при закрытии проекта прощёлкать все его поставки);
  для многопроектных Закупок-план фильтр читается как «затрагивает проект».

## 8. Навигация и поиск

- **Переходы по связям**: любое упоминание объекта (лот в строке поставки,
  изделие в дефиците, документ-origin) — кликабельно, ведёт на его форму.
  Последовательно везде, а не местами.
- **История**: «назад» к предыдущей форме (уже есть) — сохраняется.
- **Палитра ⌘K**: глобальный поиск-переход по коду/номеру/названию из любого
  места; позже туда же лягут команды-действия.

## 9. Действия над объектами

`[ОТКРЫТО]` Где живут операции (провести, закрыть, удалить, дублировать…) —
решение отложено до **инвентаря действий**: Claude собирает по каждой форме и
каждой строке полный список возможных операций (из кода и движка).
Правило решения: если на строку/форму набирается ≤3 действия — кнопки на месте
допустимы; если больше — проектируем меню (модель TortoiseGit). Правый клик в
вебе смущает — без инвентаря не вводим.

- **Только тёмная тема**, палитра — **VS Code Dark 2026** (новый дефолт VS Code

  `index.css`). Источник значений — `extensions/theme-defaults/themes/2026-dark.json`

  | `--bg-widget` | поповеры/виджеты | `#202122` |
  | `--bg-hover` | hover строки | `#FFFFFF14` (альфа) |
  | `--bg-sel` | выбранная строка | `#FFFFFF22` (альфа), текст `#EDEDED` |
  | `--border` | границы панелей/инпутов | `#2A2B2C` / `#333536` |
  | `--fg` | основной текст | `#BFBFBF` |
  | `--fg-dim` | приглушённый | `#8C8C8C` |
  | `--fg-bright` | акцентированный текст | `#FFFFFF` |
  | `--accent` | фокус, бейджи, ссылки | `#3994BC` (ссылки `#48A0C7`) |

  🔒 читается как «зафиксировано» (бывш. «заморожено замком»), поправить в

  на более тёмном фоне их контраст только растёт — перекалибровка на глаз
  при чистовом проходе.
- Цвет — только смыслу; монохромные Codicons это правило поддерживают.

## 12. Заголовки секций внутри формы

Внутри формы объекта (не путать с заголовком формы §4) секции — это `.section-h`:
приглушённая капитель с нижней границей. Правило:

**Заголовок секции — одна строка: НАЗВАНИЕ + мета-информация.** Не описание того,
что это, а счётные факты о содержимом секции — то, ради чего на неё смотрят.
Вторых строк-подзаголовков под секцией не заводим.

```
ПРИБОРЫ        типов приборов 2 · всего приборов 13
ПОТРЕБНОСТЬ    типов компонентов 5 · всего штук 1020
СКЛАД ПРОЕКТА  живых лотов 4 · аномалий 1 ▲
```

- Название — капитель (как сейчас `.section-h`); мета — приглушённый `.hint`
  справа в той же строке, Inter, разделитель `·`.
- Мета = «сколько типов · сколько всего» (или ключевой статус секции: «пусто»,
  «закрыт {дата}»). Числа — счётные; аномалии/предупреждения — красным хвостом.
- Формы существительных фиксированные («приборов», «компонентов», «лотов») —
  не склоняем по числу, читается ровно при любом значении.
