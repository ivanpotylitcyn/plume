# Принципы UI — Plume

Канон интерфейса PLM-системы Plume. Это **вторая попытка**: в первой провалился
именно UI, поэтому интерфейс — зона риска №1. Документ фиксирует согласованные
принципы, чтобы все страницы и формы выглядели и работали одинаково — одни и те же
контролы, одна логика.

> Источник правды по модели данных — диаграммы в [README.md](../README.md). Этот
> документ — источник правды по **интерфейсу**. Меняем принцип — правим здесь.

## Северные звёзды

- **VS Code** — лаконичность, плотность информации, клавиатура, статус-бар, боковые
  «лупы» (Explorer / Search / Git — разные представления одного репозитория).
- **TortoiseGit** — внешне «из 90-х», но непобедим по эргономике: **много
  информации, очень дозированный цвет, богатые (вложенные) контекстные меню**, через
  которые решаются все сложные операции.

Общий вектор обоих: **профессиональный инструмент, а не «школьный сайт»** — мощь в
меню и клавишах, а не в крупных ярких кнопках. Целимся в инвариант
(плотность + дозированный цвет + мощь в меню/клавишах), не в моду.

## Анти-паттерны (что убило прошлый UI — не повторять)

- ❌ Отдельные виды «просмотр» и «редактирование» одной формы.
- ❌ Кнопка «редактировать»/«удалить» на **каждой строке** таблицы.
- ❌ Шапка формы, которая уезжает влево или скрывается при редактировании.
- ❌ Бедная форма (2–3 поля) одинокой коробкой по центру пустого экрана — выглядит
  нелепо.
- ❌ Огромные яркие bootstrap-кнопки; всё крупное давит и съедает место под данные.
- ❌ Поиск/контролы во треть экрана. Крупное = враг.

## Канон

### 1. Один экран, без раздвоения «просмотр/редактирование»
Объект всегда показан в одном виде; всё редактируется **на месте** (inline). Нет
перехода в «режим редактирования». «Замороженный» замком объект (закрытый
проект/акт/приход, см. README про мягкий замок) просто получает **read-only-стиль**
полей — это не отдельная страница.

### 2. Прямое редактирование + автосейв
Редактируем прямо в ячейках/полях. **Чистый автосейв** (`PATCH` по полю — органично
для DRF); большой кнопки «Сохранить» нет. Состояние сохранения — ненавязчиво в
**статус-баре** (`сохраняется… / сохранено / ошибка`), как в VS Code. Это совпадает с
мутабельной ДНК модели: всё — живой черновик, пока не «заморожено» замком.

### 3. Шапка не теряется (детали — к реализации)
Базовая боль: шапка-свойства **не должна уезжать влево или скрываться** при
редактировании строк (грех прошлого UI). Контекст не прячется за тем, что ты делаешь.

При этом «жёсткий sticky» — **не догма**: у «толстого» документа замёрзшая шапка может
съесть половину рабочей зоны. Поэтому закладываем возможность её **сворачивать**
(collapse) для просмотра всех строк. Конкретное поведение (sticky / collapse / порог по
высоте) — **затянуть ближе к реализации**, когда увидим реальные пропорции.

### 4. Чистые строки; действия — в меню и на клавиатуре
В строках таблиц **нет кнопок-действий**. Действия живут на трёх поверхностях:
- **Командная палитра (⌘K)** — быстрый путь, поиск команды по названию.
- **Контекстное меню (правый клик)** — **основная поверхность для операций**,
  богатое и при необходимости вложенное (модель TortoiseGit). Не «подсказка для
  новичка», а рабочий инструмент, где живёт вся сложность.
- **Клавиатура** — горячие клавиши на частые операции (⌫ удалить, ⌘D дублировать,
  ↵ открыть связанный объект).

Иконки-действия по наведению (hover) на строку — **допустимо, но не канон**: уместны,
когда строки плотные и контрол стоит в одном фиксированном месте, куда «подтягивается»
следующая строка (удобно для серийного удаления). Решается по месту, не догма.

### 5. Призрачная строка вместо кнопки «Добавить»
Новая пустая строка **появляется сама**, когда заполнена последняя пустая (модель
Excel). Кнопки «+ Добавить строку» нет.

### 6. Экран объекта = панель свойств + окружение из связей
Объект **никогда не показывается одинокой формой по центру**. Шаблон любого экрана:
**компактная панель свойств** (узкая, лево-выровненная, пары «label: value» — как
Settings/Properties в VS Code) **+ окружение из связанных данных**, заполняющее
остальное место. Мало полей у объекта — не беда: рядом всегда полезное.

Примеры окружения:
- **Изделие** (`Item`, мало полей): свойства + «где применяется» (where-used по
  составу) + «партии на складе» + datasheet'ы.
- **Документ** (Приход/Комплектация/Передача…): шапка-свойства + строки + вложения
  (сканы).

Пусто не бывает, потому что объект всегда показан **в контексте своих связей**.

Окружение — это **проекции линзы (read-only)**: их не правят в представлении, правят
в документе-факте. Закрытость от редактирования служит подсказкой «это вычислено».

**Загрузка окружения — ленивая, без действий пользователя.** Свойства и то, что уже
лежит в строке, показываем сразу; связи **подтягиваются сами**, постепенно (как
человек сам подхватывает контекст формы не мгновенно). Кнопок «подгрузить» избегаем —
бесят; вводим только если заболит. Исключение — **особо тяжёлые редкие запросы**:
страница грузится с **заряженным, но не спущенным** запросом в свёрнутом аккордеоне;
9 из 10 раз он не нужен, на 10-й осознанно триггерим и терпеливо ждём отчёт ~3 сек
(руками по Excel это были бы часы).

### 7. Цвет = только смысл, дозированно
Цвет тратим **исключительно на смысл** (как git-decorations в VS Code / дозированный
цвет Tortoise), приглушённый, не bootstrap-кислота. Статус показываем **значком +
цветом текста/тонкой полосой слева**, а не заливкой фона и не «надувной плашкой» —
чтобы плотный грид не рябил.

Сквозной словарь статусов (единый везде):

| Знак | Цвет | Смысл |
|------|------|-------|
| `●`  | оранжевый | черновик / WIP / есть несохранённое или несверённое |
| `▲`  | красный   | дефицит / чего-то не хватает / нужно купить |
| `✓`  | зелёный   | покрыто / готово / лот есть |
| `🔒` / серый | серый | заморожено замком (закрыт проект/акт/приход) — read-only |
| `○`  | синий     | информационный / нейтральный статус |

### 8. Модальные окна — по минимуму
Модалка (всплывающее окно поверх, блокирующее фон) — **только** для:
- подтверждения **необратимого** (удаление с файлами; переоткрытие узла с потомками —
  «поедут N зависимых»);
- быстрого **создания мелкой сущности** (новый Item/поставщик), не уходя с экрана.

Для основного редактирования модалок нет (иначе это протаскивает назад грех
«просмотр vs редактирование»). Остальное — inline-панели/поповеры.

## Каркас приложения

**VS Code-подобный**:
- слева узкий **activity-bar** (иконки-режимы = разные «лупы» на одни данные);
- рядом сворачиваемая **панель-дерево/список**;
- в центре **рабочая область** (на MVP — один активный документ, **без вкладок**;
  автосейв + быстрое открытие делают вкладки необязательными — вернёмся, если
  заболит);
- снизу **статус-бар**: состояние сохранения + сводка линзы (дефицит / бюджет).

### Навигация — по сущностям, проект как измерение
Корень навигации — **сущности** (Справочник изделий, Склад, Закупки, Приходы,
Комплектации, Передачи, Отчёты), потому что все проекты ведутся разом и содержат
одинаковые изделия. **Проект — фильтр/измерение поверх**, не корень дерева.

Отдельно — **режим «изоляции в проект»** (своя «лупа» в activity-bar, по аналогии с
Git-режимом VS Code: тот же репозиторий — другое представление): сфокусироваться на
одном проекте для чистки остатков перед закрытием.

## Помощник-линза и панель управления проектом

**Линза — не отдельный экран, а свойство всех форм** (принцип №6) плюс **один
вычислительный движок** (СППР: что заказать? сколько? успеваем? закрываемся?). Две
проекции одного движка под общим словарём статусов: **снизу вверх** (Изделие/Партия —
где применяется, остатки) и **сверху вниз** (Проект — состав, дефицит, бюджет).
Проекции read-only.

**Панель управления проектом** — главный экран-«лупа» в activity-bar и **точка старта
действий** (тот самый «ведёт», не просто «всё видно»). Отсюда стартуют
автоматизированные **большие движения**, которые потом дотачиваются руками в формах
документов:

| Этап | Действие из панели |
|------|--------------------|
| Запустить   | завести бюджет (`Project.budget`) |
| Планировать | потребность в приборах (`ProjectDemand`) |
| Планировать | заполнить закупки, перевести в заказ (`Purchase`) |
| Вести       | получать лоты по УПД (`Receipt`), комплектовать (`Kitting`) |
| Чистить     | подбивать лоты по актам комплектации |
| Закрывать   | свести остатки в 0 + замок на связанные документы |

Закрытие наступает **само** как итог аккуратного управления, а не отдельной
разрушительной операцией. Проектируется с макетами. См. README (раздел про
помощника-линзу) и память `order-assistant-lens`.

## Конформанс режимов и кокпитов (живой чек-лист)

Принципы выше — про то, *как* всё выглядит. Эта секция — про то, *что* каждый режим
и кокпит обязан **предоставлять** (инвариантные требования). Ведём её как **живой
чек-лист**: фиксируем правило один раз здесь, а фактические отклонения (исторические
хвосты read-only волн, ситуативные различия кокпитов) **накапливаем по ходу волн и
подтягиваем одним чистым проходом, когда MVP будет на месте** — так меньше артефактов
и кода, больше стройности, чем от точечных правок между волнами.

> Договорённость (2026-07-03, Иван): не фиксим отклонения ситуативно. Копим
> функциональность волнами, затем — одна синхронизация «по канону». Реализуя **новый**
> экран/режим, сразу соблюдаем правила отсюда; **старые** приводим к канону в проход.

### Правило 1 — «＋ Новая» в каждом режиме

**Требование.** У **каждого** режима activity-bar, где живут создаваемые сущности,
первым пунктом дерева/списка закреплён призрачный пункт **«＋ Новая ‹сущность›»**.
Единый способ завести объект — не только через админку.

**Поведение по клику:**
1. выбирается состояние `new-*` (ещё не персистентно, в дереве подсвечено);
2. в рабочей области — **компактная форма создания** (панель свойств, принцип №6: не
   одинокая коробка; только **минимально необходимые** поля);
3. по «Создать» — `POST`; при успехе **открывается кокпит/экран созданного объекта** и
   список дерева перезагружается (объект появляется).

**Два вида режимов:**
- **Документы** (Комплектации / Приходы / Заказы / Передачи / Списания / Требования /
  Закупки-план): создаётся **черновик**; строки набираются **потом в кокпите**, не в
  форме создания. Структурные якоря (проект документа, поставщик УПД) выбираются **при
  создании** и дальше инлайн не правятся (переселение лотов — отдельная операция).
- **Справочники** (Проекты / Изделия): создаётся справочная сущность из её мини-формы;
  открывается её экран (дефицит проекта / экран изделия). Якорь-`kind`: «＋ Новый
  проект» = только внешний (НИР/контракт; внутренние WHITE/GREY — синглтоны из сида).

**Статус конформанса** (обновлять при каждой волне и при чистовом проходе):

| Режим | «＋ Новая» | Примечание |
|-------|:---------:|-----------|
| Комплектации, Приходы, Заказы, Передачи, Списания, Требования, Закупки-план | ✓ | эталон паттерна |
| Проекты | ✓ | приведено к канону 2026-07-03 (было read-only с волны 1) |
| Изделия | ✓ | приведено к канону 2026-07-03 (было read-only с волны 1) |

_Следующие правила канона (форма шапки, набор действий кокпита, контекстное меню,
клавиши) — заводим по мере того, как отклонения становятся видны глазом._
