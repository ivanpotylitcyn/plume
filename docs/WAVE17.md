## Развилки (СНЯТЫ с Иваном 2026-07-17)

1. **Семантика `Item.status` — «точно как в StockDocument»: `posted` = ФИКСАЦИЯ.**
   `posted` → чип «🔒 зафиксирован», форма read-only целиком (свойства **и** BOM),
   удаление блокируется. `draft` → эфемерный мягкий замок 🔓/🔒 (правишь открыв, как
   сейчас). Проводка — точь-в-точь как у складских документов ([ReceiptView.tsx:29-41](../frontend/src/ReceiptView.tsx#L29)):
   `fixed = status==='posted'`, `locked = fixed || !unlocked`; кнопка «зафиксировать»
   (видна при `!fixed`, disabled пока открыт мягкий замок → сперва закрой), `onUnfix`
   расфиксирует (→ draft). Синк библиотеки ставит `posted` напрямую.

2. **Ось «Изделие» = `produced=True`** (та же, по которой уже фильтруют
   `AddDemand`/`AddComponent`). Ортогонально категории; сейчас пользователи руками
   держат `produced=True` в синхроне с категорией «Изделие» — **пока не усложняем,
   так и оставляем**. Новый режим «Изделия» фильтрует по `produced=True`.
   **Важно:** в новом режиме при создании нового Item по умолчанию `produced=True`
   (снимает боль ручного выбора типа каждый раз).

3. **Миграция существующих: все → `draft`** (дефолт нового поля). Отдельного маркера
   «из библиотеки» ретроактивно нет → прогон «Синхронизация с библиотекой» после
   деплоя проставит библиотечным `posted`. Чисто, без угадывания.

## Объём — четыре блока (Иван 2026-07-17)

### Блок 1 — статус изделия (`Item.status`)
- Добавить `Item.status {draft ⇄ posted}`, дефолт `draft`, **переиспользуя
  существующий `DocStatus`** (объявлен в [models.py:60](../backend/plume/models.py#L60)
  для StockDocument — draft/posted, ровно то, что надо). `is_posted`-property.
- **Пока бинарный** draft/posted — новых сущностей не плодим.
- **Библиотечные (импорт из ant-pcb-lib) явно `posted`**; статус **синхронизируется
  при обновлении библиотеки** (`apply_library_diff` ставит `posted` на create И на
  changed). Все прочие (не из библиотеки) — `draft`.

### Блок 2 — форма изделия и её мягкий замок
- **Мягкий замок скрывает редактирование `BomLine`** — под замком (и под фиксацией)
  «Состав (BOM)» = просто read-only список (сейчас секция редактируема всегда,
  игнорируя замок — [ItemView.tsx:149-175](../frontend/src/ItemView.tsx#L149)).
- В списке «Состав (BOM)» переименовать колонку **«Назв.» → «Описание»** (`description`).
- В этом поле показывать **ровно одну строку**, остальное скрывать многоточием
  (`допуск …`) — CSS-усечение (`nowrap` + `text-overflow: ellipsis`).

### Блок 3 — режим «Изделия»
- В левую панель режимов добавить режим **«Изделия»** — вторая копия текущего режима
  «Изделия».
- Текущий режим переименовать в **«Компоненты»** и оставить **ровно как есть** (все
  Item, фильтр по категории, кнопка синка с библиотекой).
- Новый режим «Изделия» — те же функции, **но:** нет фильтра по категории (всегда
  включён фильтр `produced=True`), нет кнопки синка с библиотекой. NewItem по
  умолчанию `produced=True`.
- Бэкенд-движок режима переиспользуем как есть; во вью добавить жёсткой конкретики
  (фильтр `produced`, дефолт при создании).

### Блок 4 — статус-глиф в строках Item
- Слева от компонента отображать `status`: `posted` — **зелёная галочка ✓**,
  `draft` — **оранжевый кружок ○**.
- Такой же статус во **всех** местах со строками Item:
  - режимы «Изделия» и «Компоненты» (списки слева);
  - «Состав (BOM)» в форме изделия ([ItemView.tsx](../frontend/src/ItemView.tsx));
  - «Потребность» в проекте ([DeficitView.tsx](../frontend/src/DeficitView.tsx) —
    `CompRow` + дерево `TreeRow`).
