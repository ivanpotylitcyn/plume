# Волна 9 — инвентаризация (`Inventory`): 4-й origin партии + серая ре-материализация

Чеклист девятой волны. Цель — замкнуть инвариант **«лот всегда из origin-документа»**:
`Inventory` — четвёртый и последний origin (`receipt`/`kitting`/`requisition`/**`inventory`**).
Инвентаризация рождает **«найденные» партии** — излишки, всплывшие при пересчёте, и
**ре-материализацию серых остатков** (списанное ушло `−ISSUE` «в серый»; нашли физически
→ вернули на баланс новым лотом с провенансом). Закрывает дыру, отмеченную с В6:
«списание в GREY ничего не кладёт — серую ре-материализацию даст `Inventory`». Продолжает
паттерн волн 2–8 (записываемая надстройка + движок = единственный источник правил +
витрина-проекция). Значимые решения — в [JOURNAL.md](JOURNAL.md) (запись 2026-07-03, «Волна 9»).

**Цель волны 9:** новый режим ⛭activity-bar 🔍 «Инвентаризации» со списком + кокпитом.
В кокпите строки акта = «найденные» лоты (`+RECEIPT`): излишек добавляется призрачной
строкой (изделие + кол-во + цена + название), а серый остаток возвращается на баланс через
панель **«Ре-материализация из списанных»** (пикер списанных лотов → born-лот с
`predecessor` → списанный, наследующий item/цену/название/зав.№).

## Объём

**Входит:** движок инвентаризации (`inventory_cockpit` — проекция;
`create_inventory` / `add_inventory_lot` (born `+RECEIPT`, опц. `predecessor`) /
`update_inventory_lot` / `remove_inventory_lot` / `update_inventory` — правка шапки;
`written_off_lots` — пикер ре-материализации) → DRF (`inventories`/detail/`lots`/
`inventory-lots`/`written-off-lots`) → фронт: новый режим `InventoryView` (кокпит +
призрачная строка + панель ре-материализации) → юнит-тесты + HTTP-smoke.

**НЕ входит (следующие волны):**
- Два числа бюджета + себестоимость/экономия (north-star окупаемости линзы).
- Логин-экран (аутентификация SPA; автор пока — дефолтный суперюзер).
- UI вложений (`Attachment` — PDF/сканы УПД/актов к документам).

## Решения (проектные — в JOURNAL 2026-07-03, «Волна 9»)

- **Схему БД не трогаем** (как В6–В8). `Inventory` уже в модели (заморожена);
  `LOT_ORIGIN_FIELDS`/`rebuild_movements` знают origin `inventory` с волны 1. Волна 9 —
  только записываемая надстройка. Диаграммы README **не меняются** (нет изменения схемы).
- **Отдельной `InventoryLine` нет** — строки акта это его лоты (`inventory.lots`, как
  приход/УПД без `ReceiptLine`). Кокпит зеркалит `receipt_cockpit`: изделие + кол-во +
  цена + название + зав.№, автосейв, born-лот = `+RECEIPT` в проекте акта.
- **Замка нет** (у `Inventory` нет поля-статуса, как у `Writeoff`/`Requisition`): правимо
  всегда; корректность держат guard'ы (`qty>0`, цена ≥0, `_lot_consumed_downstream` на
  удалении) + `PROTECT`. Кол-во не клампим (мутабельная ДНК).
- **Провенанс через `predecessor`** (как требование): при ре-материализации born-лот
  ссылается на списанный-источник, наследуя item/цену/название/зав.№ (генеалогия серого
  остатка). Излишек (без источника) — `predecessor=None` («излишек»).
- **Серая ре-материализация — payoff волны.** `written_off_lots()` = пикер списанных
  лотов (суммарно списанное с лота = сколько «серого» доступно вернуть). Дом
  ре-материализации по умолчанию — GREY «Свободные неучтённые» (где серый остаток и
  «живёт»); оттуда его забирают в реальный проект требованием (`Requisition`, В6).
  Излишек-инвентаризацию можно завести и в реальном проекте.

## Этапы

### Этап 1 — Движок инвентаризации (чистый Python + тесты) ← ядро волны
- [x] `inventory_cockpit(inv)` — шапка (№/дата/примечание/проект) + строки-лоты
  (`qty`/`live_qty`/цена/название/зав.№/провенанс/`consumed`) + `total_cost`.
- [x] `create_inventory` + `add_inventory_lot` (born `+RECEIPT`, опц. `predecessor`) +
  `update_inventory_lot` + `remove_inventory_lot` (guard `_lot_consumed_downstream`) +
  `update_inventory` (правка шапки, `_apply`/`_require_number|date`).
- [x] `written_off_lots()` — пикер ре-материализации (лоты с `writeoff_lines`, суммарно
  списанное, item/цена/название/зав.№).
- [x] Юнит-тесты (7, `InventoryCockpitTests`): born `+RECEIPT` origin=inventory; итог/
  примечание; guard пустого №; guard qty≤0 / цена<0; правка+удаление; блок удаления
  потреблённого; ре-материализация (пикер + born с `predecessor` + наследование цены/зав.№).
- _Готово, когда:_ тесты зелёные; живой остаток born-лота = рождённому. ✓

### Этап 2 — Записываемый DRF
- [x] `GET/POST /api/inventories/` (список/create), `GET/PATCH /api/inventories/<pk>/`
  (кокпит/шапка), `POST .../lots/` (строка; `predecessor_id` → наследование),
  `PATCH/DELETE /api/inventory-lots/<pk>/`, `GET /api/written-off-lots/` (пикер).
- [x] HTTP-smoke (`InventoryHttpTests`, 2): create/строка (+RECEIPT, total)/qty=0 400/
  правка шапки; ре-материализация через пикер (predecessor + наследование цены/зав.№).

### Этап 3 — Фронт: режим «Инвентаризации» (кокпит + ре-материализация)
- [x] Десятый режим activity-bar 🔍 «Инвентаризации»: список + «＋ Новая инвентаризация»
  (проект-дом по умолчанию GREY + № + дата + примечание).
- [x] `InventoryView` — кокпит: строки-лоты автосейвом/удалением, призрачная строка
  «излишек» (item-пикер), колонка «провенанс» (← списанный / «излишек»), правка шапки;
  панель «Ре-материализация из списанных» (пикер → «вернуть на баланс»).
- [x] API-клиент: типы (`InventoryRow`, `InventoryCockpit`, `InventoryCockpitLot`,
  `WrittenOffLot`) + мутирующие хелперы; статус-бар → «волна 9» + счётчик инвентаризаций.
- _Готово, когда:_ заводишь акт → добавляешь найденный лот / возвращаешь серый на баланс
  → provenance виден. ✓
