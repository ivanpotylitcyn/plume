# Волна 12 — логин-экран + аутентификация (финальная волна MVP)

Чеклист двенадцатой волны. Цель — закрыть **единственный оставшийся** отложенный пункт:
аутентификацию. До В12 весь `/api/` открыт, автор документа — fallback-суперюзер
(`_actor`), фронт анонимен. В12 ставит весь API за логин, делает авторство честным
per-user и даёт вход/выход из UI. Продолжает паттерн волн 2–11 (надстройка без правки
схемы). Значимые решения — в [JOURNAL.md](JOURNAL.md) (запись 2026-07-04, «Волна 12»).

**Модель авторизации (решение пользователя):** **один уровень доверия** — любой
залогиненный = полноправный оператор PLM. Ролей/прав нет (малая доверенная команда);
ролевую детализацию можно ввести позже отдельной волной, если понадобится.

## Объём

**Входит:** DRF-дефолты (`SessionAuthentication` + `IsAuthenticated`) → auth-эндпоинты
(`me`/`login`/`logout`, `AllowAny` где нужно) → упрощение `_actor` (fallback убран) →
`ping` открыт (health-check) → фронт: `Login.tsx` + гейтинг в `App.tsx` + глобальный
хук 401 + кнопка выхода → тесты (`force_login` в 7 HTTP-классах + `AuthHttpTests`).

**НЕ входит (уже не «волна», а полировка/эксплуатация):**
- Ролевая авторизация (staff/обычный, гейтинг режимов) — сознательно нет (один уровень).
- Чистка хвоста фронта (накопленный конформанс режимов/кокпитов, forward-жест истории).
- Деплой В12 на прод (отдельной экскурсией, как В6–В11).

## Решения (проектные — в JOURNAL 2026-07-04, «Волна 12»)

- **Схему БД не трогаем** (как В6–В11): встроенный Django `User`, миграций нет
  (`makemigrations --check` чист). Диаграммы README **не меняются**.
- **Сессионная механика, не JWT/токены.** Внутренний инструмент, SPA с того же origin на
  проде, сессии в БД — никакого Redis/долгоживущих процессов (под ограничения reg.ru
  shared). `SessionAuthentication` + `IsAuthenticated` глобально → весь `/api/` за логином
  **одним переключателем** (включая `attachment_download` — В11 это предвидел).
- **`AllowAny` только там, где обязано:** `auth/me` (401 или юзер; ставит CSRF-cookie),
  `auth/login` (400 на неверный пароль), `ping` (health-check). `logout` — под логином.
- **Сам-регистрации нет** — юзеров заводят в admin.
- **CSRF реально форсится под сессией** (`SessionAuthentication.enforce_csrf`). Фронт уже
  слал `X-CSRFToken` из cookie (задел волн 2/11) — сомкнулось. `login` от анонима CSRF не
  требует (SessionAuth не форсит для неаутентифицированного).
- **`_actor(request)` → `request.user`** (fallback на суперюзера убран) — авторство честное.
- **Фронт-гейт:** `user: User | null | undefined`; данные грузятся только под логином;
  401/403 в любом запросе → назад на логин (`setUnauthorizedHandler`).
