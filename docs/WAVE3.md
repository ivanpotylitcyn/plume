# Волна 3 — записываемый приход (УПД → рождение лотов)

Чеклист третьей волны. Цель — замкнуть **входную сторону** потока: завести УПД
прямо в UI и наполнить склад лотами, чтобы кокпит комплектации (волна 2) питался
не только из `seed_demo`/admin. Продолжает паттерн волны 2 (записываемый документ
+ движок = единственный источник правил + `rebuild_movements`). Значимые решения —
в [JOURNAL.md](JOURNAL.md) (запись 2026-07-02, «Волна 3»).

**Цель волны 3:** открыть приход → вести строки УПД (изделие + кол-во + цена +
название) → рождаются лоты (`+RECEIPT`) на складе проекта → они сразу видны кокпиту
комплектации. Замок «сверено со сканом» (`approved`) морозит форму.

## Объём

**Входит:** проекция кокпита прихода (`receipt_cockpit`) → записываемые
DRF-эндпоинты (добавить строку / автосейв / удалить / замок) → справочник
поставщиков (пикер + быстрое создание) → четвёртый режим фронта «Приходы» с
созданием и кокпитом → юнит-тесты на мутации.

**НЕ входит (следующие волны):** `Transfer`/`Writeoff`/`Requisition`/`Inventory`,
логин-экран, командный свод + `Procurement`, два числа бюджета + себестоимость/
экономия, pegging, связь `Receipt↔Purchase` (закрытие `PurchaseLine`), UI вложений.

## Решения (см. JOURNAL 2026-07-02)

- **Схему БД не трогаем** — `Receipt`/`Lot`/`Supplier` уже были.
- **Строка УПД = `Lot`.** Отдельной `ReceiptLine` в модели нет — строки прихода
  это его лоты (`receipt.lots`). Добавить строку = создать `Lot` с `receipt=этот`.
- **`project` наследуется от `Receipt`** — констрейнт `lot.project == receipt.project`
  уже в модели; в строке не выбирается.
- **`location`** — дефолтный «Основной склад» (`_main_location()`), в форме не даём
  (решение MVP про `Location`).
- **Поставщик обязателен** (`Receipt.supplier`, `PROTECT`): пикер существующих +
  быстрое создание мелкой сущностью (UI-канон «модалка только для быстрого создания»).
- **Мягкий замок `approved`** (мысль зеркалит `Kitting.closed`): `approve`/`unapprove`.
  Важный нюанс отличия от `reopen_kitting`: **снятие замка ничего не разрушает**
  (только разблокирует правку) → guard по потомкам ему не нужен. Guard стоит на
  **удалении строки** (`_lot_consumed_downstream` — лот уже спаян/передан ниже → блок,
  дружелюбно до DB-`PROTECT`).
- **Связь `Receipt.purchase`** (закрытие заказа) — отложена (записываемого `Purchase`
  ещё нет), поле остаётся `null`.
- **Правку кол-ва не клампим** по потреблению: уронить рождённое кол-во ниже
  списанного можно — живой остаток уйдёт в минус (недостача информативнее нуля,
  как в дефиците). Это в духе мутабельной ДНК.

## Этапы

### Этап 1 — Движок прихода (чистый Python + тесты) ← ядро волны
- [x] `receipt_cockpit(receipt)` — шапка УПД + строки-лоты (кол-во/цена/название/
  зав.№ + живой остаток + флаг `consumed`) + сумма.
- [x] Мутации через `rebuild_movements`: `add_receipt_lot`, `update_receipt_lot`
  (кол-во/цена/название/зав.№), `remove_receipt_lot` (guard `_lot_consumed_downstream`),
  `approve_receipt` / `unapprove_receipt`.
- [x] `_lot_consumed_downstream(lot)` — общий хелпер; `reopen_kitting` переведён на него.
- [x] Юнит-тесты: строки-лоты, добавление → `+RECEIPT` в движениях, правка qty
  пересобирает движение, констрейнт `lot.project == receipt.project`, замок блокирует
  правку, guard удаления по потреблению, приход сразу виден кокпиту комплектации.
- _Готово, когда:_ тесты зелёные, склад пересобирается корректно.

### Этап 2 — Записываемый DRF
- [x] `GET/POST /api/receipts/`, `GET /api/receipts/<pk>/` (кокпит),
  `POST .../lots/` (строка), `PATCH/DELETE /api/lots/<pk>/` (автосейв/удаление),
  `POST .../approve/` и `.../unapprove/`.
- [x] `GET/POST /api/suppliers/` (пикер + быстрое создание). Правила — в движке;
  вьюхи мапят `ValidationError` → 400.
- _Готово, когда:_ HTTP-путь через test Client: create 201, строка 201, автосейв/
  удаление 200, замок 200, валидация 400.

### Этап 3 — Фронт: режим «Приходы» + кокпит
- [x] Четвёртый режим activity-bar (📥): список УПД (глиф замка 🔒/●) + призрачная
  строка «＋ Новый УПД» (поставщик + № + дата + проект).
- [x] `ReceiptView` — кокпит: шапка, строки-лоты с автосейвом (кол-во/цена/название)
  и удалением, призрачная строка «добавить строку» (пикер изделия), замок approve/
  unapprove (форма read-only под замком), сумма УПД.
- [x] API-клиент: мутирующие хелперы, типы кокпита прихода, справочник поставщиков.
- _Готово, когда:_ заводишь УПД в UI → лоты появляются на складе и в кокпите
  комплектации; числа сходятся с движком.
