# Волна 10 — бюджет проекта: два числа денег + себестоимость/экономия (north-star окупаемости)

Чеклист десятой волны. Цель — выпустить **payoff всей линзы**: оцифровать пользу PLM.
Не новый учётный контур, а **читающая проекция** поверх уже записываемых лотов (паттерн
волн 2–9: движок = единственный источник правил + витрина-проекция). Формула бюджета
(№6) спроектирована ещё на аудите движка — см. [JOURNAL.md](JOURNAL.md) (запись
2026-06-30, «Формулы движка СППР… №6»), теперь реализуем. Значимые решения — в JOURNAL
(запись 2026-07-04, «Волна 10»).

**Цель волны 10:** в шапке проектного дашборда (`DeficitView`) — панель **«Бюджет»**:
два числа денег (**потрачено** факт / **план** прогноз) + **компас** против бюджета
(запас/перерасход), справа — окупаемость: **себестоимость** (честная цена для КП) и
**экономия** (польза внутреннего заёма = польза PLM). Бейдж «N позиций без оценки».

## Объём

**Входит:** движок бюджета (`project_budget` — проекция; `_project_spent` /
`_project_estimate` / `_project_cost` — слагаемые) → DRF (`GET
/api/projects/<pk>/budget/`) → фронт: панель `BudgetPanel` в шапке `DeficitView` +
форматтер `money()` → юнит-тесты + HTTP-smoke.

**НЕ входит (следующие волны):**
- Логин-экран (аутентификация SPA; автор пока — дефолтный суперюзер).
- UI вложений (`Attachment` — PDF/сканы УПД/актов к документам).

## Решения (проектные — в JOURNAL 2026-07-04, «Волна 10»)

- **Схему БД не трогаем** (как В6–В9). `Project.budget`, `Item.estimated_cost`,
  `Lot.unit_cost` и снимок себестоимости `_device_unit_cost` (на закрытии `Kitting`,
  В2) уже в замороженной модели. Волна 10 — только читающая надстройка. Диаграммы README
  **не меняются** (нет изменения схемы).
- **Два числа денег — не путать (как с qty).** **Потрачено** = `Σ(unit_cost×qty)` по
  приходным (`Receipt`-origin) лотам проекта — точная застывшая цифра «кэша ФЛС».
  **План** = «факт где есть, оценка где нет» = потрачено + `Σ(остаток×estimated_cost)`.
- **Заём/свои бесплатны в бюджете (правило 1).** Лоты origin
  `requisition`/`inventory`/`kitting` в «потрачено» **не** входят (платил источник /
  найдено / это уже собранный узел). Отбор по `receipt__isnull=False` даёт ровно
  «покупные материалы по реальной цене».
- **Только покупной уровень (правило 2).** Снимок цены собранного узла (лот-прибор из
  `Kitting`) в деньги бюджета не складываем (иначе двойной счёт). В плане производимые
  компоненты (`is_manufactured`) в денежную оценку не берём.
- **Позиции без оценки — флаг, не молчаливый 0 (правило 3).** Покупная позиция с
  остатком>0 и `estimated_cost=None` → в списке `unestimated` (бейдж «план неполон»), а
  не тихо 0 в плане.
- **Оценку считаем на «в пути + к заказу» (● + ▲).** ✓ уже покрыто (приходной лот — в
  «потрачено», заём — бесплатно). Потребность агрегируем **по компоненту** (1 уровень
  BOM, как `project_deficit`), один `_coverage` на компонент — без задвоения между
  потребностями. По мере прихода УПД оценки замещаются фактами → **в конце план = факт**.
- **Себестоимость — только верхние приборы.** `Σ(qty×снимок)` по лотам-приборам закрытых
  комплектаций, чьё изделие — **цель потребности** проекта (`ProjectDemand.target_item`).
  Подсборки (тоже `Kitting`, но не цель) не суммируем — их цена уже в снимке верхнего
  прибора (без двойного счёта). Снимок включает заём **по реальной цене**
  (`Requisition`-лот наследует цену предка, В6) — честная цена для КП.
- **Экономия = себестоимость − потрачено** — оцифрованная польза внутреннего заёма
  (= польза PLM). Отрицательна, пока приборы не собраны (только потрачен кэш) — это
  честно (окупаемость проявляется по мере сборки/заёма).

## Этапы

### Этап 1 — Движок бюджета (чистый Python + тесты) ← ядро волны
- [x] `_project_spent(project)` — факт по `Receipt`-лотам (заём/свои = 0).
- [x] `_project_estimate(project)` — прогноз по `estimated_cost` (агрегация по
  компоненту, ●+▲, покупные), + список `unestimated` (коды без оценки).
- [x] `_project_cost(project)` — себестоимость верхних приборов (снимок лотов закрытых
  `Kitting`, `item ∈ target_item потребностей`).
- [x] `project_budget(project)` — сборка: `budget`/`spent`/`plan`/`compass`/
  `unestimated`/`cost`/`economy`.
- [x] Юнит-тесты (5, `ProjectBudgetTests`): потрачено считает только приход (заём=0);
  план=оценка→замещается фактом при приходе (компас сходится); `unestimated` поднят, не
  0; экономия = стоимость заёма (снимок с заёмом по реальной цене); компас=None без
  бюджета.
- _Готово, когда:_ тесты зелёные; в конце проекта план=факт. ✓

### Этап 2 — Записываемый DRF (тут только read)
- [x] `GET /api/projects/<pk>/budget/` — проекция.
- [x] HTTP-smoke (`ProjectBudgetHttpTests`, 1): план=Σ(нужное×оценка), компас=budget−план.

### Этап 3 — Фронт: панель «Бюджет» в шапке DeficitView
- [x] `BudgetPanel` (в `DeficitView` под подзаголовком): два числа (потрачено/план) +
  бюджет + компас (зелёный запас / красный перерасход), справа блок окупаемости
  (себестоимость + экономия, зелёная при >0), бейдж «N поз. без оценки».
- [x] API-клиент: тип `Budget` + `api.budget(id)`; форматтер `money()` в `status.tsx`;
  статус-бар → «волна 10».
- _Готово, когда:_ открываешь внешний проект → видишь два числа денег + окупаемость. ✓

## Проверено

136 юнит-тестов зелёные (130 волн 1–9 + 5 движка + 1 HTTP-smoke волны 10);
`tsc -b && vite build` чист; `makemigrations --check` — **схема не менялась**. Живой
runserver-smoke на MySQL 8.0.25: проект (budget 5000) с прибором (BOM: корпус
`estimated_cost=800` ×1 + винт без оценки ×4), потребность ×3, куплен 1 корпус @ 800 →
`GET /api/projects/<id>/budget/` = `spent 800 · plan 2400 (800 + 2×800 оценка) ·
compass 2600 · unestimated [винт] · cost 0 · economy −800`; smoke-данные удалены за собой.
