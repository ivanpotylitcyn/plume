## 2026-07-10 — Сброс истории миграций в один 0001_initial + разовый прод-reset
**Тип:** решение + грабли
**Контекст:** после WAVE13 накопилось 10 миграций plume (`0002`–`0010` — болтанка
разморозки: MTI/StockDocument, `collapse_arcs`, `lift_common_fields` и т.д.). Иван не
хочет тащить legacy-поля и обратную совместимость с ранними версиями. Встал вопрос:
дропнуть таблицы и пересоздать — это моветон или норма? Прод (поднят 2026-07-01) на
**тестовых** данных, боевые ещё не заводили.
**Итог:**
- **Разобрали ложную дихотомию «дропнуть vs мигрировать».** Это три независимые оси:
  (1) чистота моделей (нет legacy-полей) — правкой `models.py`, к миграциям не относится;
  (2) чистота истории миграций — сквошем/пересозданием, к данным не относится;
  (3) сохранность данных — только здесь живёт «дропать или нет».
- **Принцип: ценим чистоту, до 1.0 не стесняемся делать сквош.** Пока нет боевых данных,
  сброс истории до единого `0001_initial` — легитимная гигиена, а не моветон. `squashmigrations`
  не понадобился: раз БД везде обнуляется, проще удалить файлы миграций и перегенерировать
  (`find backend/plume/migrations -name '0*.py' -delete` → `makemigrations plume`).
- **`User` штатный** (`django.contrib.auth`, в settings нет `AUTH_USER_MODEL`) — юзеры в
  `auth_*`, к миграциям `plume` не привязаны; FK идут `plume → user` односторонне, дроп
  plume-таблиц юзеров не рвёт.
- **Прод-reset (разовый):** бэкап юзеров (`dumpdata auth.User`) → drop всех таблиц →
  деплой WAVE13 по пустой базе (`migrate` штатным `deploy.sh`) → возврат юзеров
  (`loaddata`). Сайт обновлён, юзеры на месте.
- **Грабля shared-хостинга:** у app-юзера reg.ru, скорее всего, **нет** прав
  `DROP/CREATE DATABASE` → базу целиком не пересоздать. Дропали **таблицы** внутри
  существующей БД через Django-коннект (креды из `.env`, без ручного `mysql`):
  `SET FOREIGN_KEY_CHECKS=0` → `SHOW TABLES` → `DROP TABLE` в цикле (сниппет через
  `manage.py shell`). Локально проще — есть root, пересоздавали всю БД
  (`docker exec plume-mysql mysql -uroot ... "DROP DATABASE plume; CREATE DATABASE plume ..."`).
- **Посев в деплое уже opt-in:** `seed_demo` gated за `--seed` в `deploy.sh`/`deploy_local.sh`;
  штатный деплой не сеет. Боевой прод катим **без** `--seed`, данные заводим руками.
- **Валидация сквоша (локально):** `migrate` + `seed_demo` + `check` + `makemigrations
  --check` (No changes) + **263 теста зелёные**. Тестовая БД строится из миграций → зелёные
  тесты доказывают, что единственный `0001_initial` даёт рабочую схему.
**Последствия:** дропы БД имели смысл, пока прод на тестовых данных (сейчас — бета-тест).
Как только начнём заполнять боевые — переходим на дисциплинированные миграции с сохранением
данных, без дропов. Явную пометку «прод на боевых» решили пока не ставить (ещё бета);
вернёмся к ней, когда поработаем с приложением.

## 2026-07-10 — Волна 13, Ф3+Ф4: вьюхи перемещения + склад как сущность
**Тип:** решение + реализация
**Контекст:** согласованный с Иваном объём сессии. Последний dev-остаток волны 13 —
«вьюхи Ф2e» (движок/модель `Relocation`+мультисклад готовы с Ф2e, HTTP+React отложены).
Иван расширил задачу: `Location` — не просто пикер, а **полноценная сущность «Склады»**
как `Item`/`Project` (мягкий замок, мутабельная ДНК, своя форма), а в специфичной части —
«что физически лежит на этом складе с указанием проекта». Чтобы не расползлось, разбили
на две фазы и записали обе в план: **Ф3 (перемещение)** и **Ф4 (склад-сущность)**.
**Итог:**
- **Ф3.** HTTP-слой relocation по образцу Transfer + `update_relocation` (шапка/якорь).
  Ход перемещения = пара знаковых `StockLine` (`−q`@источник/`+q`@приёмник), **ключ хода —
  лот** (один лот = один ход, guard движка), поэтому REST-эндпойнт правки/удаления —
  `relocations/<pk>/lines/<lot_pk>/`, а не по pk строки. `relocation` добавлен в
  `ATTACHMENT_OWNER_MODELS`. React: `RelocationView`/`NewRelocation` на общем
  `useOrderCockpit`; ход собирается из живого лота (дефолт-источник = место с макс.
  остатком) + место-приёмник; `'relocation'` — 7-й вид в едином режиме «Ордера».
- **Ф4.** Движок: `location_stock` (инверсия `lot_locations` — «что на складе», qty>0, с
  проектом-владельцем) + `location_cockpit` + `create/update_location` (код уникален —
  дружелюбный guard до IntegrityError). HTTP: `locations/` GET/POST, `locations/<pk>/`
  GET+PATCH. **Удаления склада нет** — место с движениями бережём (осознанно, не как
  ордера). React: новый режим «Склады» (`ci-database`, рядом с Ордерами), `LocationView`
  с мягким замком §5 (как `Item`) + панель «На складе» (лоты с проектом/остатком).
**Грабли:**
- `update_relocation` изначально поставил **выше** определения `_UNSET` в engine.py →
  `NameError` при импорте. Функции-часовые (`_UNSET`, `_set_author`, `_set_project`)
  определены в середине файла; все `update_*` держим **после** них.
- Codicon `ci-database` (`\eb2c`) не был заведён в `index.css` (там ручной подмножество
  глифов, не весь шрифт) — добавил определение; TTF полный, глиф есть.
**Последствия:** мультисклад активирован сквозным UX (форма перемещения + экран склада).
Схема **не тронута** (миграций нет — только проекции/хелперы/эндпойнты/вьюхи), диаграммы
README без изменений. 263 теста (+15) зелёные на MySQL 8.0.25. Остаток волны 13 —
**только миграция данных на живой прод-базе** (отдельной экскурсией, как В6–В12).

## 2026-07-10 — Волна 13, Ф2k: структурные якоря ⚓ на форме (вторая связка #A)
**Тип:** решение + реализация
**Контекст:** согласованный с Иваном объём сессии — вторая (и последняя) связка «Свода
расхождений #A» матрицы: **структурные якоря шапки** `project`/`target_item`/
`procurement`. Поля есть в модели и подняты в MTI-родителя (Ф2c), но форма их не
показывала как редактируемые (проставлялись только при создании). Ф2j закрыла первую
связку — авторство; здесь взяли якоря тем же вертикальным срезом бэк↔движок↔форма,
зеркаля паттерн `_set_author`. `contractor` (Receipt/Transfer) уже редактируем с Ф2g —
не трогали.
**Итог:** `project` редактируем под замком на **всех ордерах** (Receipt/Kitting/
Inventory/Requisition/Transfer/Writeoff) + Purchase; `target_item` — на Kitting;
`procurement` — на Purchase. Procurement — вне (проект-less, маркер командной высоты).
- **Ключевое решение — якорь ≠ обычное поле.** `project` — **якорь**: `Lot.project`
  выводится из ордера-origin, а строки `StockLine` ссылаются на лоты того же проекта
  (движок гейтит эту чистоту при добавлении: `lot.project_id != doc.project_id` →
  отказ). Поэтому менять якорь можно только у **пустого** ордера — без born-лотов
  (`doc.lots`) и строк (`doc.lines`); иначе рассинхрон `Lot.project ↔ document.project`.
  Решение: `_set_project(doc, project)` — **дружелюбный отказ** «сперва удалите строки/
  лоты» при непустом (а не молчаливый каскад на лоты — лоты живут в одном проекте всю
  жизнь, каскадная перепрошивка их проекта опаснее явного удаления/пересоздания).
- **Движок:** часовые `_set_project`/`_set_target_item` рядом с `_set_author`
  (`_UNSET`→не трогаем; `None`→отказ, FK NOT NULL; тот же id→ноль-оп; иначе пустота-guard).
  `target_item` — тот же принцип (целевое изделие определяет BOM-состав и рождаемый
  прибор → менять только у пустой комплектации). Purchase — своя семантика (не
  StockDocument): смена `project` у заказа со связанными приходами ломает инвариант
  «УПД ↔ проект заказа» → отказ при `purchase.receipts.exists()`; `procurement` —
  свободная перепривязка плана в черновике. `project`/`target_item`/`procurement`
  добавлены в соответствующие `update_*` (унаследовали edit-freeze из `_require_draft`).
  `purchase_cockpit` теперь несёт `procurement_id`.
- **Views:** резолверы `_resolve_project`/`_resolve_item(key)`/`_resolve_procurement`
  (ключ прислан → объект, нет → `_UNSET`; `DoesNotExist` → дружелюбный 400). 7 PATCH
  прокинули `project=` (+ kitting `target_item=`, purchase `procurement=`).
- **Фронт:** общий `<AnchorSelect>` (подписанный `<select>`, держит текущее значение
  видимым даже вне опций — подпись не гаснет под замком) + `<ProjectField>` (кэш
  `api.projects()`, как `AuthorField` кэширует юзеров) в `FormHeader`. Вставлены в
  `hdr-edit` 7 форм рядом с `AuthorField`; Kitting доп. грузит `api.items()` для
  «изделие», Purchase — `api.procurements()` для «закупка».
- **Схема не тронута** (миграции нет — `makemigrations --check` чист; диаграммы не
  меняются). Только проекции/хелперы/резолверы.
**Проверка:** 248 тестов (было 235; +13 `Wave13Fase2kTests` — смена якоря у пустого,
отказ при строках/born-лотах, часовой, `None`-отказ, gate замком, target_item пустой/
занятый, purchase project с/без приходов, procurement, HTTP project_id+target_id и
дружелюбный 400) зелёные на боевом MySQL 8.0.25; `tsc -b`+`vite build` чисты; `check`
0 issues; `seed_demo` ок.
**Последствия:** «Свод расхождений #A» **закрыт целиком** (авторство Ф2j + якоря Ф2k).
Остаток Ф2g+ — вьюхи перемещения Ф2e (HTTP + React + справочник `Location`) и миграция
данных на прод. Прод не трогали (Ф2 на прод — отдельной экскурсией).

## 2026-07-10 — Волна 13, Ф2j: авторство `user` редактируемо под замком (сквозная #A)
**Тип:** решение + реализация
**Контекст:** следующий укус хвоста Ф2g+ (согласован с Иваном). Из «Свода расхождений
#A» матрицы — сквозная авторская дыра: поле `user` (автор документа) есть на всех
ордерах и поднято в MTI-родителя (Ф2c), но форма его нигде не показывала как
редактируемое — автор проставлялся только при создании (`_actor` = залогиненный).
Матрица помечала это «правка ещё не на форме, очередь §свод #A». #A состоит из двух
связок (авторство + структурные якоря `project`/`contractor`/`target_item`); взяли
**первую** — самую частую и цельную — вертикальным срезом бэк↔движок↔форма. Якоря ⚓ —
следующим заходом.
**Итог:** авторство редактируемо под замком на **всех 8** документах (6 ордеров +
Purchase + Procurement).
- **Движок:** часовой `_set_author(doc, user)` (`_UNSET` → не трогаем; `User` →
  выставить; `None` → отказ, автор обязателен — FK NOT NULL). Поднят выше по файлу
  (`_UNSET` переехал перед `update_receipt` — раньше жил между update-функциями, что
  мешало добавить `user=_UNSET` в те, что объявлены выше). `user=_UNSET` добавлен в 8
  `update_*`; каждая уже гейтит черновик (`_require_draft`/`_require_*_draft`) → правка
  авторства унаследовала edit-freeze бесплатно. Все 9 form-кокпитов (вкл. relocation)
  несут `**_author(doc)` = `user_id`+`user_name` (`get_full_name() or username`).
- **Views:** `User = get_user_model()`; эндпойнт `GET /api/users/` (активные, по
  человеческому имени — пикер); `_resolve_author(d)` (ключ `user_id` → `User`, нет
  ключа → `_UNSET`); 8 PATCH прокинули `user=_resolve_author(d)` + ловят
  `User.DoesNotExist` → дружелюбный 400 (не 500).
- **Фронт:** общий `<AuthorField>` в `FormHeader.tsx` (модульный кэш `api.users()` —
  список редко меняется, дёргать в каждой из 8 вьюх незачем; неактивный текущий автор
  держится первой опцией по `user_name`, чтобы подпись не пропала под замком). Врезан в
  `hdr-edit` всех 8 форм. `api.ts`: `UserRow`/`Authored`-миксин (расширил 8 кокпит-типов),
  `api.users()`, `user_id?` в 8 update-параметрах.
- **Схема не тронута** — миграции нет (`--check` чист, диаграммы не меняются): только
  проекции + класс-хелперы + новый read-only эндпойнт.
**Проверка:** 235 тестов (было 227; +8 `Wave13Fase2jTests` — автор в кокпите, Purchase/
Procurement несут автора, смена/часовой-удержание/`None`-отказ, гейт замком проведённого,
`/api/users/` прячет неактивных, HTTP PATCH `user_id` + невалидный→400) зелёные на боевом
MySQL 8.0.25 (Docker) **и** SQLite; `makemigrations --check` чист; `check` 0 issues;
`tsc -b`+`vite build` чисты (37 модулей); `seed_demo` ок + живой shell (кокпит отдаёт
автора). Прод не трогали (Ф2 на прод — отдельной экскурсией). Остаток #A — структурные
якоря ⚓ на форме (`project`/`contractor`/`target_item`/`purchase`).

## 2026-07-10 — Волна 13, Ф2i: свод матрицы полей + единая `<OrderForm kind=…>`
**Тип:** решение + реализация
**Контекст:** второй укус из хвоста Ф2g+ (согласован с Иваном). Два незакрытых долга
флагмана «Ордер»: (1) `FIELD_MATRIX` держал шесть отдельных таблиц документов-близнецов
(B1–B4 + C1–C2), устаревших относительно модели после Ф2c–g (там ещё `serial_number`/
`received_name`, `supplier`, `project`/`user` как read-only на детях); (2) шесть
detail-вьюх «Ордера» несли **одинаковую оболочку** (~40 строк: fetch кокпита + `run`/`del`/
замок/ошибка + `FormHeader` + `AttachmentPanel`), а диспетчер в `App` был шестью условными
ветками. План волны: свернуть матрицу в один kind-row-set и тела вьюх — в одну форму.
**Итог:**
- **Матрица → один row-set.** B1–B4 + C1–C2 схлопнуты в раздел «B. Ордер (`StockDocument`)»:
  две таблицы (шапка + строка/born-лот) с колонкой «Виды» (RCP/KIT/INV/REQ/TRF/WOF/REL) —
  видимость поля определяет `kind`. Сверено с моделью после Ф2c–g: общие поля в MTI-родителе,
  `part_number`+`lot_name`, `contractor` (роли), единый `status {draft⇄posted}`,
  `REQUIRED_HEADER_BY_KIND`, мультисклад. `Purchase`/`Procurement` (C3/C4) оставлены вне
  объединения. Канон свёрнут в `UI_GUIDE §6` (пункт был `[ОТКРЫТО]` → `[ЗАКРЫТО Ф2i]`).
  Остаток очереди — «Свод расхождений #A»: правка якорей/авторства шапки на форме (поля в
  модели есть, форма пока не показывает).
- **Свод оболочки, а не «мегакомпонент».** Тела шести кокпитов **несводимы** (разные
  кокпит-типы, пикеры, API-вызовы) — впихивать их в один `kind`-switch дало бы 1300-строчного
  монстра, хуже раздельных вьюх. Поэтому свели **оболочку**: генеричный хук
  `useOrderCockpit<C>(id, load, cb)` в `FormHeader.tsx` (fetch по id, обёртка мутации `run`,
  дружелюбное `del`, замок формы, ошибка); специфика вида (тело, выражение `fixed`, side-load
  пикеров) осталась во вьюхе и входит колбэками. Все 6 вьюх перешли на хук (−~90 строк
  боилерплейта). **Грабля-развилка:** «одна форма» из плана трактуется как оболочка+тело, а
  не один файл — формулировка «общая шапка + kind-переключаемое тело» это поддерживает.
- **Один вход диспетчера.** Новый `OrderForm.tsx` (`export type OrderKind` + switch по kind)
  заменил шесть условных веток в `App`; App зовёт единственный `<OrderForm kind id …>` с
  колбэком `reloadOrderKind(kind)`. Шесть view-импортов ушли из `App`.
- **Стабильность эффекта:** колбэки/тексты в хуке пересоздаются каждый рендер — держим свежими
  через `ref`, чтобы `useEffect` зависел только от `[id, load]` (перезагрузка ровно при смене
  документа, без лишних ре-фетчей). Побочный плюс: `run` теперь дёргает и side-load (пикеры/
  список заказов освежаются после каждой мутации — раньше приход не освежал `purchases`).
**Последствия:** только фронт — **схема не тронута** (`makemigrations --check` чист, миграции
и диаграммы README не меняются). `tsc -b` + `vite build` чисты (37 модулей, вкл. `OrderForm`);
227 тестов на MySQL 8.0.25 зелёные (бэкенд-контракт не поехал); `seed_demo` ок. Rules-of-hooks
соблюдены (хук — до ранних `return`), висячих ссылок нет. Браузерный смоук в неинтерактивной
сессии не гонялся — покрыто типами+сборкой+тестами+вычиткой. Остаток Ф2g+: вьюхи перемещения
(Ф2e) + прод-миграция; отложенный #A матрицы (правка якорей на форме).

**Тип:** решение + реализация

режим «Ордер» во фронте) не было. План волны требовал **гибрид**: родитель = read-only
обзор, дети = правка по типу.
**Итог:**
- **`StockDocumentAdmin` — read-only витрина**, а не ещё одна редактируемая форма. Три
  `has_*_permission → False` (add/change/delete) + `list_display_links = None` (строки
  некликабельны). Смысл разделения: bare-родителя создавать нельзя (вид штампует ребёнок
  в `save()`), а удаление/правка должны идти через дочерние админки, где стоят инлайны

- **Зеркалит `OrderList` фронта:** `list_filter = (kind, status, project)`,
  `ordering = ('-id',)` (новейшие сверху), `list_select_related` под проект/автора.
- **Почему read-only, а не редактируемый parent:** править шапку через MTI-родителя в
  обход ребёнка = потерять специфику (`contractor`/`target_item`/`reason`) и per-kind

  правка внизу по типу.
**Последствия:** схема не тронута (миграции нет, `makemigrations --check` чист). 227
тестов (+3 `Wave13Fase2hTests` — read-only пермишены, смешанный список видов,

## 2026-07-10 — Волна 13, Ф2g: `Supplier → Counterparty` (роли) + контрагент на передаче
**Тип:** решение + реализация
**Контекст:** второй укус из остатка Ф2 (согласован с Иваном как объём сессии).
Прежний `Supplier` был только стороной прихода; у передачи структурного получателя

передаче структурного контрагента (симметрия «передача = перемещение к внешней точке»).
**Итог:**
- **`Counterparty` с ролями-булями `is_supplier`/`is_customer`**, а не единым `kind`:
  одно юрлицо реально бывает и поставщиком, и заказчиком — два флага честнее категории

  автоматом). Пикеры фильтруют по роли, быстрое создание проставляет роль по контексту.
- **`Receipt.supplier → contractor`** (RenameField, FK цел) + **`Transfer.contractor`**
  (новый nullable FK). Виды `receipt`/`transfer` **не слиты** — born-direct vs StockLine
  остаётся хребтом ясности; общность выражена через `Counterparty` + `kind`-направление
  (как и записано в «Отложено» шапки волны). Nullable у передачи — исторические накладные
  получателя-сущности не имели.
- **Проекции унифицированы на `contractor_*`** (симметрия приход/передача во всех кокпитах
  + purchase-receipts-row), UI-метка расходится по `kind` («Поставщик» на приходе,
  «Заказчик» на передаче) — ключ внутренний, ярлык доменный. Это осознанный отход от
  «API байт-в-байт» прошлых фаз: данный укус по плану включал «api.ts и вьюхи».
- **Часовой `_UNSET` в `update_transfer`:** отличает «поле не передано» (правка №/даты не
  трогает получателя) от «выставить None» (снять заказчика). Нужен, т.к. `_apply`
  пропускает None и не умеет чистить nullable-FK.
- **Эндпойнт `/api/suppliers/` → `/api/counterparties/?role=supplier|customer`** — один
  справочник обслуживает оба пикера (приход берёт поставщиков, передача — заказчиков).
- **Миграция `0010` реверсивна, но роль-флаги лоссовы** на round-trip: reverse сбрасывает
  роли (в 0009 их нет), forward возвращает дефолт `is_supplier=True` — заказчик после
  reverse→forward становится поставщиком. Это аддитивная метаданность, а не FK-связь;

  dev-MySQL) — тот же класс допустимого лосса, что «в каком поле лежал зав.№» у Ф2f.
**Последствия:** 224 теста (+6) зелёные на MySQL 8.0.25 и SQLite. Обе mermaid-диаграммы
README + глоссарий обновлены (`Counterparty`, дуги поставщик/заказчик). Остаток Ф2 (теперь
Ф2g+): admin-гибрид, свод матрицы полей, вьюхи Ф2e (перемещение/Location), миграция на

## 2026-07-09 — Волна 13, Ф2d: условная валидация специфики по `kind`
**Тип:** решение + реализация
**Контекст:** Четвёртый укус Ф2 (по согласованию с Иваном — из 5 кандидатов: валидация /
переименования Lot+Counterparty / relocation+мультисклад / admin-гибрид). Ф2c подняла общие
поля шапки (`project`/`user`/`date`/`number`/`note`) в MTI-родителя, но **осознанно ослабила**
их: `date` → nullable, `number` → blank. Причина — одной колонкой на общий родитель per-kind
NOT NULL не выразить (у 5 видов дата+номер обязательны, у kitting дата nullable, поля номера
нет вовсе — см. reverse-часть 0007). Этот укус восстанавливает строгость **прикладной
валидацией**, а не колоночным NOT NULL. Дисциплина «модель+движок сейчас, вьюхи потом».
**Ключевое решение — единый kind-driven источник на модели.** `StockDocument.REQUIRED_HEADER_BY_KIND`
(словарь `Kind → ('date','number')` для receipt/inventory/requisition/transfer/writeoff; `()` для
kitting/relocation) + `clean()` с ошибками **по полям** (дружелюбны и в админ-форме, и через
`e.messages` в API). Правило зеркалит до-Ф2c NOT-NULL и живёт в **одном месте** — прежде оно
было размазано по 5 create-функциям (`if not number` с per-kind текстом + `date or today`).
**Врезка в два уровня без регрессии:**
- **admin** гейтится автоматически — `ModelForm.full_clean() → model.clean()` (закрывает дыру:
  после Ф2c админ мог сохранить Transfer с пустым номером, поле стало blank=True).
- **проведение** — новый гейт полноты `_require_header(doc)` (тонкая обёртка над `doc.clean()`)
  в `post_document`/`approve_receipt`/`post_transfer`: не выпускает неполный ордер независимо от
  пути создания (API/админ/прямой ORM). Kitting-закрытие не гейтим (kitting без обязательной шапки).
- **create_*/update_*** оставлены как быстрый фаст-фейл входного слоя (дружелюбные per-kind
  сообщения «Нужен № накладной» + дефолт даты на сегодня); авторитетная политика — на модели
  (кросс-ссылка в комментарии). Валидные документы из `create_*` всегда полны → **ноль регрессии**,
  API байт-в-байт, фронт/вьюхи не тронуты.
**Проверка:** 200 тестов (было 193; +7 `Wave13Fase2dTests` — карта зеркалит до-Ф2c NOT-NULL,
`clean` ловит пустой номер/дату по полю, полная шапка проходит, kitting освобождён, проведение
гейтит неполноту мимо create-guard ×2 [writeoff/receipt]) зелёные на MySQL 8.0.25.
`makemigrations --check` чист — **без миграции** (`clean()` + класс-атрибут + движок, схема не
тронута; README-диаграммы схемы не меняются — валидация не в ER). `seed_demo` ок; system check
0 issues. Прод не трогали.
**Последствия:** per-kind строгость восстановлена как единое правило. Остаток Ф2 (переименовано в
Ф2e+): переименования `Lot` (`part_number`/`lot_name`) + `Supplier → Counterparty` (+ контрагент

опорой на `REQUIRED_HEADER_BY_KIND` для видимости), миграция прод-базы. См. [WAVE13.md](WAVE13.md).

## 2026-07-09 — Волна 13, Ф2c: подъём общих полей ордера в MTI-родителя
**Тип:** решение + реализация + грабли
**Контекст:** Третий укус Ф2 (по согласованию с Иваном — из 4 кандидатов: подъём полей /
переименования Lot+Counterparty / relocation+мультисклад / admin-гибрид). Ф2a/Ф2b дали
конкретного MTI-родителя и схлопнули дуги, но общие поля документа (`project`/`user`/
`date`/`number`/`note`) всё ещё жили **копиями на каждом из 6 детей** — этот укус их
дедуплицирует, поднимая в `StockDocument`. Дисциплина «модель+движок сейчас, вьюхи потом».
**Итог (сделано, только бэкенд):** `project`/`user`/`date`/`number`/`note` → на родителя;
специфика (`Receipt.supplier`/`purchase`, `Kitting.target_item`/`qty`, `Writeoff.reason`)
осталась на детях. `Inventory`/`Requisition`/`Transfer` стали **безполевыми** MTI-детьми
(только ptr + `kind`). Реверс-аксессор общего `project` — единый `project.documents`
(было 6: `project.receipts`/…). `date` поднят **nullable**, `number`/`note` — **blank**
(на детях `date` был NOT NULL у 5 из 6, `number` — у 5, `note` — только Inventory):
строгость per-kind осознанно ослаблена, её восстановление — условной валидацией (Ф2d).
**Ключевое решение — `SeparateDatabaseAndState` (как 0005), а НЕ обычные Add/RemoveField.**
Замер ряби (Ф2a обещал «1 аксессор») подтвердился: в коде правится ровно один —
`project.writeoffs` (engine bridge закрытия проекта) → `Writeoff.objects.filter(project=…)`
(зеркалит соседний `Requisition.objects.filter(project=dest)`). Всё остальное прозрачно
через MTI: прямой `receipt.project`, `.objects.create(project=…)`, `.filter(project=…,
status=…)`, `select_related('project')`, admin `list_display`/`list_filter`/`search_fields`
— **ноль правок**, API байт-в-байт, фронт/вьюхи не тронуты.
**Грабли — почему нельзя обычным AddField+RemoveField:** в MTI поле родителя и
одноимённое поле ребёнка **не могут сосуществовать в состоянии** (Django падает на клэше
имён при рендере промежуточного state). Автодетектор это обходит **порядком** (сперва все
`RemoveField` детей, потом `AddField` родителя) — поэтому СОСТОЯНИЕ берём его же
операциями (гарантирует `--check` чист), а ФИЗИКУ пишем raw-SQL: на DB-уровне колонки на
разных таблицах не конфликтуют, и мы контролируем nullability (project/user: `ADD ... NULL`
→ backfill → `MODIFY ... NOT NULL` → FK) и порядок. Бэкфилл **без ремапа** — id ребёнка ==
`stockdocument_ptr_id` == id родителя (Ф2a), поэтому `UPDATE stockdocument p JOIN child ch
ON p.id=ch.stockdocument_ptr_id`. `user_id` — **INT** (auth.User = AutoField), не BIGINT;

## 2026-07-09 — Волна 13, Ф2b: коллапс дуг в единый FK на `StockDocument`
**Тип:** решение + реализация + грабли
**Контекст:** Второй укус Ф2. Ф2a унифицировала id-пространство ордеров (PK ребёнка =
id MTI-родителя), что открыло **коллапс трёх exclusive-arc** в один FK на
`StockDocument`. По согласованию с Иваном взяты **все три дуги** (не только `Lot.origin`):
`Lot.origin` (4 FK+Check), `StockLine.document` (4 FK+Check), `Attachment.owner`
(7 FK+Check → двухпутный `item ↔ document`). Дисциплина «модель+движок сейчас, вьюхи
потом»; фронт не тронут (API байт-в-байт).
**Ключевое решение — бэкфилл через COALESCE, без ремапа id.** В отличие от 0005 (где
физику id пришлось раздавать raw-SQL), здесь старая FK-колонка (`receipt_id`/…) **уже
хранит id ребёнка, равный id родителя** — поэтому
`origin_id = COALESCE(receipt_id, kitting_id, inventory_id, requisition_id)` — прямое
значение нового FK. Миграция `0006_collapse_arcs` — обычные `AddField`/`AlterField`/
`RemoveField`/`Remove|AddConstraint` (state автодетект прозрачен, `--check` чист) +
три `RunSQL` только для данных. Реверсивна: обратный `RunSQL` раздаёт свёрнутый FK по
старым колонкам согласно `document.kind` (`UPDATE … WHERE id IN (SELECT id FROM
plume_stockdocument WHERE kind='…')`) — портируемо MySQL+SQLite.
**Проекция движений — байт-в-байт.** `origin_kind`/`doc_kind` теперь читаются из
`origin.kind`/`document.kind` (дискриминатор Ф2a совпадает со старыми именами FK), а
`source_id` = `origin_id`/`document_id` (== прежнему child-id). Реверсы дуг
(`receipt.lots`, `kitting.lines`, `receipt.attachments`) **уцелели через MTI**: reverse
на родителе доступен с ребёнка. Ripple в движке — только query-lookups
(`receipt__isnull` → `origin__kind='receipt'`; `receipt__purchase` →
`origin__receipt__purchase`) и 6 detail-вью. Attachment API-контракт `owner_type`
(строки item/receipt/…) сохранён: 'item' → поле `item`, вид ордера → `document`.
**Итог (сделано, только бэкенд):** `Lot.origin`/`StockLine.document` — по одному
NOT-NULL FK (Check «ровно один» умер); `Attachment` — `item ↔ document` (Check жив,
двухпутный). Снесены `LOT_ORIGIN_FIELDS`/`STOCKLINE_DOC_FIELDS`; `ATTACHMENT_OWNER_FIELDS
= ('item','document')`. Админ-инлайны `StockLine` переключены на `fk_name='document'` —
Django принимает FK на MTI-родителя через `get_parent_list()`.
**Грабли:**
- `makemigrations --check` ловит дрейф `condition` у `CheckConstraint`: hand-made
  `Q(...)` не совпал бы по deconstruct с модельным `_exactly_one_q(('item','document'))`
  → в миграцию вставлен **инлайн-клон** `_exactly_one_q` (историческая миграция не
  зависит от будущих правок модели, но даёт байт-идентичный condition).
- **Порядок ↔ реверсивность:** `RemoveConstraint` идёт ПЕРВЫМ в forward, чтобы его
  реверс (`AddConstraint`) выполнился ПОСЛЕДНИМ при откате — уже после обратного
  бэкфилла. Иначе Check «ровно один» упал бы на пустых восстановленных колонках.
- **Stale `source_id` на dev:** dev-движения хранили pre-Ф2a child-id (напр. 20),
  который 0005 перецепил на parent-id (1), НЕ пересобрав движения. После 0006 `rebuild`
  привёл их к корректному parent-id — детектор «дрейфа» показал это как расхождение,
  на деле это **починка dangling-id** (idempotent-прогон → drift=0). `seed_demo` тоже
  чинён (создавал `Lot(receipt=…)`).
**Проверка:** 188 тестов (было 183; +5 `Wave13Fase2bTests` — одиночный FK+смерть Check
у Lot/StockLine, двухпутный владелец Attachment, инвариант проекции, реверсы через MTI)
зелёные на MySQL 8.0.25; `makemigrations --check` чист. **Круговой прогон** на scratch-БД
(seed@0006 → reverse@0005 → forward@0006): SNAP_A==SNAP_C **побайтово** (sha256 остатков/
движений/строк/владельцев), при 0005 арк-колонки восстановлены по видам (4 лота → 2
receipt/1 kitting/1 inventory). Свежая сборка @0006 — drift=0. Dev-MySQL догнан на 0006
(stale source_id починены, drift=0). Фронт не тронут (`l.origin`/`owner_type` те же).
Обе mermaid-диаграммы README + прозу обновил (Lot/StockLine/Attachment → один FK).
**Последствия:** три дуги мертвы. Остаток Ф2: подъём общих полей (project/user/date/
number/note) в родителя, условная валидация по `kind`, переименования `Lot`
(`part_number`/`lot_name`) + `Supplier → Counterparty`, гибрид-admin, `relocation`+
мультисклад, свод матрицы полей, миграция прод-базы. Прод не трогали. См.

## 2026-07-09 — Волна 13, Ф2a: MTI-ядро (`StockDoc` → конкретный `StockDocument`)

**Контекст:** Первый укус Ф2 (осознанный слом заморозки). По согласованию с Иваном —

**Ключевое решение — lean-глубина (не полный подъём полей):** родитель несёт только
`status` (уже общий) + новый дискриминатор `kind` (7 значений, «Тип = поле одной
сущности»); специфичные И общие FK-поля (`project`/`user`/`date`/`number`/`supplier`/…)

(`project.writeoffs`, `purchase.receipts`) — оба уцелеют, если `project` не поднимать;

- **`StockDoc`(abstract) → `StockDocument`(concrete MTI-parent).** 6 документов —
  наследники; `KIND` класс-атрибут + `save()` штампует `kind` в родителя. `DocStatus`/
  `is_posted`/guard'ы работают прозрачно (унаследованы).
- **Миграция `0005_stockdocument_mti`** (реверсивна): `CreateModel(StockDocument)` реально
  + `SeparateDatabaseAndState` — **состояние декларируем сами** (`RemoveField` status/id +
  `AddField` parent_link-ptr ×6), **физику** делаем raw-SQL под `FK_CHECKS=0`: раздать
  parent-id, перецепить 14 входящих дуг (Lot×4/Attachment×6/StockLine×4) old-child-id →

**Грабли:**
- Django-автодетектор **не может** выразить обретение MTI-родителя на живых данных:
  `RemoveField(id)` с входящими FK на MySQL упадёт, а parent_link нельзя добавить без
  дефолта (`makemigrations` спотыкается интерактивным вопросом). → миграция целиком руками.
- **DDL на MySQL авто-коммитится** (нетранзакционно): сбой посреди `mti_forward` оставит
  БД полураздельной без отката. → валидировал НЕ на dev-данных сразу, а на отдельной
  scratch-БД (создать/снести), dev догнал лишь после зелёного круга.
- Верный способ выверить state MTI без БД — `makemigrations --check`: сойдётся «No changes»
  ⇒ `state_operations` описывают модель точно (петля обратной связи, поймала `AutoField`

**Проверка:** 183 теста (было 179; +4 `Wave13Fase2aTests`) зелёные на MySQL 8.0.25.

**Последствия:** id-пространство ордеров едино — **дуги теперь схлопываются в один FK на

## 2026-07-05 — Два замка: замок формы + фиксация документа (термин и сцепка)
**Тип:** решение
**Контекст:** вычитка UI_GUIDE. Иван предложил объединить UI-замок формы со
статусом документа (мутабельная ДНК позволяет «дописывать в накладную»).
Разобрали минусы объединения: замок перестаёт быть бесплатным (закрыть =
провести с валидацией, открыть = каскад), черновик нельзя посмотреть чисто,
клик меняет общее состояние для всех и для движка (забыл закрыть — балансы
врут), двойная семантика на справочниках. Бесплатность UI-замка признана
ценностью — выбрана **сцепка вместо объединения** (мокап-артефакт «Два замка»).
**Итог (внесено в UI_GUIDE §4–5):**
- **Один слот** справа вверху заголовка, различие видом: **контур = вид,
  заливка = фиксация**. Контурный 🔓/🔒 — бесплатный личный переключатель
  инпутов (черновик рождается открытым); клик мгновенный, данных не трогает.
- **Фиксация** — слово-объединитель терминальных статусов (posted/closed,
  в README-модели «мягкий замок»): проведённая поставка/комплектация, закрытый
  проект = «документ **зафиксирован**». Перебрали: «печать» (отклонена —
  «распечатать» путает с print), «пломба», «учтён» («снять с учёта» сталкивается
  со складом), «заморожен» (коннотация приостановки), «проведён» (не тянет
  проект). Победила фиксация — универсальна + git-резонанс (commit) для команды.
- Зафиксированный документ: **чип с заливкой** `🔒 зафиксирована` в слоте
  замка, форма read-only, индикатора сохранения нет. Клик по чипу = предложение
  **снять фиксацию**, подтверждение **всегда** (при зависимых — модалка
  «поедут N»); после согласия форма сразу разлокована.
- **Асимметрия осознанная:** снять фиксацию через слот замка можно, поставить —
  нельзя; «Зафиксировать» — отдельная операция с валидацией (место кнопки —
  в инвентаре действий).
- Индикатор на разлокованной форме: `● редактируется` (покой) →
  `● сохраняется…` → `✓ сохранено` / `ошибка`.
**Последствия:** термин «мягкий замок» в README/речи постепенно заменяется
«фиксацией»; правка README — при ближайшем осознанном проходе по нему.

## 2026-07-04 — Волна 12: логин-экран + аутентификация (финальная волна MVP)
**Тип:** решение
**Контекст:** после В11 оживлены все модели схемы — из отложенного каждую волну
оставался **ровно один** пункт: логин-экран (журнал В9–В11 звал его «кандидатом
финальной волны перед многопользовательской эксплуатацией»). До В12 аутентификации не
было вовсе: DRF без `DEFAULT_PERMISSION_CLASSES` → весь `/api/` открыт, `_actor(request)`
подставлял первого суперюзера как fallback-автора, фронт анонимен. Пользователь на
утверждении плана выбрал **один уровень доверия** (любой залогиненный = полноправный
оператор PLM; ролей/прав нет — под малую доверенную команду). Объём — надстройка
паттерном волн 2–11, **без правки схемы** (встроенный `User`, миграций нет).
**Итог (схему БД НЕ трогали — `makemigrations --check` чист; диаграммы README без изменений):**
- **Механика — сессионная** (Django session + DRF `SessionAuthentication`), не JWT/токены:
  внутренний инструмент, SPA с того же origin на проде, сессии в БД — **никакого
  Redis/долгоживущих процессов** (ровно под ограничения reg.ru shared). Глобальные
  DRF-дефолты: `SessionAuthentication` + `IsAuthenticated` → весь `/api/` уходит за логин
  **одним переключателем**, включая скачивание вложений (`attachment_download` — журнал
  В11 прямо предвидел это как «естественное место навесить логин»).
- **Три `AllowAny`-точки, не больше:** `GET /api/auth/me/` (`@ensure_csrf_cookie` — отдаёт
  `{id,username,full_name,is_superuser}` или **401**; заодно ставит CSRF-cookie, чтобы фронт
  получил токен до логина/мутаций), `POST /api/auth/login/` (`authenticate`+`login`, 400 на
  неверный пароль), и `ping` (health-check). Всё прочее — под `IsAuthenticated`. `logout`
  (`POST /api/auth/logout/`) требует логина. **Сам-регистрации нет** — юзеров заводят в
  admin (внутренний инструмент).
- **CSRF теперь реально форсится.** До В12 фронт был анонимен (DRF не форсил CSRF); под
  сессией `SessionAuthentication.enforce_csrf` требует `X-CSRFToken` на небезопасных методах.
  Фронт (`api.ts`) уже слал токен из cookie (задел волн 2/11) — сомкнулось: boot→`me()`
  ставит cookie → мутации шлют `X-CSRFToken`. **Проверено живьём:** мутация без токена → 403,
  с токеном → 201. `login` (POST от анонима) CSRF не требует — SessionAuth не форсит для
  неаутентифицированного; DRF-вьюхи `csrf_exempt` на уровне Django-middleware.
- **`_actor(request)` → `request.user`** (fallback на суперюзера убран): под глобальным
  `IsAuthenticated` в мутационных вьюхах юзер всегда настоящий — авторство на документах
  стало честным per-user (проверено тестом: `Kitting.user` = залогиненный).
- **Фронт:** состояние `user: User | null | undefined` (undefined=грузим `me()`,
  null=`<Login/>`, иначе приложение). Данные грузятся **только под логином** (гейт по `user`
  в эффекте). `setUnauthorizedHandler` — глобальный хук: 401/403 в любом `get/send/upload` →
  назад на логин (протухшая сессия). `me()` — собственный fetch (401=null, без хука).
  `Login.tsx` — моно-forward карточка (логин/пароль, ошибка). Выход — кнопка ⏻ внизу
  activity-bar (спейсер), имя юзера в статус-баре.
- **Тесты:** 7 HTTP-классов волн 6–11 получили `force_login` в `setUp` (они уже создавали
  суперюзера `admin`); движковые юнит-тесты не трогались (зовут движок напрямую). Новый
  `AuthHttpTests` (+5): гейтинг анонима (403), `ping` открыт, `me` 401+csrf-cookie,
  логин-флоу (400 неверный → 200 + авторство), logout закрывает сессию. Django test Client
  по умолчанию не форсит CSRF (`_dont_enforce_csrf_checks`), поэтому `force_login` достаточно.
**Проверено:** 150 тестов зелёные (145 + 5 auth); `tsc -b && vite build` чист;
`makemigrations --check` — схема не менялась. Живой runserver-smoke (10/10): ping 200,
анон projects 403, me 401+csrf, неверный пароль 400, login 200 (`full_name` «Смоук»), authed
200, мутация без CSRF 403 / с CSRF 201, logout 204, после logout 403; smoke-юзер и данные
удалены за собой.
**Последствия:** **MVP-контур замкнут** — весь `/api/` за аутентификацией, авторство
per-user, вход/выход из UI. Отложенное **исчерпано** (все модели схемы оживлены, поток
дефицит→заказ→приход→комплектация→передача→закрытие + планирование/pegging/инвентаризация/
бюджет/вложения/логин — в UI). Схема заморожена (В6–В12 без миграций). Дальнейшее — уже не
«волны оживления», а полировка/эксплуатация: чистка хвоста фронта (накопленные отклонения
конформанса, forward-жест истории), ролевая авторизация (если понадобится), деплой В12 на
прод. План — [docs/WAVE12.md](WAVE12.md).

## 2026-07-04 — Волна 10: бюджет проекта — два числа денег + себестоимость/экономия (north-star окупаемости)
**Тип:** решение
**Контекст:** после В9 инвариант «лот из origin-документа» замкнут (4 origin оживлены),
закупочный и складской контуры полны. Из отложенного каждую волну оставались три кандидата:
**бюджет/экономия**, логин-экран, UI вложений. Пользователь выбрал бюджет/экономия — это
заявленный **north-star окупаемости линзы** (payoff = оцифровать пользу PLM), к тому же
формула (№6) уже спроектирована на аудите движка (запись 2026-06-30), а поля давно в
замороженной схеме. Объём — читающая надстройка паттерном волн 2–9, без правки схемы.
**Итог (схему БД НЕ трогали — `Project.budget`/`Item.estimated_cost`/`Lot.unit_cost` и
снимок `_device_unit_cost` из В2 уже в модели; диаграммы README без изменений):**
- **Два числа денег — не путать (как с qty).** **Потрачено** (факт) = `Σ(unit_cost×qty)`
  по приходным (`Receipt`-origin) лотам проекта — застывшая цифра «кэша ФЛС». **План**
  (прогноз) = «факт где есть, оценка где нет» = потрачено + `Σ(остаток×estimated_cost)`.
  Компас = `budget − план` (запас/перерасход). По мере прихода УПД план → факт.
- **Три правила бюджета (из №6) реализованы отбором по origin:** (1) заём/свои/найденное
  бесплатны — «потрачено» берёт только `receipt__isnull=False` (origin
  requisition/inventory/kitting = 0 ₽, платил источник); (2) только покупной уровень —
  снимок собранного узла (лот-`Kitting`) в деньги не складываем, производимые компоненты
  в оценку не берём (иначе двойной счёт); (3) нет `estimated_cost` → в список `unestimated`
  (бейдж «план неполон»), не тихий 0.
- **Оценку считаем на ●+▲** (в пути + к заказу); ✓ уже покрыто (приход — в «потрачено»,
  заём — бесплатно). Потребность **агрегируем по компоненту** (1 уровень BOM), один
  `_coverage` на компонент — без задвоения между потребностями (в отличие от
  по-строчного `project_deficit`).
- **Себестоимость — только верхние приборы** (решение пользователя, дефолт): `Σ(qty×снимок)`
  по лотам-приборам закрытых `Kitting`, `item ∈ target_item потребностей`. Подсборки не
  суммируем — их цена уже в снимке верхнего прибора. Снимок включает заём **по реальной
  цене** (`Requisition`-лот наследует цену предка, В6) — честная цена для КП. **Экономия
  = себестоимость − потрачено** = оцифрованная польза заёма (отрицательна, пока приборы
  не собраны — честно).
- **Движок:** `project_budget` (сборка) + `_project_spent`/`_project_estimate`/
  `_project_cost` (слагаемые). DRF: `GET /api/projects/<pk>/budget/` (read). Фронт:
  `BudgetPanel` в шапке `DeficitView` (по выбору пользователя — не отдельный режим) +
  форматтер `money()`.
**Последствия:** north-star линзы («знать остатки в разрезе проектов» + «окупаемость»)
теперь на экране. Отложенное сократилось до двух пунктов: логин-экран, UI вложений
(`Attachment`). Схема заморожена (В6–В10 без миграций). 136 тестов зелёные.

## 2026-07-03 — Волна 9: инвентаризация (`Inventory`) — 4-й origin + серая ре-материализация
**Тип:** решение
**Контекст:** после В8 закупочный контур замкнут (свод → план → pegging → заказ → приход →
комплектация → передача → закрытие). Из отложенного каждую волну с В6 самым откладываемым
пунктом оставалась **инвентаризация** (`Inventory`) — четвёртый и последний origin-документ
партии, единственный ещё не «оживлённый» (receipt/kitting/requisition были записываемыми, а
inventory — только в модели). Плюс с В6 висела зафиксированная дыра: **«списание в GREY
ничего не кладёт — серую ре-материализацию даст `Inventory`»**. Пользователь выбрал В9 =
инвентаризация из четырёх кандидатов (ещё бюджет/экономия, логин, вложения). Объём —
записываемая надстройка паттерном волн 2–8, без правки схемы.
**Итог (схему БД НЕ трогали — `Inventory` была в модели; origin `inventory` знают
`LOT_ORIGIN_FIELDS`/`rebuild_movements` с В1; диаграммы README без изменений):**
- **Отдельной `InventoryLine` в модели нет** — строки акта это его лоты (`inventory.lots`),
  как приход/УПД без `ReceiptLine`. Кокпит зеркалит `receipt_cockpit`: born-лот =
  `+RECEIPT` в проекте акта (изделие + кол-во + цена + название + зав.№, автосейв).
- **Движок:** `inventory_cockpit` (проекция: строки + провенанс + `total_cost`),
  `create_inventory`, `add_inventory_lot` (born `+RECEIPT`, опц. `predecessor`),
  `update_inventory_lot`, `remove_inventory_lot` (guard `_lot_consumed_downstream`),
  `update_inventory` (правка шапки через `_apply`), `written_off_lots` (пикер).
- **Замка нет** (у `Inventory` нет поля-статуса, как у `Writeoff`/`Requisition`): правимо
  всегда, корректность держат guard'ы (`qty>0`, цена ≥0, потреблённый лот не удалить) +
  `PROTECT`. Кол-во не клампим (мутабельная ДНК).
- **Серая ре-материализация — payoff волны.** Списание = чистый `−ISSUE` (лот покинул учёт
  «в серый»). Нашли физически → инвентаризация возвращает на баланс born-лотом с
  `predecessor` → списанный (наследует item/цену/название/зав.№ — генеалогия/провенанс,
  как отпочкование `Requisition`). `written_off_lots()` = пикер списанных лотов (суммарно
  списанное с лота). Дом ре-материализации по умолчанию — GREY «Свободные неучтённые» (где
  серый остаток и «живёт»); оттуда его забирают в реальный проект требованием (В6). Излишек
  (без источника) — `predecessor=None`, добавляется призрачной строкой в реальном проекте.
- **DRF:** `GET/POST /api/inventories/`, `GET/PATCH /api/inventories/<pk>/`, `POST .../lots/`
  (`predecessor_id` → наследование полей источника), `PATCH/DELETE /api/inventory-lots/<pk>/`,
  `GET /api/written-off-lots/`. Отдельный `inventory-lots/<pk>/` (не общий `/lots/<pk>/`,
  тот заякорен на приход).
- **Фронт:** десятый режим activity-bar 🔍 «Инвентаризации» + `InventoryView` (кокпит:
  строки-лоты автосейвом, призрачная строка «излишек», колонка «провенанс», правка шапки;
  панель «Ре-материализация из списанных» — пикер → «вернуть на баланс»). Статус-бар → В9.
**Проверено:** 130 юнит-тестов зелёные (121 волн 1–8 + 7 движка + 2 HTTP-smoke),
`tsc -b && vite build` чист, `makemigrations --check` — **схема не менялась**. Живой
runserver-smoke на MySQL 8.0.25: `/api/inventories/` и `/api/written-off-lots/` 200; create
акта в GREY → излишек (`total_cost` растёт) → ре-материализация реального списанного лота #24
(written 12) → born с `predecessor=#24`, унаследованы цена 800 / «Корпус Al», `live=12`;
smoke-данные удалены за собой.
**Последствия:** инвариант «`Lot` всегда из origin-документа» замкнут — все 4 origin
записываемы. Серый цикл замкнут: `Writeoff`(−ISSUE «в серый») → `Inventory`(+RECEIPT «на
баланс», provenance) → `Requisition`(в проект). Следующие волны: бюджет/экономия (north-star
окупаемости), логин-экран, UI вложений (`Attachment`).

## 2026-07-03 — Волна 8: pegging (нарезка плана-`Procurement` на веер проектных `Purchase`)
**Тип:** решение
**Контекст:** после В7 два уровня закупок жили порознь: командный план (`Procurement`,
В7) обособлен, а проектные заказы (`Purchase`, В4) висели каждый на throwaway-родителе
`_solo_procurement` (вырожденный 1:1). Не хватало обратного среза — из согласованного
плана «купить N всего» породить проектные обязательства (проект A ×60, проект B ×40),
**разделяющие один план-родитель**. Тему В8 (pegging) journal называл прямым долгом
В4/В7. Объём — записываемый pegging поверх существующих сущностей, паттерном волн 2–7.
**Итог (схему БД НЕ трогали — `Procurement`/`Purchase`/`PurchaseLine` были; диаграммы README без изменений):**
- **Ломали не поле, а эвристику `_plan_procurements`.** Старый признак плана
  `purchases__isnull=True` рушился ровно от pegging (пегнутый план обзаводится
  `Purchase` и исчезал бы из списка). Различаем **структурно**: solo-заглушка =
  `Procurement` с ≥1 `Purchase` **и без** строк плана (`ProcurementLine`); план — всё
  прочее (`annotate(Count).exclude(_n_purch__gt=0, _n_lines=0)`). Пегнутый план (есть и
  строки, и заказы) остаётся в списке; ad-hoc проектный заказ (мост «дефицит → заказ»,
  В4) по-прежнему на solo-заглушке — обе схемы сосуществуют, схема заморожена.
- **Движок:** `procurement_pegging(proc)` — проекция: по строке плана пегнуто/остаток/
  статус ✓●▲ + разбивка по проектам (наводка из `command_deficit` + фактически пегнутое)
  + веер проектных `Purchase`. `peg_procurement_line(proc,item,project,qty,user)` —
  find-or-create draft-`Purchase` проекта **под этим планом** (`_project_purchase_under`
  вместо `_solo_procurement`) + инкремент `PurchaseLine`. `autopeg_procurement` — топ-ап
  каждой `(item,project)` до наводки свода; **идемпотентно** (`delta=to_order−пегнуто`,
  пегаем только >0) и не трогает ручной перепег. `unpeg_procurement_line` — снять пег
  (удалить строку в draft-заказах; в отправленном заказе — просим сперва снять send).
- **Гварды:** item — из строк плана; проект — активный внешний; кол-во >0; отменённый
  план не пегается. Кол-во **не клампим** по остатку плана (перепегнуть можно — в духе
  мутабельной ДНК; `remaining<0` подсвечивается).
- **DRF:** `GET .../pegging/`, `POST .../peg|unpeg|autopeg/` (мапят `ValidationError`→400).
- **Фронт:** `PeggingPanel` в `ProcurementView` (под строками, если план не отменён):
  по строке плана раскрытие с распределением по проектам (ручной пег/снятие + «＋наводку»),
  кнопка «Разрезать по проектам» (autopeg), веер проектных заказов со ссылками в кокпиты
  (`openPurchase`). Панель освежается счётчиком `rev` на мутациях кокпита. Статус-бар → В8.
**Проверено:** 121 юнит-тест зелёный (114 волн 1–7 + 5 движка pegging + 2 HTTP-smoke),
`tsc -b && vite build` чист, `makemigrations --check` — **схема не менялась**. Живой
runserver-smoke на MySQL 8.0.25: мост→план (60) → pegging (наводка 40/20) → autopeg
(веер из двух draft-заказов 40/20, remaining 0, ✓) → повторный autopeg идемпотентен
(веер по-прежнему 2) → unpeg P1 (60→20) → peg item-не-в-плане → 400; dev-БД очищена за собой.
**Терминология (уточнено на обкатке):** *pegging* (MRP/ERP-термин — трассирующая связь
«поставка ↔ потребность», «привязать закупку к проекту») **остаётся каноном в коде,
схеме и записях** (движок/API/типы/журнал — консистентность внутри). В **UI** слово
русифицировано косметически: «Привязка к проектам», кнопки «привязать»/«отвязать»/
«Разрезать по проектам», статус-бар «привязка плана-закупки к проектным заказам» — без
англицизма (предметник: «русскому глазу глаже, ложится хорошо»). Код-идентификаторы
(`peg_*`/`autopeg_*`/`PeggingPanel`) НЕ переименовывали — не рвём связи с журналом/памятью;
разом переименуем при чистке хвоста перед инструкцией, если решим.
**Последствия:** два уровня закупок слиты — план становится родителем веера проектных
обязательств, `_solo_procurement` остаётся только под ad-hoc проектные заказы. Следующие
волны: инвентаризация (`Inventory`), два числа бюджета + себестоимость/экономия,
логин-экран, UI вложений. План — [docs/WAVE8.md](WAVE8.md).

## 2026-07-02 — Навигация «Назад» (предыдущая форма) + перехват браузерной истории
**Тип:** решение
**Контекст:** пользователь-предметник (привычка от SAP) интуитивно тянется к «назад»
и несколько раз браузерным «Назад» уходил с SPA на предыдущий сайт вместо предыдущей
формы. Нужна «предыдущая форма»: ненавязчивая визуальная кнопка + клавиатурное
сокращение; в идеале — перехватить и саму браузерную кнопку.
**Итог:**
- **Стек истории форм в `App.tsx`** — пишем предыдущее состояние `{mode, sel}` при
  любой смене **единым эффектом** на `[mode, sel]` (а не правкой десятков call-site
  навигации). `skipRef` исключает из истории откат назад и стартовый автовыбор проекта.
- **Браузерный «Назад» перехвачен через `popstate`.** На каждую форвард-навигацию
  кладём `window.history.pushState(null, '')`; `back()` вызывается из `popstate` и
  откатывает наш стек. Все три триггера (браузерная кнопка, визуальная кнопка,
  сокращение) идут через `window.history.back()` → `popstate` — стеки браузера и
  приложения не рассинхронятся. Cmd+[ на Mac работает автоматически (это нативный
  back → popstate). На «корне» истории back уводит с сайта — осознанно ок.
- **Клавиатура — `Alt+←`** (не `Cmd+[`, чтобы не конфликтовать с нативным жестом,
  который и так перехватывается). **Визуальная кнопка** «‹ Назад» — вверху рабочей
  области, всегда в одном месте, тусклая (`.crumb`/`.back`), disabled при пустой истории.
- **StrictMode-safe:** трек-эффект без cleanup идемпотентен на двойной mount (guard по
  `prevRef`), pushState срабатывает только на реальных сменах.
**Последствия:** боль «назад уводит с сайта» закрыта. URL состоянием форм пока НЕ
кодируем (pushState(null) — просто счётчик записей); при желании глубоких ссылок/
перезагрузки-на-той-же-форме позже завести сериализацию `sel` в URL. Косметика волны 3:
кнопка замка прихода «Сверить со сканом (замок)» → «Сверено со сканом».
**Известная шероховатость (отложена, найдена на обкатке 2026-07-02):** браузерный
жест **«Вперёд»** ощущается кривым. Причина: `popstate` стреляет и на back, и на
forward, а наш `onPop` **всегда** вызывает `back()` → «Вперёд» ведёт себя как ещё один
«Назад» (форвардного стека у нас нет). Не критично (основной поток — «Назад»), фиксим
на полировке: либо гасить/не поощрять forward, либо вести полноценный two-way стек
(различать направление по `event.state`-счётчику depth). Пользователь дал знать — держим
в списке полировки, не трогаем между волнами.

  между проектами нет:** человек выбирает диспозицию, движок предлагает акт — заказать
  (`Procurement`/`PurchaseLine`), взять с белого (`Requisition`), спаять из своего
  (прямая `KittingLine`), заимствовать у соседнего активного проекта (`Requisition` B→A).
  Заём = явный документ: у источника `qty` просядет, его дефицит честно пересчитается.
  Порядок-подсказка (белый → этот проект → соседние → серый) — мягкая сортировка.
- **№3 — «заказано» (оранжевый член).** = `Σ max(0, PurchaseLine.qty − поступившее)` по
  **открытым** строкам проекта. **«Заказано» начинается с `sent`** (реальное
  обязательство перед контрагентом); `draft` **остаётся красным** («твой ход, ещё не
  отправил»), `cancelled` не считается. **Поступившее = приход (`+RECEIPT`), не текущий
  остаток**: получили 100 → спаяли 40 → заказ закрыт (поступило 100), а 60 живут в члене
  «склад». Переполучение клампится в 0 (отрицательного «заказано» нет). **Документ = УПД
  правда:** лот по УПД = ровно что в УПД; физический излишек над документом (УПД 100,
  факт 120) «находим» актом `Inventory` на сером складе. **Оранжевый член обобщён по типу
  Item:** покупной → открытая `PurchaseLine` (`sent`); производимый → wip-`Kitting`
  (из №5). Один цвет `●`, одно значение «запущен процесс, ждём».
- **№4 — частичное покрытие (бинарность снята).** Строка ≠ «всё или ничего»: показываем
  **тройной разбор** `нужно 10 → 4 ✓ есть · 3 ● заказано · 3 ▲ заказать`. **Цвет шапки =
  худшее из присутствующего** (есть `▲` → красная; иначе `●` → оранжевая; иначе `✓`) +
  бейдж лучшего прогресса — **то же правило, что у цвета прибора, одно на всё**. `▲`/`●`/
  `✓` — это три **сегмента** строки, не взаимоисключающие ярлыки. **Распределение при
  нехватке:** покрытие считаем на уровне **Item в проекте** (суммарно), **без привязки**
  к конкретным приборам — какому прибору достанется дефицитный лот, решается **только при
  пайке** (пробил `KittingLine` → забрал лот, кто первый). Привязка раньше = «резерв», от
  которого отказались.
- **№6 — бюджет.** Два счётчика денег (не путать, как с qty): **потрачено (факт)** =
  `Σ (unit_cost × qty)` приходных (`Receipt`) лотов проекта — точная застывшая цифра;
  **план** = прогноз полной стоимости «факт, где есть; оценка, где нет» (пришедшее по УПД
  — по реальной цене, остальное нужное — по `estimated_cost`). Компас: `budget − план` =
  запас/перерасход (прогноз). По мере прихода УПД оценки заменяются фактами → **в конце
  проекта план = факт** (сходятся). Правила: (1) **заёмные/свои бесплатные лоты = 0 ₽** в
  бюджете (платил источник); (2) считаем **только покупные материалы** до покупного
  уровня — снимок цены собранного узла в бюджет **не** складываем (иначе двойной счёт);
  (3) **нет `estimated_cost`** → подсветка «N позиций без оценки, план неполон», не молча 0.
- **Красивое: «обошёлся vs потратил» = два разных числа, оба уже в модели.** Бюджет
  (кэш ФЛС): заём = 0, считается по `Receipt`-лотам. **Себестоимость** (честная цена
  прибора, для КП): заём стоит реальную цену — `Requisition`-лот наследует `unit_cost` от
  предка (снимок, решено 28.06), поэтому снимок себестоимости прибора сам подхватывает
  цену заёмного компонента. Расходятся по origin лота. **`экономия проекта = Σ
  себестоимость − потрачено`** — оцифрованная польза внутреннего заёма (= польза PLM).
**Последствия:** **Движок СППР полностью определён** (№1–6). Схема не менялась — ER не
трогаем. README обогащён (раздел линзы: счётчики заказано/склад, тройной разбор строки,
карта остатков, два числа бюджета + себестоимость/экономия). Дальше — реализация на
Django (план — по явному запросу).

## 2026-06-30 — Движок СППР: wip-Kitting, глубина разузла, два дашборда, консолидация=проекция
**Тип:** решение
**Контекст:** Детально проектировали вычислительный движок линзы (см. прошлую запись —
«начать с движка»). Шли по развилкам: №5 (двойной счёт wip-`Kitting`) разобрали
исчерпывающе, он потянул за собой №1 (глубина разузлования) и архитектуру дашбордов.
Сквозной нерв проектирования модели данных не менялся — все решения про **проекции/
поведение движка/представления**, новых полей и сущностей нет.
**Итог:**
- **№5 — wip-`Kitting` и двойной счёт (закрыт).** Опасность: wip-`Kitting` уже списал
  компоненты (ISSUE прошёл, `qty` лотов просел), но лот прибора рождается лишь на
  `closed`. Если движок считает потребность незакрытой и разузловывает её заново —
  компоненты насчитываются повторно. Решение: **wip-`Kitting` закрывает счётчик
  потребности прибора (→ оранжевый), но НЕ обнуляет остаток его BOM.** Три корзины
  спроса на прибор: **готово** (закрытый лот, `✓`) / **делается** (wip-`Kitting`, `●`)
  / **не начато** (потребность − закрытые − wip, `▲`). Дефицит компонентов =
  разузел «не начатых» + Σ остаток wip-`Kitting` (`BOM target − пробитые KittingLine`).
  Тройная согласованность даёт анти-двойной-счёт: лоты по текущему qty (уже просели) +
  делающиеся не разузловываем заново + их остаток по непробитым строкам.
- **Призрачные строки = линза на форме `Kitting` (кокпит сборки).** Пробитые
  `KittingLine` — реальные зелёные строки; непробитые строки BOM — **призрачные**,
  read-only, покрашены движком по доступности компонента (`▲` не заказан / `●` заказан /
  зелёный-доступный лот). **Пайка = промоушн призрака в реальную `KittingLine`.** Ничего
  не храним (это проекция), сходится с принципом «призрачной строки» и «`KittingLine`
  по факту пайки». Глубина кокпита — **ровно 1 уровень BOM** (плата = атом, её резисторы
  в её акте).
- **Жизненный цикл строки = конвейер `▲→●→✓`.** `▲` красный = дефицит, не заказан
  (**нужна работа**); `●` оранжевый = заказан/делается (запущен процесс, **ждём** —
  одно значение для открытой `PurchaseLine` и wip-`Kitting`); `✓` зелёный-доступный =
  лот на складе; `✓` зелёный-потреблён = пробит, просадил лот.
- **Цвет прибора — инвертированный бейдж (радикальный).** База = **worst-of**
  остаточных строк (есть красный призрак → прибор **красный**, внимание); бейдж =
  **лучший достигнутый прогресс** (оранжевый/зелёный → «работа идёт, просто тупим»).
  Прибор становится оранжевым в точке выдоха — **когда не осталось красного** = всё
  критичное, зависящее только от нас, сделано (дальше контрагенты/обстоятельства).
- **№1 — глубина разузлования.** Единый ограничитель: **лот терминирует разузел**
  (лот воплощает детей → не раскрываем → чинит двойной счёт). Самоподобно по уровням
  (плата в wip-`Kitting` = оранжевая плата, её непробитые резисторы — красные).
  Терминация на лотах (вкл. частичный wip) и листьях BOM. **Две разные разузлования, не
  путать:** BOM-разузел (план, `item→BomLine→item`) vs генеалогия (факт,
  `Lot→Kitting→Lot`, паспорт прибора, всегда полная глубина). Глубина по формам:
  дашборд — динамическая «по красному»; кокпит `Kitting` — 1 уровень; паспорт —
  полная по фактам; where-used — вверх.
- **Два дашборда — две высоты одного движка.** **Командный** (по всем проектам, ось
  Item) → плоский роллап дефицита → **`Procurement`**; `Procurement` **без `project`** —
  это маркер командной высоты, а не пробел. **Проектный** (ось Project) → **`Purchase`** +
  жизненный цикл. Командный дефицит = **Σ проектных дефицитов по Item** (не перенеттинг в
  общий котёл — чужие ФЛС/деньги); «Собственный склад» — **предложение** через
  `Requisition`, не авто-зачёт. Круговая целостность: проектный дефицит → консолидируем в
  `Procurement` → нарезаем в `Purchase` по проектам → УПД сажает лоты в проекты.
- **Pegging (BOM-дерево) — один механизм, три работы.** Разворот строки роллапа →
  «откуда сложилось число» (источники по проектам/приборам). Это и **проверка** глазами,
  и **нарезка по контрагенту** (чек-бокс-дерево: галка на «печатная плата» → её ЭКБ;
  снять «голый PCB» → отдельный завод/`Procurement`; «сформировать закупку»), и
  **авто-срез** консолидированного `Procurement` обратно в проектные `Purchase`. Запас %
  и ручной override (кратно 1000 при дефиците 194) → **`ProcurementLine.qty` (документ)**;
  расчёт движка (194) остаётся **read-only проекцией**.
- **Сквозной принцип: консолидация = проекция, изоляция = хранение.** Роллап,
  серая куча, `Procurement` — проекции «свести вместе»; `Lot`/`Purchase`/`Kitting`/
  закрывающие акты — по-проектно, атомарно, врозь. Слить можно запросом в любой момент,
  разорвать слитое — нельзя → слияния в хранении не делаем. `Procurement` нарезается
  **начисто** (проектные доли круглые, сумма = общая закупка); запас держим **по-проектно**,
  не общим котлом; «194 в 0 без запаса» = честный сигнал связи производств, не прячем и
  не наводим искусственно.
- **Излишки и закрытие — модель уже готова, новых сущностей нет.** Излишки выводятся из
  проекта **отдельными по-проектными актами** (изоляция документооборота — это умение).
  **Белый** путь (держать на балансе под будущий kitting, ~10%) = `Requisition` →
  «Собственный склад», **отдельные лоты важно видеть**. **Серый** (выбросить из учёта,
  дефолт ~90%) = `Writeoff` → «Свободные неучтённые», **показываем сумму в аккордеоне**.
  Провенанс = `Lot.predecessor` → `Receipt` → УПД (не новое поле), нужен для kitting из
  остатков.
- **Единый вид склада.** Любой склад (проектный/белый/серый) показываем **всегда
  свёрнутым по Item аккордеоном** с общим числом; детализация лотов (откуда что
  стеклось) — **явным запросом**. В проектах обычно 1 лот на уникальный Item, на складах
  — несколько. **Единообразие форм > спец-правила** на тип склада.
- **Партия N в M проектов = строго M `Kitting`** (подтверждение): «партия» —
  оффлайн-группировка, **не сущность**; связь актов в модели не вводим (`parent_act`
  ранее отвергнут). Неттинг приборов идёт по-проектно.
**Последствия:** **Схема БД не менялась** — ER-диаграммы README трогать не нужно.
Прозаический раздел линзы/дашборда в README можно позже обогатить этими проекциями (по
предложению, не сейчас). Движок выкристаллизовался до уровня, с которого остальные
функции (№1–4, 6) почти выводятся.
**Открытые/следующее:** добить отложенные формулы (№1 глубина — фактически закрыт;
№2 что входит в «склад», №3 «заказано», №4 бинарность/частичное покрытие, №6 бюджет
планируемый/фактический) — по явному указанию. Реализация — после.

## 2026-06-30 — Линза = один движок-СППР + панель управления проектом
**Тип:** решение
**Контекст:** Прорабатывали помощника-линзу. Заметили, что линза уже «случайно»

  Тот же движок следит за целостностью таблиц.

  формах документов. **Закрытие наступает само** как итог управления.
- **Проекции read-only** в представлении — подсказка «менять в документе, не в линзе».
- **Загрузка окружения ленивая, без действий пользователя**: свойства сразу, связи
  подтягиваются сами; кнопки «подгрузить» избегаем (бесят). Особо тяжёлые редкие
  запросы — «заряжены, но не спущены» в свёрнутом аккордеоне (9/10 не нужны, на 10-й
  осознанно триггерим, ждём ~3 сек vs часы в Excel). StockMovement и всё вычислимое
  держим свежим без кэша.

## 2026-06-29 — Мутабельная ДНК: ledger → проекция, мягкий замок (пересмотр иммутабельности)
**Тип:** решение (пересматривает «неизменяемый ledger» от 2026-06-27)
**Контекст:** Закрывали открытые вопросы прошлой записи (правка `wip`-строк vs
иммутабельность). Предметник сформулировал глубже: инструмент agile, цикл
«red → green → refactor» — команда кругами «подбивает» проект (приборы готовы,
компонентов хватило, документы сошлись), и почти всё должно править**ся**, пока не
сошлось. Иммутабельный `StockMovement` оказался белой вороной на фоне ДНК «всё —
вычисляемый отчёт, а не мутабельная таблица».
**Итог:**
- **`StockMovement` = пересчитываемая проекция, не append-only журнал.** Источник
  правды — мутабельные документы (УПД/акты); движения **не правят руками**, их
  пересчитывают из документов синхронно при изменении (данных мало, без Celery).
  «Плавание движений по количествам» по ходу подбивки — норма. Таблицу оставили
  ради удобных `SUM` по лоту/месту/проекту (радикальный вариант «убрать
  `StockMovement`, считать из документов» отвергли — дороже по схеме: место
  хранения лота переехало бы на строки/лоты, потеряли бы единый `SUM`).
- **«Закрытие» — мягкий замок, не разрушительная запись.** Поле-замок гасит
  редактирование в формах, когда всё сведено: `Kitting.status=closed`,
  `Project.status=closed`, **`Receipt.approved`** (новый bool — всё сверено со
  сканом; ручной флаг уверенности «внесено правильно», **к проекту не привязан**,
  чисто для удобства/доверия). Лот «заморожен», когда закрыт его проект или
  origin-акт — отдельных полей на лот не плодим.
- **Переоткрытие с учётом потомков.** Лист цепочки (вниз никто не ссылается) —
  свободно; узел с потомками (лот потреблён/передан/закрыт ниже) — с предупреждением
  «поедут N зависимых». Осознанность, не запрет (в духе «форма переспрашивает»).
- **Префилл себестоимости — кнопка остаётся явной.** UX не меняем: ручная «обновить
  по акту изготовления», ничего тихо под капотом. Для удобства — в UI **подсвечиваем
  расхождение** снимка с текущей суммой лотов комплектации (видно, когда жать). Сам
  снимок рождается на `closed` лота прибора (момент Х сместился с открытия акта).
**Последствия:** Принцип README «неизменяемый ledger» переписан на «мутабельные
документы + пересчитываемая проекция + мягкий замок»; добавлен принцип про
agile-ДНК/замок/переоткрытие; ER: `Receipt.approved`. Открытые вопросы прошлой
записи закрыты. Реализация: пересчёт `StockMovement` в сигналах/`save()` документов;
guard переоткрытия по наличию потомков; форм-level блокировка по замку.
**Следующий шаг:** переходим к **реализации** (держать модель в голове уже тяжело,
хочется видеть). Но СНАЧАЛА обсудить **принципы построения UI** — в первой попытке
PLM провалился именно UI.

## 2026-06-29 — Закупки в два уровня, бюджет, помощник-линза, Kitting как процесс
**Тип:** решение
**Контекст:** Интервью по UX закупок. Реальная работа: много заказов, их надо
**консолидировать** (один поток общения с поставщиком) и **нарезать по проектам**
для оплаты с разных ФЛС. Пример-эталон: партия 20 одинаковых приборов делится по 5
проектам (6/6/6/1/1) — поставщик везёт одной кучей, но оформляет 5 УПД (по одной на
проект), и на складе рождается 5 наборов лотов, каждый в своём проекте. Плюс
выяснилось главное напряжение: **составление заказа** сейчас держится в голове
экспертно (хватит ли Item на комплектацию, хватит ли денег) — больно, ошибки, риск.
Ради снятия этого PLM и затевался.
**Итог:**
- **Бюджет — новое измерение.** Источник оплаты = проект (ФЛС не моделируем, деталь).
  `Project.budget` (деньги на материалы), `Item.estimated_cost` (оценочная стоимость,
  руками, по мере КП). Бюджет-отчёт двусторонний: **планируемый** (открытые строки —
  по оценке, закрытые — по факту УПД) / **фактический** (строго `Lot.unit_cost×qty`).
- **Помощник-линза — центральный узел, но НЕ таблица.** Сквозная вычисляемая
  проекция по всем проектам сразу (та же ДНК, что дефицит/балансы): красит строки
  BOM красным/зелёным, считает дефицит и его стоимость против бюджета, подсвечивает
  профицит на «Собственном складе» и предлагает закрыть через `Requisition` (рукой).
  «Лупа», через которую команда ведёт agile-цикл проекта от денег на счету до акта
  приёмки. Ничего сама не создаёт — артефакты (Procurement→Purchase→Receipt) заводит
  пользователь.
- **План уже лежит в модели — новых плановых сущностей не нужно.** «Красный будущий
  лот прибора» = единица `ProjectDemand`; красные строки = разузлование BOM, не
  покрытое лотами; зелёные = покрытые. Цвет вычисляется, принцип «лот из
  origin-документа» цел. **Черновой `Kitting` и есть красный лот прибора.**
- **Два уровня закупки.** Новые `Procurement` (поток общения с одним контрагентом =
  один `order.xlsx`, без project — может охватывать несколько; author/date/note/
  status `draft/sent/cancelled`) и `ProcurementLine` (item, qty-итог). Строка
  Procurement **нарезается** в `PurchaseLine` по проектам (`Σ по item = итог`).
  `Purchase` получает `procurement_id`, остаётся полноценным проектным документом со
  своим `status` (каждый кусок застревает на согласовании сам) — author/date оставлены
  и на Purchase, и на Procurement (1:1 случай частый, обе даты осмысленны: переговоры
  / подписание). Поставщик — **по-прежнему только в `Receipt`** (чистота БД важнее;
  на этапе заказа контролируется оффлайн). Разные контрагенты = разные `Procurement`
  (ЭКБ / металл / платы), а не разбивка внутри одного.
- **`Kitting` переосмыслен: инструмент ведения сборки лота, не атомарный акт.** Это
  снимает злейшую оффлайн-боль — «вспоминать всю комплектацию в конце, когда приборы
  уехали». Статус `wip → closed` (closed = «сделано всё, что можно», не «отменён»;
  `cancelled` — для брошенных). `KittingLine` добавляется **по факту интеграции
  (пайки)**, получает поле `date` (история сборки) и **сразу постит `ISSUE`** в
  ledger (бизнес-дата из строки). `+RECEIPT` на лот прибора — при `closed`. Строки/
  движения `wip` провизорны и правятся до закрытия; после — компенсирующий `RETURN`.
  Вычисляемый «резерв» из прошлой идеи отпал — расход реальный сразу.
**Последствия:** ER пополнен `PROCUREMENT`/`PROCUREMENTLINE`, поля `Project.budget`,
`Item.estimated_cost`, `Purchase.procurement_id`, `KittingLine.date`; уточнена
семантика `Kitting.status` (wip/closed/cancelled). Обе диаграммы README и принципы
обновлены, `Procurement` добавлен в список документов с автором. Помощник-линза и
двусторонний бюджет-отчёт — **вычисляемые**, без таблиц (реализация при Django-бэке).
**Открытые вопросы (не блокеры):**
- Точные правила правки/удаления строк `wip`-акта vs строгая иммутабельность ledger —
  зафиксировать при реализации учёта.
- Экспорт `order.xlsx` из `Procurement` (формат для поставщика) — при реализации.
- Префилл `unit_cost` лота прибора теперь на `closed` (не на создании) — учесть в
  помощнике пересчёта себестоимости.

## 2026-06-28 — Вложения: единая `Attachment` (exclusive arc), файлы на диске
**Тип:** решение
**Контекст:** Нужны PDF/сканы в двух местах: datasheet'ы покупных `Item` и сканы
подписанных накладных/актов (несколько файлов на владельца). Разобрали с
предметником (новичок в БД) технику до байтов/битов и индексов, чтобы выбор был
осознанным, а не «по UX».
**Итог:**
- **Хранилище — диск, не БД.** `FileField` → `MEDIA_ROOT`; в таблице только путь +
  метаданные. BLOB в MySQL отвергнут: раздувает дампы, упирается в
  `max_allowed_packet`, грузит память (datasheet 10–30 МБ). Под reg.ru shared это
  существенно.
- **Модель — одна таблица `Attachment` с «exclusive arc»:** семь nullable-FK
  (`item`/`receipt`/`transfer`/`kitting`/`inventory`/`writeoff`/`requisition`),
  задан **ровно один** (инвариант: `CheckConstraint` в БД + `clean()` в приложении).
  Тот же паттерн, что origin у `Lot` — сознательно переиспользован ради
  единообразия. Отвергнут `GenericForeignKey` (`type+id`): теряет настоящий FK →
  нет ссылочной целостности и каскада, кривые JOIN'ы. Разобрали, что экономия места
  у `type+id` мнимая: в InnoDB NULL-колонки стоят ~1 бит в NULL-bitmap (значение не
  пишется вовсе), реальная цена 7 FK — индексы, а не байты строки; на масштабе
  таблицы (десятки-сотни строк) ничтожно. Решение по целостности, не по байтам.
- **Поля «тип файла» нет** — datasheet/скан ясно из контекста формы-владельца
  («достаточно из любого места ссылки на файл»).
- Метаданные: `filename`, `size`, `content_type` (PDF/JPEG/PNG), `label?`,
  `uploaded_at`, `user` (автор загрузки — в духе «авторство на документах»).
- **Форматы:** PDF + изображения (фото-сканы с телефона часто не PDF).
- **Охват:** `Item` + все 6 документов (`Receipt`/`Transfer`/`Kitting`/`Inventory`/
  `Writeoff`/`Requisition`).
**Последствия:** ER пополнен сущностью `ATTACHMENT` (7 FK-связей 1:N + `USER 1:N`),
добавлен принцип в README. Продуктовую блок-схему **не трогали** — вложения
сквозной документарный аспект, не шаг потока; форсить их в flow исказило бы её.
Реализация (когда дойдём до Django-бэкенда): `models.CheckConstraint` на «ровно
один FK», валидация `content_type`/размера, удаление файла с диска при удалении
строки.
**Закрыто сразу (2026-06-29):**
- **Версия СУБД — ОК.** На хостинге MySQL 8.0.25 (> 8.0.16) → `CHECK` работает
  нативно, упор на `clean()` не нужен.
- **Вложения у всех `Item`, без ограничения по типу.** К производимым приборам
  грузим чертежи/спецификации, к покупным — datasheet'ы. Соглашение «datasheet
  только покупным» снято.
- **`on_delete` владельца = `CASCADE` + удаление файла с диска.** При удалении
  `Item`/документа форма переспрашивает и предупреждает, что файлы тоже удалятся.
  Доверяем пользователю: лишний повторный дроп и ручное восстановление лучше
  осиротевшего мусора на диске. (Реализация: сигнал/`delete()` чистит файл —
  `on_delete=CASCADE` сам файлы с диска не трёт.)

## 2026-06-28 — Проект — свойство лота; авторство на документах; закрытие через акты
**Тип:** решение (пересматривает черновик этой же сессии)
**Контекст:** Ревизия `StockMovement`. Сначала предложил `project_id` и `created_by`
**на движении**; предметник возразил, и его модель оказалась чище.
**Итог:**
- **Авторство — на документах, не на движении.** Движение не существует без
  документа (`source` обязателен), документы редки и всегда заведены сотрудником →
  автор и дата берутся из документа. `created_by` добавлен в `Purchase`/`Receipt`/
  `KittingAct`/`Transfer`, убран из `StockMovement`. Сущность `Employee → User`
  (команда, не наёмники); физически Django `auth_user`, reserved-word `USER` не
  мешает (неймспейс таблицы).
- **`project` — свойство лота (`Lot.project`, immutable), не движения.** Лот живёт
  в одном проекте от origin-документа до закрывающего; `movement.project` выводится
  из лота, межпроектного переноса одного лота **нет**. `StockMovement.project_id`
  убран. «Общее через `NULL`» заменено явными **внутренними проектами**
  (`Project.kind`): «Собственный склад» (белые/на балансе) и «Свободные неучтённые»
  (серые/списанные). `project` на документах — обязателен.
- **Закрытие проекта — закрывающими документами.** Невыбранные лоты (`qty>0`)
  сводятся в 0 актом передачи/списания/постановки на баланс. Реюз = **акт
  инвентаризации** рождает новые лоты во внутреннем проекте, `Lot.predecessor`
  хранит ссылку на закрытый лот (раньше связь была оффлайн). Команда работает
  спринтами — «обнуляется» в конце, не теряя контроль над остатками.
- Отвергнут мой «двуногий межпроектный перенос одного лота» — заменён на
  `close+recreate`, чище и точнее отражает физику «обнуления».
**Последствия:** ER: `User`, `created_by` на 4 документах, `Lot.project` +
`Lot.predecessor`, `Project.kind`; убраны `StockMovement.project_id`/`created_by`;
`project` на `Purchase`/`Receipt` стал обязательным; связи `PROJECT 1:N LOT (home)`,
`LOT 1:N LOT (predecessor)`, `USER 1:N документы`. Обновлены принципы README.
**Акт инвентаризации** (3-й origin лота + механизм реюза) теперь на критическом
пути — проектировать следующим. Продуктовую схему обновить под это (диспозиции
закрытия = передача/списание/баланс; реюз через инвентаризацию).

## 2026-06-28 — Себестоимость произведённого лота: снимок + кнопка пересчёта
**Тип:** решение
**Контекст:** Нужны были оба режима цены прибора: (а) строгая сумма стоимостей
компонентов (ради этого и затевался PLM — раньше считали руками), (б) свободная
цена поставки с добавленной оффлайн стоимостью работ. Вопрос: как ужить оба в
одном `unit_cost` без раздувания схемы.
**Итог:** Выбран **третий вариант — снимок + ручной помощник**, схему **не
трогаем**.
- `unit_cost` произведённого лота = **снимок**: префилл при создании `KittingAct`
  суммой `Σ (line.qty × component_lot.unit_cost)`, дальше статичен, правится
  руками (стоимость работ поверх материалов).
- Пересчёт — **ручная кнопка «обновить по акту изготовления»** на движке
  разузлования. Три обязательных свойства помощника (в форме, не в модели):
  **(1) один уровень** — опираться на сохранённый `unit_cost` детей, не
  перезаписывать их рекурсивно (иначе тихо сносит их ручную работу); стоимость и
  так всплывает снизу вверх. **(2)** предупреждать о компонентах без цены (иначе
  сумма молча занижена). **(3)** показывать дифф перед записью (затирает ручную
  наценку — разница и есть стоимость работ).
- Для покупных лотов (`origin=Receipt`) кнопка неактивна.
**Последствия:** Никаких новых полей. Добавлен принцип в README (семантика
`unit_cost` для произведённых лотов). Реализация: подсчёт-префилл при создании
акта + помощник пересчёта; опционально live-отчёт «себестоимость по материалам»
как проверка, не подменяя снимок.

## 2026-06-28 — Заводской номер, разузлование актов, передача по Lot
**Тип:** решение
**Контекст:** Проектирование заводского номера для производимых изделий + как
быть с многоуровневым производством (батч плат → сборка прибора) и передачей.
**Итог:**
- **`Lot.serial_number`** — ручной текст, **не ключ**, nullable, без `unique`.
  Модель «серийник = экземпляр» (`Lot` на прибор, `qty=1`), но поле текстовое →
  допускает диапазон («05–25») для батч-передач. Один `KittingAct` может породить
  как N партий-приборов (номер на каждую), так и одну партию на N — мнимый минус
  «30 приборов = 30 актов» снят (акт 1:N лотов уже в модели).
- **Область:** серийник только конечным изделиям; промежуточные партии (батч
  печатных плат) — без номера.
- **Разузлование акта изготовления = рекурсия по цепочке** `Lot →(origin)
  KittingAct →(lines) Lot → …`, тот же движок, что BOM. «Паспорт прибора» и
  объединённый «акт на прибор» — **проекция на печать, не запись в БД**. Идею
  «spawn общего акта» / self-FK `parent_act` отвергли: единственный источник
  правды — атомарные акты + цепочка лотов, дублировать нельзя (риск рассинхрона).
  Ключевой довод: промежуточную партию плат (кейс 5+25) нельзя схлопнуть — она
  реальный складской `Lot` с остатком.
- **Передача — только по `Lot`.** `Transfer` (шапка: project, date, № накладной)
  + **`TransferLine`** (`lot`, `qty>0`, `display_name?`). Убраны `Transfer.item`/
  `qty`; `item` выводится из `lot.item`. `display_name` переопределяет
  `received_name` в исходящей накладной («Прибор X. Заводские номера 05–25»).
  Передаём только готовое железо; КД и пр. — оффлайн.
- `KittingAct.qty` оставили как есть (стал denormalized, но не трогаем — без
  лишних возможностей).
**Последствия:** Обновлены обе диаграммы README (ER: `Lot.serial_number`,
`Transfer`/`TransferLine`, связи `LOT 1:N TRANSFERLINE`, `TRANSFER 1:N
TRANSFERLINE`; убрана `ITEM 1:N TRANSFER`) и принципы модели. Передача теперь
lot-центрична, как и весь склад.

## 2026-06-27 — Закупки: Purchase, покрытие через Receipt, origin партий
**Тип:** решение
**Контекст:** Финал блока закупок/партий. Три уточнения: имя `Order` конфликтует
с SQL; нужна ли двойная ссылка партии на заказ; как партия появляется не от
закупки.
**Итог:**
- Сущность заказа названа **`Purchase` / `PurchaseLine`** (вместо `Order` —
  зарезервированное слово). `Receipt.order_id → purchase_id`.
- **Убран `Lot.order_line_id`.** Партия ссылается только на `Receipt` (УПД
  синхронизирована с закупкой). Покрытие строки выводится по цепочке
  `Lot → Receipt → Purchase` + совпадению `item`; для однозначности — **уникальная
  пара `(purchase, item)` в `PurchaseLine`**. Кейс 100 = 60 + 40 работает (две
  УПД под одной закупкой → две партии). Связи стали однозначнее, потерь нет.
- **Происхождение `Lot` — всегда origin-акт, не «из воздуха»:** поставка
  (`Receipt`), изготовление (`KittingAct` = акт комплектации/изготовления, добавлен
  `Lot.kitting_act_id`), инвентаризация (акт — **планируется**). Инвариант: ровно
  один origin задан. Этим закрыт прежний открытый вопрос про самодельные партии.
**Последствия:**
- `KittingAct` теперь и потребитель (списывает компоненты), и производитель
  (создаёт партию изделия) — детали проработать при дизайне комплектации.
- Добавить сущность «Акт инвентаризации» при проработке учёта остатков.
- При реализации на Django зафиксировать имена таблиц (избежать SQL-слов);
  инвариант origin и `(purchase, item)` — через констрейнты/`clean()`.

## 2026-06-27 — Lot-центричный разворот склада
