# Волна 1 — план реализации

Живой план-чеклист первой волны движка-СППР. Источник «что делаем сейчас и в
каком порядке». Значимые решения по ходу уносятся в [JOURNAL.md](JOURNAL.md),
здесь остаются мелкие развилки реализации.

**Цель волны 1:** локально, своими глазами увидеть, как движок-СППР красит
реальность — проектный **дефицит** и **карту остатков** по складам-проектам.
Без прода.

## Объём

**Входит:** полная схема БД (миграции) → демо-данные через admin/сид → движок
дефицита + карты (чистый Python + юнит-тесты) → read-only DRF-проекции →
VS Code-каркас фронта → две витрины (дашборд дефицита, карта остатков).

**НЕ входит (отдельная экскурсия в соседний проект):** деплой/прод (reg.ru
shared), cron, `verify`-периодика, рантбук.
_Апдейт 2026-07-01: экскурсия деплоя проведена — прод поднят (Django 6.0/Python 3.14),
рантбук — `deploy/README.md`, детали — JOURNAL 2026-07-01. cron/`verify`-периодика пока не
делались._

**Отложено в следующие волны:** pegging/разворот источников, командный дашборд +
`Procurement`, два числа бюджета + себестоимость/экономия, призрачный кокпит
`Kitting`, интерактивные формы редактирования (автосейв/призрачная строка),
UI вложений.

## Решения (утверждены 2026-06-30)

- **Рантайм: Python 3.14 + Django 6.0.x** (DRF 3.17.x). _Обновлено 2026-07-01:_
  на reg.ru shared под `/opt/python/` реально есть бинари до 3.14, поэтому ушли с
  4.2/3.10.1 на Django 6.0/Python 3.14 (см. JOURNAL 2026-07-01). Локальный venv —
  на Python 3.14 (локально = прод). Актуальные фичи Django разрешены.
- **Локальная БД — MySQL 8.0.25** (как на хостинге; `CHECK` нативно, utf8mb4,
  InnoDB). Не SQLite — иначе «локально» врёт относительно констрейнтов модели.
- **Сразу вся замороженная схема** (все сущности как Django-модели) — origin-FK
  у `Lot` форсят существование всех origin-сущностей для exclusive-arc. Движок и
  формы при этом — строго подмножество волны 1.
- **Ввод данных — Django admin + сид-команда.** React в волне 1 показывает
  только read-only витрины движка (канон «проекции read-only»).
- **Имена:** Django-app `plume`, конфиг-пакет проекта `config`.

## Этапы

### Этап 0 — Каркас и локальное окружение
- [x] Django-проект (конфиг `config`) + app `plume`, DRF подключён.
- [x] Локальный MySQL 8.0.25 (utf8mb4/InnoDB), настройки через env.
- [x] Vite + React + TS, dev-прокси на Django.
- _Готово, когда:_ `runserver` + `vite dev` поднимаются, browsable API и пустой
  фронт открываются.

### Этап 1 — Модели и миграции (полная замороженная схема)
- [x] Все сущности как Django-модели, имена таблиц без SQL-reserved слов.
- [x] Инварианты: exclusive-arc origin у `Lot` и у `Attachment` (`CheckConstraint`
  + `clean()`); unique `(purchase, item)`; `Lot.project == Receipt.project`;
  синглтоны по `Project.kind`.
- [x] Деньги/qty — `DecimalField`. Django admin на все сущности.
- _Готово, когда:_ `migrate` проходит, admin открывает всё, констрейнты на месте.

### Этап 2 — Сид демо-данных (`seed_demo`)
- [x] Синглтоны «Собственный склад» / «Свободные неучтённые» (`Project.kind`),
  дефолтный `Location` «Основной склад».
- [x] Демо-проект: прибор с 2-уровневым BOM (видна терминация на лотах),
  `Receipt→Lot`, открытый `Purchase/PurchaseLine` (`sent`), wip-`Kitting` с парой
  `KittingLine`.
- _Готово, когда:_ одной командой получаем проект с непустым дефицитом и непустой
  картой.

### Этап 3 — Движок-проекции (чистый Python + юнит-тесты) ← ядро волны
- [x] `rebuild_movements(lot)` (пересборка `StockMovement` из документов, тип из
  origin+знак) + `lot_qty`; аксессор `lot.origin`/`origin_kind`.
- [x] Дефицит проекта: `надо (BOM×потребность, 1 ур., терминируя на лотах) − склад
  (свои лоты, живой qty) − заказано (Σ max(0, PurchaseLine.qty−поступившее) по
  `sent` + остаток wip-`Kitting`)`. Тройной разбор `✓/●/▲`, цвет = worst-of +
  бейдж, отрицательный остаток не клампим (+ аномалия-бейдж).
- [x] Карта остатков по складам-проектам: где Item лежит по всем проектам с
  доступным qty.
- [x] Юнит-тесты на каждую формулу (тесты — гарантия корректности вместо
  прод-обкатки).
- _Готово, когда:_ тесты зелёные, числа согласованы на демо-данных.

### Этап 4 — DRF read-only проекции
- [x] Сериализаторы + GET-эндпоинты: дефицит проекта (строки с тройным
  разбором/цветом), карта остатков по Item. Только чтение.
- _Готово, когда:_ эндпоинты отдают JSON проекций, видно в browsable API.

### Этап 5 — Фронт-каркас (VS Code-подобный)
- [x] Layout по канону: activity-bar + дерево + рабочая область (одна вкладка) +
  статус-бар; API-клиент.
- [x] Переиспользуемые компоненты словаря статусов (●▲✓🔒○), цвет = смысл.
- _Готово, когда:_ каркас рендерится и тянет данные с бэка.

### Этап 6 — Витрины волны 1 (read-only проекции движка)
- [x] **Проектный дашборд дефицита** (ось Project): BOM-строки с тройным разбором
  `4✓·3●·3▲`, цвет шапки = worst-of + бейдж, аномалия-бейдж.
- [x] **Карта остатков** (north-star, ось Item): где Item лежит по всем
  складам-проектам.
- [x] Окружение из связей минимально (экран Item: where-used/лоты), read-only с
  подсказкой «менять в документе».
- _Готово, когда:_ открываешь проект — видишь крашеный дефицит; открываешь Item —
  видишь карту. Глазами видно, что движок работает.

## Решения по ходу / открытые вопросы

- **Драйвер MySQL — PyMySQL** (`config/__init__.py: install_as_MySQLdb`), не
  mysqlclient: чистый Python, без сборки C-расширения на Apple Silicon. СУБД —
  настоящий MySQL 8.0.25, паритет по констрейнтам сохранён.
- **MySQL 8.0.25 в Docker — под эмуляцией amd64** (`platform: linux/amd64`): у
  образа 8.0.25 нет arm64-манифеста. Порт хоста **3307** (не 3306, чтоб не
  конфликтовать). `docker compose up -d` в `backend/`.
- **Node поднят до 22 LTS (nvm), Vite 8.** Скаффолд Vite требует Node ≥20.19;
  было 20.18 → поставлен Node 22 (`nvm install 22 && nvm alias default 22`),
  rolldown-биндинг чинится чистой переустановкой (`rm -rf node_modules
  package-lock.json && npm install`).
- **`plume`-юзеру выданы глобальные права** (root: `GRANT ALL ON *.*`) — чтобы
  Django мог создавать тестовую БД `test_plume` (нужно для `manage.py test`).
- **Vite слушает `localhost` (IPv6 `::1`), не `127.0.0.1`** — открывать
  `http://localhost:5173/`. Прокси `/api`→Django работает.
- **`Lot.qty` добавлено в схему** (дом рождённого количества) — единственная
  разморозка, см. JOURNAL 2026-06-30.
- **Рантайм пересобран под хостинг (рефакторинг):** venv на **Python 3.10.1**
  (pyenv: `pyenv install 3.10.1`), **Django 4.2.30 + DRF 3.15.2**. Изначально по
  ошибке собрали на 3.11/5.1 → правка только в зависимостях + перегенерация
  миграции (`condition=`→`check=`); исходники не менялись. См. JOURNAL (грабли).
  _Устарело 2026-07-01:_ ушли на **Django 6.0 + Python 3.14** (`check=`→`condition=`,
  `STORAGES`, спуф mysqlclient ≥ 2.2.1), т.к. на хостинге доступен Python 3.14.
  См. JOURNAL 2026-07-01.
- (значимое — уносится в `JOURNAL.md`)

## Локальный запуск (памятка)

```
# БД
cd backend && docker compose up -d           # MySQL 8.0.25 на :3307
# бэк (нужен backend/.env — скопировать из .env.example)
cd backend && .venv/bin/python manage.py migrate
cd backend && .venv/bin/python manage.py runserver
# фронт (Node 22 через nvm)
cd frontend && npm run dev                    # http://localhost:5173
```
