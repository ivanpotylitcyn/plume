# Волна 5 — записываемая передача (Transfer → отгрузка заказчику)

Чеклист пятой волны. Цель — открыть **правую половину** потока
`комплектация → передача`: завести передачу (`Transfer`) прямо в UI и отгружать
готовое железо заказчику по накладной, списывая партии со склада проекта
(`−ISSUE`). Продолжает паттерн волн 2–4 (записываемый документ + движок =
единственный источник правил + `rebuild_movements` = склад остаётся проекцией).
Значимые решения — в [JOURNAL.md](JOURNAL.md) (запись 2026-07-02, «Волна 5»).

**Цель волны 5:** из режима «Передачи» завести накладную проекта → выбрать
отдаваемую партию (пикер лотов проекта с остатком > 0) → указать кол-во и имя для
накладной → строка списывает партию (`−ISSUE`), живой остаток источника проседает;
правка/удаление — коррекция (источник возвращает остаток).

## Объём

**Входит:** проекция `transfer_cockpit` + пикер `project_available_lots` →
записываемые DRF-эндпоинты (создать накладную / строки / автосейв / удаление) →
шестой режим фронта «Передачи» (📦) со списком, созданием и кокпитом →
юнит-тесты на мутации/пикер/guard-чужого-проекта.

**НЕ входит (следующие волны):** закрытие проекта целиком (`Writeoff`/`Requisition`/
`Inventory` + панель сведения остатков + `close_project`), командный свод +
консолидированный `Procurement`-дашборд, экспорт `order.xlsx`, pegging, два числа
бюджета + себестоимость/экономия, логин-экран, UI вложений.

## Решения (см. JOURNAL 2026-07-02, «Волна 5»)

- **Схему БД не трогаем** — `Transfer`/`TransferLine` уже были. Диаграммы README
  не меняются.
- **Мягкий замок «отгружено» добавлен на обкатке** (`Transfer.posted` — разморозка
  схемы, миграция `0002`). Зеркалит `Receipt.approved`: под замком форма read-only,
  снятие ничего не разрушает (guard по потомкам не нужен), `post` требует непустую
  накладную. Корректность правки вне замка держат guard «лот не потреблён ниже»
  (`_lot_consumed_downstream`, уже учитывает `transfer_lines`) + `PROTECT` на строках.
  Загрузку подписанного скана под замок отложили в отдельную волну «UI вложений»
  (`Attachment` сквозной — делаем один раз для всех документов).
- **Движок уже строил проекцию.** `rebuild_movements` с волны 1 генерит `−ISSUE`
  по `lot.transfer_lines` (место — `MAIN`). Волна 5 добавляет только записываемую
  надстройку — кокпит + мутации + пикер.
- **Кол-во не клампим по остатку** (как в приходе/дефиците): переотдать можно, лот
  уйдёт в минус — недостача информативнее нуля (мутабельная ДНК). UI подсвечивает
  отрицательный остаток источника (▲).
- **Отдаём только своё** — `add_transfer_line` требует `lot.project == transfer.project`
  (чужой проект — через требование `Requisition`, следующая волна). Проект — свойство
  лота: передача не меняет `Lot.project`, лот доживает в проекте до конца.
- **Отображаемое имя строки** (`TransferLine.display_name`) — что печатается в
  накладной. Пусто → авто-метка лота (`#id зав.№/название/артикул`).

## Этапы

### Этап 1 — Движок передачи (чистый Python + тесты) ← ядро волны
- [x] `transfer_cockpit(transfer)` — шапка (проект/№/дата) + строки-лоты (партия,
  кол-во, имя в накладной, живой остаток источника) + итог отданного.
- [x] `project_available_lots(project)` — пикер отдаваемых лотов (live>0, все item).
- [x] Мутации: `create_transfer` (№ обязателен, дата → сегодня если пусто),
  `add_transfer_line` (guard чужого проекта, qty>0, авто-имя), `update_transfer_line`
  (автосейв кол-ва/имени), `remove_transfer_line` — каждая с `rebuild_movements`.
- [x] Юнит-тесты: строка списывает `−ISSUE`, кокпит-итоги + живой остаток, авто-имя,
  пустой № → отказ, чужой проект → отказ, nonpositive → отказ, переотдача в минус
  (не клампим), правка/удаление восстанавливают источник, пикер исключает нулевые/чужие.
- _Готово, когда:_ тесты зелёные, живой остаток источника совпадает с движком.

### Этап 2 — Записываемый DRF
- [x] `GET/POST /api/transfers/`, `GET /api/transfers/<pk>/` (кокпит),
  `POST .../lines/`, `PATCH/DELETE /api/transfer-lines/<pk>/`,
  `GET /api/projects/<pk>/available-lots/` (пикер).
- _Готово, когда:_ HTTP-путь через test Client: create 201, строка 201, автосейв/
  удаление 200, пикер 200, валидации 400.

### Этап 3 — Фронт: режим «Передачи» + кокпит + пикер
- [x] Шестой режим activity-bar (📦): список накладных + призрачная строка
  «＋ Новая передача» (проект + № + дата).
- [x] `TransferView` — кокпит: строки с автосейвом кол-ва и имени (переиспуемый
  `CommitInput`) и удалением, призрачная строка добавления (пикер партии проекта),
  подсветка отрицательного остатка источника, итог отданного.
- [x] API-клиент: типы кокпита передачи + пикер + мутирующие хелперы.
- _Готово, когда:_ заводишь накладную → отдаёшь партию → живой остаток источника
  проседает; числа сходятся с движком.

## Докрутка на обкатке (2026-07-02)

По итогам клика предметника (см. JOURNAL, «докрутка на обкатке»):
- [x] **Замок «отгружено»** (`Transfer.posted`): `post/unpost_transfer`, read-only
  под замком, глиф 🔒/📦 в списке и шапке. Скан подписанной накладной — отдельной
  волной «UI вложений».
- [x] **Экран «Изделие»: слияние блоков** — «Карту остатков» убрали, остались
  «Партии на складе» (единственный блок остатков, живой > 0 подсвечен).
- [x] **Экран «Изделие»: «Отгруженные партии»** (`engine.item_shipments`) — куда и по
  какой накладной ушло заказчику; замыкает петлю визуально.
