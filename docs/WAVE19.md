# Волна 19 — консистентность закупочного контура + шов «движок ↔ вью»

**Статус:** план принят, развилки сняты (2026-07-20). Не стартовала.

**Контекст.** Plume в проде, пользователи работают, БД нагружена. Самая отработанная
часть — **Изделия**: type-ahead-пикер, статус-глиф ✓/○, замок формы, чистая форма.
Волна 19 подтягивает к этому «стилю и настроению» закупочный контур
(Закупка → Заказ → УПД) и одновременно закладывает **шов между движком и
представлением** — чтобы поверх одного API жили разные темы интерфейса.

**Две цели волны, равноправные:**

1. **Функциональная** — закупки/заказы/УПД становятся такими же зрелыми, как изделия.
2. **Архитектурная** — фронт перестаёт быть сросшимся с бэкендом: движок (Иван) даёт
   стабильный контракт, вью (кто угодно) строится поверх. Проверка шва — скелет
   второй темы («надувной», не-VS-Code).

## Принятые развилки (2026-07-20)

| # | Развилка | Решение |
|---|----------|---------|
| Р1 | `cancelled` у Закупки/Заказа | **Отмена = удаление**, как у ордеров с волны 13. Ось строго `{draft, posted}` |
| Р2 | Пеггинг: сколько заказов на проект под планом | **Несколько**, выбор конкретного заказа прямо в строке аккордеона |
| Р3 | Откуда поставщик при «Заказ → УПД» | **`Procurement.contractor`** — контрагент у Закупки (она и есть «один поток общения») |
| Р4 | Объём темизации в этой волне | **Шов + скелет второй темы** (оболочка + режим «Изделия») |

## Сквозное решение: одна ось статуса на весь продукт

Все сущности со статусом переезжают на **`DocStatus {draft, posted}`** — тот же enum,
что уже несут `StockDocument` и `Item` (волна 17). Одна ось → один глиф, один замок,
один набор правил, ноль дублирующего кода.

- Значения в БД — `draft` / `posted` (не заводим третий синоним `approved`: `Item`
  и шесть ордеров уже на `posted`, менять их ради слова дороже пользы).
- **Подписи — по сущности**: «Черновик» / «Утверждена» (Закупка), «Утверждён» (Заказ),
  «Проведён» (ордер), «Зафиксировано» (Изделие). Меняются в одном словаре.
- **Глиф — везде один**: ✓ зелёный (posted) / ○ оранжевый (draft), компонент
  `ItemStatusGlyph` переименовать в `StatusGlyph` и применить ко всем спискам.

**Что это ломает и как чиним — Ф1.**

## Ф1. Статусы: свести всё к `draft ⇄ posted`

### Найденное в коде (обоснование)

- `Purchase.Status.PARTIAL` и `RECEIVED` — **мёртвые**: не присваиваются нигде
  (`engine.py`, `views.py`, `admin.py`), 0 упоминаний в тестах. Существуют только в
  enum и в README. «Получен / частично получен» — это **вычисляемая** величина из
  приходов, у неё уже есть источник: `_line_received` (`engine.py:416`).
- `Project.Status.DRAFT` — **мёртвый** (`create_project` сразу ставит `ACTIVE`).
- `Procurement.Status` и `Purchase.Status` — свои enum-ы, дублирующие `DocStatus`.

### Работы

- [ ] **Модель.** `Procurement.status` и `Purchase.status` → `DocStatus`.
      `Project.Status` → `{active, closed}` (убрать мёртвый `draft`; это **жизненный
      цикл**, а не замок — осознанно не сводим к draft/posted).
- [ ] **Миграция данных** (прод живой, аккуратно):
      `sent → posted`; `partial`/`received` → `posted` (если вдруг есть строки —
      проверить запросом, ожидаем 0); **`cancelled` → `draft`**, НЕ удалять
      автоматически. Миграция печатает количество затронутых; отменённые закупки/заказы
      Иван разбирает руками через UI (удалить или вернуть в работу).
- [ ] **Движок.** Убрать `cancel_purchase` / `restore_purchase` / `cancel_procurement` /
      `restore_procurement`; `send_*`/`unsend_*` → `post_*`/`unpost_*` (имена как у
      ордеров). Все `exclude(status=CANCELLED)` в счётах (`_procurement_pegs`,
      `_purchased_on_order` `engine.py:432`, `views.py:859`, `engine.py:2458`) —
      исчезают вместе со статусом: удалённого в счёте нет по построению.
      `_require_purchase_draft` / `_require_procurement_draft` остаются, переезжают на
      общий `_require_draft`.
- [ ] **Вычисляемое «получено»** вместо мёртвых статусов: `purchase_cockpit` уже даёт
      `received`/`remaining` по строкам — поднять агрегат в шапку заказа и в строку
      списка (`✓ получен` / `● частично` / `▲ не получен` — глифы дефицита, не статус-замок).
      Две оси не путать: **замок** (draft/posted, ✓/○) и **покрытие** (▲/●/✓).
- [ ] **API.** Маршруты `send/unsend/cancel/restore` → `post/unpost` (+ `DELETE` уже есть).
- [ ] **Фронт.** `StatusGlyph` во всех списках со статусом: закупки, заказы, ордера
      (все шесть видов), проекты. Убрать ветки `cancelled`/`restore` из
      `ProcurementView.tsx:66-67`, `PurchaseView.tsx:72-73`.
- [ ] **README.** Обе диаграммы: `PROCUREMENT.status`, `PURCHASE.status`, `PROJECT.status`
      + текст §про закупки (строки 517–519).

**Риск:** удаление отменённого заказа упирается в `PROTECT` (на нём висят `Receipt`).
Проверить `delete_purchase` — friendly-guard должен объяснять человеческим текстом, а
не отдавать 500. То же для `delete_procurement` с висящими заказами.

## Ф3. Изделие: цена — вне замка

- [ ] `estimated_cost` **не подчиняется** гейту фиксации: `update_item` пропускает
      только это поле у `posted` изделия (остальные — по-прежнему `_require_item_draft`).

      Роллап (`_rollup_cost`) уже пишет цену в обход гейта — теперь это не исключение,
      а правило.
- [ ] Форма изделия: поле «Оценка» всегда редактируемо, снятие замка не требуется →
      статус не «слетает» в черновик из-за цены.
- [ ] **README:** у `ITEM` в диаграмме нет `status` — добавить
      (`string status "draft/posted — фиксация по библиотеке"`). Дрейф с волны 17.

## Ф5. Форма закупки: один функциональный блок вместо двух

Сейчас закупка показана двумя разъехавшимися блоками: строки плана
(`procurement_cockpit`) и привязка к проектам (`procurement_pegging`). Пользователь
видит одно и то же изделие в двух местах и правит в разных. **Сливаем в один
раскрывающийся блок.**

### Целевая структура строки

```
▸ ✓ BGB707L7ESDE6327XTSA   усилитель    120 шт   ▸ разложено 100 / остаток 20
  └ ▸ РЕЗ-10К · Проект А        нужно 60   в заказ [ 60 ] → [Заказ #12 ▾]
  │   └ ИЗД-1 ×2 (потребность 20) = 40 шт     ← обратное разузлование
  │   └ ИЗД-7 ×1 (потребность 20) = 20 шт
  └ ▸ Проект Б                  нужно 40   в заказ [ 40 ] → [＋ новый заказ]
```

- [ ] **Движок: одна проекция вместо двух.** `procurement_cockpit` и
      `procurement_pegging` схлопнуть в `procurement_cockpit` (строки плана уже несут
      разбивку по проектам). Старую `procurement_pegging` и её маршрут удалить, не
      оставляя мёртвого эндпойнта.
- [ ] **Правка количества строки плана прямо здесь** (сейчас `qty` линии правится
      только в заказе; в форме закупки можно лишь увеличивать — шероховатость).
      `update_procurement_line` уже есть, довести до формы.
- [ ] **Обратное разузлование** — новая функция движка
      `item_usage_in_project(item, project)`: по потребностям проекта
      (`ProjectDemand`) обойти BOM целевых изделий и вернуть
      `[{target_item, qty_per_unit, demand_qty, total}]` — «по сколько штук этого Item
      идёт в то или иное производимое изделие проекта».
      **Переиспользовать существующую рекурсию** (`_explode_demand` / `_demand_tree`,
      `engine.py:494-590`), выделив общий обход, а не писать третий проход по BOM.
- [ ] **Пеггинг в конкретный заказ (Р2).** `peg_procurement_line` получает явный
      параметр `purchase=None`: передан — пегаем туда (проверив, что заказ под этим
      планом, этого проекта и в `draft`); не передан — прежнее поведение
      (найти-или-создать черновик проекта). `_project_purchase_under` остаётся
      фолбэком, а не единственным путём. В строке — выпадашка заказов проекта под
      планом + «＋ новый заказ».
- [ ] **Шероховатость из отчёта:** заказы не появлялись в списке «Заказы», пока не
      нажат «черновик заказа». Причина — заказ рождается лениво при первом пеге.
      Оставляем ленивое рождение (пустые заказы-призраки не нужны), но **веер заказов
      под планом виден прямо в форме закупки** — искать в другом режиме больше не надо.
- [ ] Аккордеон: раскрытие строки → проекты; раскрытие проекта → изделия-применения.
      Состояние раскрытия не персистим.

## Ф6. Заказ → УПД (прямая и обратная связь)

В 98% случаев накладная повторяет заказ 1:1 — набивать строки заново не должно быть
нужно. `Receipt.purchase` (FK, nullable) **уже существует** — достраиваем поток.

- [ ] **Движок: `create_receipt_from_purchase(purchase, user, number, date)`.**
      Создаёт `Receipt` с `project` из заказа, `contractor` — из
      `purchase.procurement.contractor` (Р3), `purchase` — ссылка на заказ; лоты
      преднабиваются по строкам заказа **остатком** (`qty − уже получено`, через
      существующий `_line_received`), `unit_cost` — из `item.estimated_cost` (или 0),
      имя лота — из `item.description`. Всё это черновик: пользователь правит перед
      проведением.
      Повторный вызов для частично закрытого заказа даёт вторую накладную на остаток —
      сценарий «поставка частями» работает без ручной работы.
- [ ] **Кнопка «Создать УПД» в форме заказа** → открывает готовый черновик прихода.
- [ ] **Обратная связь.** `purchase_cockpit`: по каждой строке — какими накладными она
      закрыта (`[{receipt_id, number, date, qty}]`). Источник — лоты прихода
      (`Lot.origin → Receipt.purchase`), сопоставление по `item`; отдельного FK
      «строка заказа ↔ лот» не заводим (схема не меняется, атрибуция по изделию
      однозначна — `PurchaseLine` уникален по `(purchase, item)`).
- [ ] В форме УПД — ссылка на заказ-источник (`ReceiptView.tsx:67` уже привязывает
      заказ вручную; теперь связь чаще будет проставлена сама).
- [ ] README: описать поток «Заказ → УПД» в разделе закупок; на диаграмме связь
      `PURCHASE ||--o{ RECEIPT` уже есть.

## Ф7. Шов «движок ↔ вью» + скелет второй темы

**Курс (формулировка Ивана):** «Я делаю движок, логику которого не дам никому менять.
Другие пользователи поверх этого движка делают свои вью». Сейчас фронт и бэк слегка
сросшиеся — это нормально для текущей стадии, но дальше расходимся строго.

### Что уже хорошо (не ломать)

Бэкенд почти готов к этому: `engine.py` — чистые функции, возвращающие словари-проекции;
`views.py` — тонкий HTTP-слой. CSS-токены (`index.css:49-75`) уже вынесены в `:root`.

### Найденные протечки представления в движок

- [ ] **`Category.icon` отдаёт коды Codicon** (`views.py:252`, `engine.py:101-120`) —
      это иконочный набор VS Code, то есть **тема протекла в API**. Чиним: в модели
      хранится **семантическое** имя (`capacitor`, `mcu`, `connector`), каждая тема
      сама маппит его в свой набор глифов. Миграция кодов + правка `LIBRARY_CATEGORIES`.
- [ ] Ревизия остальных проекций на предмет UI-специфики (цвета, коды глифов, ширины,
      порядок колонок). Правило: **движок отдаёт смысл, тема выбирает знак.**
      Статусы (`to_order`/`on_order`/`available`) — смысл, остаются; `✓`/`▲` и цвета —
      тема.

### Раскладка фронта

```
frontend/src/
  core/            ← не зависит от темы: api-клиент, типы домена, хуки данных,
                     форматирование (money/num), словарь статусов (смысл, не глиф)
  themes/
    ide/           ← текущая тема (VS Code / JetBrains Mono), сюда переезжают
                     нынешние *View.tsx + index.css
    soft/          ← скелет второй темы: «надувные» кнопки, крупные отступы, «сайт»
  App.tsx          ← выбор темы + маршрутизация режимов
```

- [ ] Распилить `App.tsx` (1270 строк): загрузка данных и состояние → хуки в `core/`,
      разметка списков/оболочки → тема. Формы (`*View.tsx`) переезжают в `themes/ide/`
      **как есть** — переписывать их в этой волне не надо, надо только вынести из них
      обращения к API в хуки `core/`.
- [ ] Токены: разбить `index.css` на `core/reset.css` + `themes/<name>/tokens.css` +
      `themes/<name>/components.css`. Переключатель тем — атрибут на корне
      (`data-theme`), выбор в localStorage, дефолт — `ide`.
- [ ] **Скелет темы `soft`** (проверка, что шов держит): оболочка (навигация) +
      режим «Изделия» — список и форма. Самая отработанная часть продукта, значит
      самый честный тест. Остальные режимы в `soft` пока показывают заглушку
      «в этой теме ещё не оформлено» — это нормально и видно.
- [ ] **`docs/API.md`** — контракт как документ: перечень эндпойнтов, форма проекций,
      правила именования, что считается ломающим изменением. С этого момента
      изменение формы ответа — осознанный акт, а не побочка правки вью.

**Критерий приёмки шва:** ни один файл в `themes/` не ходит в `fetch`/`api` напрямую,
ни один файл в `core/` не импортирует CSS-классы и не знает про Codicon.

## Порядок исполнения

Миграции: Ф1/Ф3/Ф4 свести в **одну** миграцию схемы + одну данных, чтобы прод
