# Волна 4 — записываемый заказ (Purchase) + связь с приходом

Чеклист четвёртой волны. Цель — замкнуть **левую половину** потока
`дефицит → закупка → заказ → приход`: завести заказ (`Purchase`) прямо в UI и
связать приход с заказом, чтобы оранжевый член «заказано» в дашборде дефицита стал
**реальным и правимым** (сейчас `project_deficit` его считает из `PurchaseLine (sent)`,
но создать заказ в UI негде — колонка мёртвая). Продолжает паттерн волн 2–3
(записываемый документ + движок = единственный источник правил + кокпит-режим фронта).
Значимые решения — в [JOURNAL.md](JOURNAL.md) (запись 2026-07-02, «Волна 4»).

**Цель волны 4:** из ▲-строки дефицита нажать «＋ в заказ» → рождается/дополняется
draft-заказ проекта → отправить заказ (`sent`, ●) → завести приход и **связать его с
заказом** → поступившее гасит строки заказа (✓) и режет «заказано» в дефиците.

## Объём

**Входит:** проекция `purchase_cockpit` → записываемые DRF-эндпоинты (создать заказ /
строки / отправка-замок / отмена) → связь `Receipt↔Purchase` (пикер в кокпите прихода)
→ мост «дефицит → заказ» → пятый режим фронта «Заказы» с созданием и кокпитом →
юнит-тесты на мутации/замок/связь/мост.

**НЕ входит (следующие волны):** командный свод + консолидированный `Procurement`-
дашборд, экспорт `order.xlsx`, pegging (разворот источников), два числа бюджета +
себестоимость/экономия, `Transfer`/`Writeoff`/`Requisition`/`Inventory`, логин-экран,
авто-переходы статусов `partial`/`received`, UI вложений.

## Решения (см. JOURNAL 2026-07-02, «Волна 4»)

- **Схему БД не трогаем** — `Procurement`/`Purchase`/`PurchaseLine`/`Receipt.purchase`
  уже были. Диаграммы README не меняются.
- **`Procurement` — авто-родитель, не UI-сущность.** `Purchase.procurement` —
  обязательный `PROTECT` FK; командный свод отложен, поэтому под каждый проектный
  заказ движок создаёт тонкий draft-`Procurement` (`_solo_procurement`). Вырожденный
  1:1, forward-совместимо (будущая командная волна введёт общие Procurement, веерно
  нарезаемые на проектные Purchase).
- **Статусы: `draft`/`sent`/`cancelled` — только они «живут».** `partial`/`received`
  **не храним** — показываем как вычисляемую закрытость (received vs ordered). Движок
  «заказано» корректен и так: `SENT` минус поступившее → полностью полученный заказ
  выпадает в 0 без стораемого перехода.
- **Мягкий замок = `send`/`unsend`** (зеркалит `approve`/`unapprove` прихода): снятие
  ничего не разрушает (Purchase лотов не рождает) → guard по потомкам не нужен. Строки
  заказа правятся только в `draft`. `cancel`/`restore` — выход из «заказано» и возврат.
- **Мост «дефицит → заказ»** — payoff темы. На ▲-строке дашборда действие «＋ в заказ»:
  находит-или-создаёт draft-заказ проекта, добавляет/инкрементит `PurchaseLine`
  (`qty = to_order`), ведёт в кокпит заказа.
- **Связь `Receipt↔Purchase`** (долг волны 3): пикер в шапке кокпита прихода — заказы
  того же проекта. Валидация `purchase.project == receipt.project` в движке.
  «Поступившее» = `Σ Lot.qty` лотов с `receipt__purchase == purchase` (как уже читает
  `_purchased_on_order`). `rebuild_movements` тут не нужен — Purchase лотов не двигает.

## Этапы

### Этап 1 — Движок заказа (чистый Python + тесты) ← ядро волны
- [x] `purchase_cockpit(purchase)` — шапка (проект/статус/дата/примечание) + строки
  (item, заказано, поступило, остаток, закрытость) + связанные приходы + итоги.
- [x] Мутации: `create_purchase` (авто `_solo_procurement`), `add_purchase_line`
  (unique(purchase,item) guard, qty>0, draft-only), `update_purchase_line`,
  `remove_purchase_line`, `send_purchase`/`unsend_purchase`,
  `cancel_purchase`/`restore_purchase`.
- [x] `set_receipt_purchase(receipt, purchase|None)` — связь/отвязка, валидация проекта.
- [x] `add_to_project_order(project, item, qty, user)` — мост дефицита
  (find-or-create draft Purchase + find-or-add PurchaseLine).
- [x] Юнит-тесты: кокпит + закрытость, отправка → счёт в «заказано», unsend/cancel
  выводят из счёта, связанный приход режет «заказано» и закрывает строки, мост
  создаёт/инкрементит строку, отказ по чужому проекту/дублю item/правке не-draft.
- _Готово, когда:_ тесты зелёные, «заказано» дефицита совпадает с движком.

### Этап 2 — Записываемый DRF
- [x] `GET/POST /api/purchases/`, `GET /api/purchases/<pk>/` (кокпит),
  `POST .../lines/`, `PATCH/DELETE /api/purchase-lines/<pk>/`,
  `POST .../send|unsend|cancel|restore/`.
- [x] `POST /api/receipts/<pk>/link/` (связь с заказом),
  `GET /api/projects/<pk>/purchases/` (пикер), `POST /api/projects/<pk>/order/` (мост).
- _Готово, когда:_ HTTP-путь через test Client: create 201, строка 201, автосейв/
  удаление 200, send/unsend 200, link 200, мост 200, валидации 400.

### Этап 3 — Фронт: режим «Заказы» + кокпит + мост + пикер прихода
- [x] Пятый режим activity-bar (🛒): список заказов (глиф статуса) + призрачная строка
  «＋ Новый заказ» (проект).
- [x] `PurchaseView` — кокпит: шапка + строки с автосейвом qty (переиспуемый
  `CommitInput`) и удалением, призрачная строка добавления (пикер изделия), связанные
  приходы, действия send/unsend/cancel/restore (форма read-only под замком `sent`).
- [x] Мост в `DeficitView`: на ▲-строках «＋ в заказ» → `api.addToOrder` → переход в кокпит.
- [x] Пикер «Заказ (закрывает)» в шапке `ReceiptView` → `api.linkReceiptPurchase`.
- [x] API-клиент: типы кокпита заказа + мутирующие хелперы.
- _Готово, когда:_ из дефицита заводишь заказ → отправляешь → приход, связанный с
  заказом, гасит «заказано»; числа сходятся с движком.
