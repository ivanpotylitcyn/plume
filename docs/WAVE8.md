# Волна 8 — pegging: нарезка плана-закупки (`Procurement`) на веер проектных заказов (`Purchase`)

Чеклист восьмой волны. Цель — **слить два уровня закупок**: из согласованного командного
плана (`Procurement`, В7) породить веер проектных обязательств (`Purchase`, В4),
**разделяющих один план-родитель** (проект A ×60, проект B ×40 из плана «купить 60+20
всего»). Ломает вырожденную 1:1-заглушку `_solo_procurement`: план становится родителем
веера заказов, а не только своих строк. Продолжает паттерн волн 2–7 (записываемая
надстройка + движок = единственный источник правил + витрина-проекция). Значимые решения
— в [JOURNAL.md](JOURNAL.md) (запись 2026-07-03, «Волна 8»).

**Цель волны 8:** в кокпите плана (режим ⛁) под строками — панель **pegging**: по каждой
строке плана `(item, qty)` видеть распределение по проектам (наводка из командного свода +
фактически разложенное), пегать вручную или **«разрезать по проектам»** (autopeg по
наводке в один клик), снимать пег; внизу — веер проектных заказов со ссылками в их кокпиты.

## Объём

**Входит:** движок pegging (`procurement_pegging` — проекция; `peg_procurement_line` /
`unpeg_procurement_line` / `autopeg_procurement` — раскладка) + структурный разрез
плана↔заглушки (`_plan_procurements` через `Count`, а не `purchases__isnull`) → DRF
(`pegging`/`peg`/`unpeg`/`autopeg`) → фронт `PeggingPanel` в `ProcurementView` →
юнит-тесты + HTTP-smoke.

**НЕ входит (следующие волны):**
- Инвентаризация (`Inventory` — «найденные»/излишки партии, серая ре-материализация).
- Два числа бюджета + себестоимость/экономия, логин-экран, UI вложений (`Attachment`).

## Решения (проектные — в JOURNAL 2026-07-03, «Волна 8»)

- **Схему БД не трогаем** (как В7). `Procurement`/`Purchase`/`PurchaseLine` заморожены;
  pegging — записываемая надстройка над ними. Диаграммы README **не меняются**.
- **Разрез плана↔заглушки — структурный, не полем.** Старый признак плана
  `purchases__isnull=True` рушится ровно от pegging (пегнутый план обзаводится `Purchase`
  и исчезал бы из списка). Solo-заглушка (`_solo_procurement`, В4) = `Procurement` с ≥1
  `Purchase` **и без** строк плана; план — всё прочее. Обе схемы сосуществуют: ad-hoc
  проектный заказ (мост «дефицит → заказ») по-прежнему на заглушке.
- **Пег рождает проектный `Purchase` под планом** (`_project_purchase_under`, а не
  `_solo_procurement`): `procurement=<план>`, веер обязательств висит на общем плане.
- **autopeg идемпотентен** (`delta = to_order_наводки − уже_пегнуто`, пегаем только >0):
  повтор ничего не добавляет, ручной перепег не трогается. Кол-во **не клампим** по
  остатку плана (перепегнуть можно — расхождение информативнее; мутабельная ДНК).
- **Наводка — из командного свода** (`command_deficit.by_project`): сколько проекту ещё
  докупить по этому Item. Между проектами не перенеттим (как В7).

## Этапы

### Этап 1 — Движок pegging (чистый Python + тесты) ← ядро волны
- [x] `_plan_procurements()` через `annotate(Count).exclude(...)` — отсекает solo-заглушки
  структурно (есть `Purchase`, нет строк плана), не по `purchases__isnull`.
- [x] `procurement_pegging(proc)` — `{id, status, editable, rows[], fan[]}`: по строке
  плана пегнуто/остаток/статус ✓●▲ + `by_project` (наводка + пегнутое); веер заказов.
- [x] `peg_procurement_line` / `unpeg_procurement_line` / `autopeg_procurement`
  (`_project_purchase_under`, `_procurement_pegs`). Гварды: item из плана, проект
  активный внешний, qty>0, план не отменён; unpeg блокирован для отправленного заказа.
- [x] Юнит-тесты (5): пег рождает заказ под планом + инкремент; autopeg по наводке +
  идемпотентность + статус проекции; гварды (item/qty/kind/status); unpeg + блок send;
  список планов включает пегнутый, исключает solo-заглушку.
- _Готово, когда:_ тесты зелёные; веер сходится с наводкой свода. ✓

### Этап 2 — Записываемый DRF
- [x] `GET /api/procurements/<pk>/pegging/` — проекция.
- [x] `POST .../peg/` (`item_id`,`project_id`,`qty`), `POST .../unpeg/`
  (`item_id`,`project_id`), `POST .../autopeg/` — возвращают свежую проекцию.
- [x] HTTP-smoke (`PeggingHttpTests`, 2): проекция + autopeg (веер из двух заказов,
  ✓); ручной peg/unpeg + гвард item-не-в-плане 400.

### Этап 3 — Фронт: панель pegging в кокпите плана
- [x] `PeggingPanel` (в `ProcurementView` под строками, если план не отменён): таблица
  строк плана (пегнуто/остаток/статус) с раскрытием по проектам (наводка + ручной
  пег/снятие + «＋наводку»), кнопка «Разрезать по проектам» (autopeg), веер проектных
  заказов со ссылками в кокпиты (`openPurchase`). Освежается счётчиком `rev`.
- [x] API-клиент: типы (`Pegging`, `PeggingRow`, `PeggingProject`, `PeggingFanRow`) +
  `pegging`/`peg`/`unpeg`/`autopeg`; статус-бар → «волна 8».
- _Готово, когда:_ из плана «разрезаешь по проектам» → правишь раскладку → идёшь в
  проектные заказы. ✓

## Проверено

121 юнит-тест зелёный (114 волн 1–7 + 5 движка pegging + 2 HTTP-smoke волны 8);
`tsc -b && vite build` чист; `makemigrations --check` — **схема не менялась**. Живой
runserver-smoke на MySQL 8.0.25: мост→план (60) → pegging (наводка 40/20) → autopeg
(веер из двух draft-заказов 40/20, remaining 0, ✓) → повторный autopeg идемпотентен →
unpeg P1 (60→20) → peg item-не-в-плане → 400; dev-БД очищена за собой.
