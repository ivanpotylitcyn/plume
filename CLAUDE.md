# Plume — PLM-система

Веб-приложение для управления жизненным циклом изделий. Внутренний инструмент:
изделия/BOM, закупки, склад, комплектация образцов, запросы КП, ведение проектов.

## Стек

- **Бэкенд:** **Django 6.0.x + Django REST Framework** на **Python 3.14**
  (встроенная админка ускоряет внутренние операции PLM). DRF 3.17.x.
- **Фронтенд:** React + TypeScript (Vite).
- **БД:** **MySQL / MariaDB** (InnoDB, utf8mb4). Выбрана под целевой хостинг
  reg.ru shared, где PostgreSQL недоступен. БД-специфику держим переносимой.

## Ограничения хостинга (reg.ru shared) — учитывать при проектировании

- **Рантайм: Python 3.14 + Django 6.0.x + DRF 3.17.x.** На reg.ru shared под
  `/opt/python/` доступны бинари до 3.14 (проверено на текущем сайте 2026-07-01 —
  старое «максимум 3.10.1» больше не действует, см. JOURNAL). Локальный venv тоже
  на **Python 3.14** (pyenv: `pyenv install 3.14`) — локально = прод; прод собирает
  venv из `/opt/python/python-3.14/bin/python` (в `deploy/deploy.sh`, параметр
  `--python`). Актуальные фичи Django можно использовать: `CheckConstraint` —
  `condition=` (в 6.0 `check=` удалён); статика — через `STORAGES` (не
  `STATICFILES_STORAGE`). Драйвер БД — PyMySQL со спуфом `version_info` в
  `config/__init__.py` (Django 6.0 требует mysqlclient ≥ 2.2.1). 6.0 — не LTS;
  план — ехать 6.0 → 6.1 → 6.2 LTS по мере выхода.
- Только MySQL/MariaDB (PG нет). **Без `ArrayField`** (списки → отдельные
  таблицы), `JSONField` — точечно. Деньги/кол-ва — `DecimalField`, не float.
- **Нет Celery/Redis и долгоживущих процессов.** Тяжёлые операции (разузлование
  BOM, расчёт дефицита/заказа, пересчёт балансов) — синхронно в запросе; периодика — через
  **cron**. Realtime/WebSocket не закладываем.

## Домен (MVP)

Глоссарий и сущности — единый язык проекта:

- **Изделие (`Item`)** — единица справочника, **абстракция** (КД для прибора,
  datasheet для покупного). Едина для приборов, компонентов и материалов (вид —
  `kind`). BOM-версии отложены (один состав на изделие).
- **Состав изделия / BOM** — иерархия «изделие → компоненты» (`BomLine`).
- **Партия (`Lot`)** — физическое воплощение изделия, главная учётная единица склада.
- **Приход / УПД** — приёмка по универсальному передаточному документу (`Receipt`),
  рождает партии.
- **Комплектация (`Kitting`)** — списание компонентов под сборку + рождение
  партии-прибора.
- **Закупка (`Procurement`) → Заказ (`Purchase`) — два уровня.** Закупка =
  **планирование** (что и сколько решили купить; экспорт `order.xlsx`; может
  охватывать несколько проектов). Заказ = проектное **исполнение** (документальное
  обязательство); `PurchaseLine` закрывается одним/несколькими `Lot` через `Receipt`.
  Отдельных RFQ/КП-сущностей нет; пара `(purchase, item)` уникальна. **Один УПД ↔ один
  проект** (следим оффлайн + проверка в БД).
- **Дефицит к заказу** — вычисляемый отчёт: надо (BOM×потребность) − склад −
  заказано. Не таблица.
- **Проект** — сквозная сущность: закупки → комплектация → передача изделий →
  сверка балансов.

Ключевой поток: проект задаёт потребность (кол-во изделий) → разузлование BOM →
дефицит → закупка (order.xlsx) → заказ → приход на склад (УПД/Lot) → комплектация
(акты) → передача → сверка.

Русские имена сущностей и полный словарь терминов — в [README.md](README.md).

Решённое в модели (детали и диаграммы — в [README.md](README.md)):
- **`Item` единый** для изделий и компонентов; изделие может состоять из изделий
  (рекурсивный BOM).
- **Передача — только заказчикам**, отдельной сущности получателя нет: это
  `Transfer` в рамках проекта по накладной (документооборот вне PLM).
- **`Lot` — главная учётная единица.** Склад двигается только по `Lot` (в
  `StockMovement` нет `item`). `Lot` хранит цену и название из УПД; поставщик/дата
  — через `Receipt`. Каждый приход = новый `Lot`, отдельной `ReceiptLine` нет.
- **`Lot` всегда из origin-документа**, не «из воздуха»: поставка (`Receipt`),
  изготовление (`Kitting`), инвентаризация (`Inventory`) или отпочкование
  (`Requisition`). Ровно один origin задан (явные FK по типу).
- **Заводской номер (`Lot.serial_number`)** — ручной текст, только конечным
  приборам, которые мы производим; не ключ. Генеалогия прибора («паспорт») —
  разузлованием цепочки `Lot →(origin) Kitting →(lines) Lot`.
- **Проект — свойство лота (`Lot.project`), не движения.** Лот живёт в одном
  проекте всю жизнь; `movement.project` выводится из лота. «Общего через `NULL`»
  нет — его заменяют внутренние проекты (`Project.kind`): «Собственный склад»
  (белые, на балансе) и «Свободные неучтённые» (серые, списанные).
- **Закрытие проекта — закрывающими документами**, не правкой: невыбранные лоты
  (`qty>0`) сводятся в 0 передачей (`Transfer`), списанием (`Writeoff`) или
  постановкой на баланс (`Requisition` → «Собственный склад»). Реюз остатков —
  отпочкованием (`Requisition`) или через `Writeoff`→`Inventory` с `predecessor`.
- **Авторство — на документах** (`Purchase`/`Receipt`/`Kitting`/`Transfer`/
  `Inventory`/`Writeoff`/`Requisition` имеют `user` → `User`); движение без
  документа не существует, автор/дата берутся из документа.
- Цель проекта — НИР/контракт, ограниченный по времени: автоматизировать акты
  комплектации и знать остатки изделий в разрезе проектов.

## Подход к реализации (с 2026-06-30)

Схема БД **заморожена** (см. README — диаграммы). Дальше движемся **волнами**:
вычислительный движок-СППР (линза) выпускаем поэтапно, а не целиком. **Деплой на
прод — проведён отдельной экскурсией** (2026-07-01): прод поднят на reg.ru shared
(Django 6.0/Python 3.14), воспроизводимый путь — `deploy/` (рантбук
`deploy/README.md`). Основной цикл разработки остаётся **локальным**; на прод катим
осознанно через `deploy_local.sh`. cron/`verify`-периодика пока не заведены.
MVP-движок = дефицит (надо − склад − заказано, по проекту, 1 уровень, терминируя на
лотах) + карта остатков по складам-проектам; pegging, командный свод, два числа
бюджета и «экономию», призрачный кокпит — следующими волнами. Принятые на аудите
решения — в [docs/JOURNAL.md](docs/JOURNAL.md) (запись 2026-06-30, аудит MVP).
**План волны 1 (чеклист) — в [docs/WAVE1.md](docs/WAVE1.md)** (Django-app `plume`,
конфиг `config`, локальный MySQL 8.0.25, раскладка `backend/` + `frontend/`).

## Как работать в этом репозитории

- **Никогда не коммитить самостоятельно.** Коммитит только пользователь — он
  держит это под контролем. Не запускать `git commit`/`git push` даже после
  завершённой задачи; вместо этого сообщить, что готово к коммиту.
- **Проект открытый — никаких секретов в коммит.** В репозиторий НИКОГДА не
  должны попасть секреты, пароли, ключи, имена пользователей, IP-адреса и
  доменные имена — **даже публичные** (git-история остаётся навсегда). Все
  чувствительные данные передавать **аргументами при запуске** скрипта деплоя
  (не хардкодить, не класть в трекаемые файлы). `.env`/секреты — только в
  `.gitignore`. Перед любым коммитом проверять диф на утечки.
- **Журнал обязателен.** После значимого решения, выбора архитектуры или
  пойманных грабель — дописывай запись в [docs/JOURNAL.md](docs/JOURNAL.md)
  (формат — в шапке файла). Это память проекта между сессиями.
- **Диаграммы модели = источник правды.** Mermaid-диаграммы в
  [README.md](README.md) (продуктовая + техническая) обязаны строго
  соответствовать фактической модели БД. При **любом** изменении схемы — сразу
  обновлять обе диаграммы в том же изменении.
- **README-инструкция:** не тратить контекст на её обновление на каждом шаге.
  Когда достигнут понятный результат (рабочая сборка, новый модуль) — *предложить*
  пользователю обновить README, а не делать молча.
- Общение и доменные термины — на русском. Код, идентификаторы — по обычным
  конвенциям (англ. для кода ок).
- Пользователь — разработчик и предметник PLM: можно технически, без разжёвывания
  базовых вещей.
